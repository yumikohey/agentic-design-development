反思（Reflection）

一种让智能体评估自身输出并迭代改进的模式——通过生成、批评、改进的反馈循环，产出更高质量的结果。

模式概览
模式名称
反思模式
核心策略
反馈循环
关键优势
自我修正
最终目标
高质量输出

什么是反思模式？

反思模式（Reflection Pattern）是指智能体对自身的工作成果、输出内容或内部状态进行评估，并利用这种评估来改善其性能或精炼其响应。这是一种自我修正或自我提升的形式，允许智能体基于反馈、内部批判，或与期望标准的对比来迭代地改进输出或调整方法。

反馈循环机制

与简单的顺序链不同，反思模式引入了一个反馈循环。智能体不只是产生输出；它还会检查该输出，识别潜在问题或改进空间，然后利用这些洞察生成更好的版本。反思有时可以由一个独立的智能体来实施，其专门职责是分析初始智能体的输出。

生产者-批评者模型

一个高效的实现方式是将过程分为两个不同的逻辑角色：生产者（Producer）和批评者（Critic），即"生成器-评判器"或"生产者-审阅者"模型。使用两个具有不同系统提示词的专门智能体，通常比单个智能体自我审查产生更稳健、更客观的结果。

与其他模式的关联

当大语言模型保有对话历史时（第8章），反思的效果会显著增强。它还与目标设定和监控（第11章）相互融合，此时反思充当纠正引擎，利用监控反馈分析偏差并调整策略，将被动的执行者转变为有目的性、能自适应的系统。

反思模式解决的问题
即使是复杂的智能体工作流，初始输出或计划也可能不够优化、准确或完整。点击卡片查看这一模式所应对的具体失败场景：

首次输出质量不佳

初始响应通常缺乏打磨和完整性

智能体的初稿往往是功能性的，但并非最优。在没有评审循环的情况下，这些第一稿输出就成为了最终答案，即使通过额外的轮次本可以实现显著改进。

事实错误

大语言模型可能生成似是而非的错误信息

语言模型有时会以高置信度产生事实不准确的陈述。一个充当严格事实核查者角色的独立批评者智能体，可以对照原始任务要求核实声明，并标记不准确之处以便纠正。

代码缺陷与低效

生成的代码可能包含错误或风格问题

生成代码时，大语言模型可能产出能运行但有缺陷、遗漏边界情况或违反风格要求的实现。扮演高级工程师角色的批评者可以捕获这些问题——缺陷、缺少错误处理、低效——并指导改进。

自我审查中的认知偏差

单个智能体审查自身输出时存在盲点

当同一个大语言模型调用既生成又审查输出时，它容易确认自己的假设。具有不同系统提示词的独立批评者智能体则以全新的视角接触输出，专注于发现错误和改进空间。

缺乏改进机制

基本工作流缺少内置的自我修正能力

标准智能体模式（链式、路由、并行化）不包含任何让智能体识别并修复自身错误的机制。初始响应不管质量好坏都会成为最终答案。反思模式正是解决这一结构性空白的关键。

反思循环：解决方案
反思模式引入了一个结构化的生成-评估-改进循环。点击每个步骤查看详情：
1
执行（生成）
角色：生产者智能体
生产者智能体执行任务，生成初始输出。其唯一焦点是生成内容——编写代码、起草文本、制定计划。它接收初始提示词并生成输出的第一个版本。

输入：任务提示词 + 需求说明
输出：第一稿 / 初始尝试
2
评估 / 批评
角色：批评者智能体
批评者智能体（通常是具有不同系统提示词的独立大语言模型调用）分析前一步的结果，检查事实准确性、连贯性、风格、完整性和对指令的遵从情况。批评者旨在发现缺陷并提供结构化反馈，或者在输出已经完美时发出信号。

输入：原始任务 + 第一稿
输出：结构化批评 / "CODE_IS_PERFECT"
3
反思 / 改进
角色：生产者智能体（携带批评上下文）
根据批评，生产者智能体决定如何改进。这涉及生成精炼的输出、调整参数，甚至修改整体计划。包含任务、先前输出和批评的对话历史提供了有针对性改进所需的完整上下文。

输入：原始任务 + 先前输出 + 批评意见
输出：改进版本
4
迭代（循环或停止）
角色：编排器
精炼后的输出可以重新被评估，过程重复直到达到满意结果或触发停止条件（如批评者信号"完美"，或达到最大迭代次数）。这个迭代循环逐步提升质量，产出更准确、更连贯、更可靠的结果。

停止条件：批评者认可 或 达到最大迭代次数
输出：最终高质量结果
核心概念

双智能体架构

反思模式最强大的实现使用两个不同的智能体：生产者（完全专注于生成内容）和批评者（唯一职责是使用专门角色评估生产者的输出，如"高级软件工程师"或"严格事实核查员"）。这种分离防止了认知偏差，并实现了专业化的结构化反馈。

对话历史作为上下文

反思循环维护着包含原始任务、生成输出和批评意见的对话历史。该历史为每个后续步骤提供完整上下文，使智能体能够从过往批评中学习并避免重复错误。没有记忆，每次反思是孤立的；有了记忆，反思成为累积性过程。

停止条件设计

关键的设计决策是何时停止迭代。常见方法包括：(1) 来自批评者的哨兵短语（如"CODE_IS_PERFECT"），(2) 控制成本的最大迭代次数，或 (3) 质量阈值分数。在彻底性与延迟、成本之间取得平衡对于实际部署至关重要。

实际应用场景
在输出质量、准确性或对复杂约束的遵守至关重要的场景中，反思模式非常有价值。选择一个应用场景：

创意写作与内容生成

通过迭代批评来精炼生成的文本、故事、诗歌或营销文案，以达到更高的质量和感染力。

1
应用场景：智能体撰写一篇博客文章。
2
反思过程：生成草稿，从流畅度、语气和清晰度批判它，然后根据批评重写。重复直到文章达到质量标准。
3
收益：产出更精致、更有效的内容，远超单次生成所能达到的质量。

代码生成与调试

通过迭代批评来编写代码、识别错误并修复它们。

1
应用场景：智能体编写一个Python函数。
2
反思过程：编写初始代码，通过高级工程师批评者角色进行静态分析，识别错误或低效之处，然后根据发现修改代码。
3
收益：生成更健壮、功能更完整的代码，具备正确的边界情况处理和文档。

复杂问题求解

在多步骤推理任务中评估中间步骤或提出的解决方案。

1
应用场景：智能体解决逻辑难题或数学问题。
2
反思过程：提出一个步骤，评估它是否更接近解决方案或引入了矛盾，如需要则回溯或选择不同步骤。
3
收益：提升智能体在需要回溯的复杂问题空间中的导航能力。

摘要与信息综合

精炼摘要以提高准确性、完整性和简洁性。

1
应用场景：智能体对长篇文档进行摘要。
2
反思过程：生成初始摘要，将其与原始文档的关键点进行对比，精炼以补充遗漏信息或提高准确性。
3
收益：创建更准确、更全面的摘要，减少遗漏。

规划与策略

在执行前评估提出的计划，识别潜在缺陷或改进空间。

1
应用场景：智能体规划一系列行动以实现目标。
2
反思过程：生成计划，模拟执行或评估其对约束条件的可行性，根据评估结果修订计划。
3
收益：制定出更有效、更现实的计划，减少盲点。

对话智能体

回顾之前的对话轮次以维持上下文、纠正误解或提升响应质量。

1
应用场景：客户支持聊天机器人。
2
反思过程：在生成用户响应后，回顾对话历史和上一条消息，确保连贯性并准确回应用户的最新输入。
3
收益：带来更自然、更有效的对话，减少误解。
代码实战
以下示例展示了使用 LangChain（迭代循环）和 Google ADK（生成器-批评者管道）实现反思模式的方法。

LangChain：迭代反思循环

from langchain_openai import ChatOpenAI from langchain_core.messages import SystemMessage, HumanMessage llm = ChatOpenAI(model="gpt-4o", temperature=0.1) def run_reflection_loop(): # --- 任务定义 --- task_prompt = """编写名为 calculate_factorial 的Python函数。 要求：接受整数n，计算n!，包含文档字符串，处理n=0， 对负数抛出 ValueError。""" max_iterations = 3 message_history = [HumanMessage(content=task_prompt)] current_code = "" for i in range(max_iterations): # --- 第一阶段：生成/改进 --- if i == 0: response = llm.invoke(message_history) else: message_history.append( HumanMessage(content="请根据批评意见改进代码。")) response = llm.invoke(message_history) current_code = response.content message_history.append(response) # --- 第二阶段：反思（批评者角色） --- reflector_prompt = [ SystemMessage(content="""你是一名高级软件工程师。 对照原始任务严格评估代码。 查找缺陷、风格问题、缺少的边界情况。 若完美，回复 'CODE_IS_PERFECT'。 否则，提供有序的批评列表。"""), HumanMessage(content=f"任务:\n{task_prompt}\n\n代码:\n{current_code}") ] critique = llm.invoke(reflector_prompt).content # --- 停止条件 --- if "CODE_IS_PERFECT" in critique: break message_history.append( HumanMessage(content=f"批评意见:\n{critique}")) return current_code

Google ADK：生成器-批评者管道

from google.adk.agents import SequentialAgent, LlmAgent # 第一个智能体生成初稿 generator = LlmAgent( name="DraftWriter", instruction="根据用户提供的主题，撰写一段简短的信息性段落。", output_key="draft_text" # 输出保存到状态键 ) # 第二个智能体批评第一个智能体的草稿 reviewer = LlmAgent( name="FactChecker", instruction="""你是一名严格的事实核查员。 阅读状态键 'draft_text' 中的文本。 仔细核实所有声明的事实准确性。 输出字典：{'status': 'ACCURATE'|'INACCURATE', 'reasoning': '...'}""", output_key="review_output" ) # SequentialAgent 确保生成器在审阅者之前运行 review_pipeline = SequentialAgent( name="WriteAndReview_Pipeline", sub_agents=[generator, reviewer] )
深入解析：设计维度
点击每一层了解有效实现反思模式的关键设计考量：
单智能体 vs 双智能体反思
影响客观性和质量的架构选择
单个智能体可以通过审查自身输出来执行自我反思，但这引入了认知偏差——模型倾向于认同自己之前的输出。使用具有不同系统提示词的两个独立大语言模型调用（生产者+批评者）产生更稳健、更客观的结果。对于高风险任务，始终优先选择双智能体方法。
成本与延迟的权衡
每次反思循环都需要额外的大语言模型调用
迭代过程显著提升了输出质量，但代价是延迟和计算费用的增加。每一轮改进循环至少需要一次新的大语言模型调用，这使得该模式不适用于对时间敏感的应用。该模式还占用大量内存——每次迭代后，对话历史都会扩展。需要仔细设计停止条件，在质量与成本和速度之间取得平衡。
迭代的状态管理
如何在多个反思循环中维护上下文
真正的迭代反思需要有状态的工作流。在LangChain中，可以通过支持状态管理和条件转换的LangGraph等基于图的框架来实现。在更简单的实现中，维护对话历史列表（任务→输出→批评→改进输出）提供了必要的上下文。Google ADK的LoopAgent也可用于具有状态管理的迭代循环。
批评者角色的设计
批评者智能体的系统提示词至关重要
反思的质量在很大程度上取决于你如何指导批评者。有效的批评者提示词需要赋予专门角色（"你是一名高级软件工程师"），定义具体评估标准（事实准确性、代码质量、风格要求、完整性），指导产出结构化输出（有序的批评列表），并提供停止信号（发现无问题时回复"CODE_IS_PERFECT"）。特定领域的批评者比通用审阅者产生好得多的结果。
与记忆和目标的集成
上下文感知能力增强反思效果
当大语言模型保有对话记忆时，反思的效果会显著增强。这种对话历史为评估阶段提供了关键上下文，使智能体能够对照之前的交互、用户反馈和不断演进的目标来评估自身输出。它使智能体能够从过往批评中学习，避免重复同样的错误。与目标监控相结合时，反思充当纠正引擎，分析偏差并调整策略。
一览概要

问题所在

智能体的初始输出往往是次优的。基本工作流缺乏让智能体识别并修复自身错误的内置流程，导致初始响应不管质量好坏都会成为最终答案。

解决方案

反思模式通过反馈循环提供自我修正和精炼能力：生产者生成输出，批评者根据预定义标准评估，这一过程迭代进行以逐步提升质量。

使用时机

当最终输出的质量、准确性和细节比速度和成本更重要时使用。特别适用于精致的长文内容、代码编写与调试、以及详细计划的创建。

权衡取舍

这些好处的代价是延迟和费用的增加——每次改进循环都需要额外的大语言模型调用。完整的迭代反思需要有状态的工作流（如LangGraph）；单步反思可以用更简单的方式实现。

知识测验
核心要点
  • 反思模式引入了执行、评估/批评、改进的反馈循环，使智能体超越单次执行，实现迭代自我提升。
  • 其主要优势在于能够迭代地自我修正和精炼输出，在质量、准确性和对复杂指令的遵从度上显著提升。
  • 生产者-批评者模型（具有不同系统提示词的独立智能体）是一种强大的实现方式，能防止认知偏差并实现更专业、更客观的反馈。
  • 设计良好的批评者智能体必须拥有专门角色、明确的评估标准、结构化输出格式以及终止循环的停止信号。
  • 这些好处的代价是延迟和计算成本的增加——每次改进循环至少需要一次额外的大语言模型调用。需要仔细权衡质量与成本。
  • 完整的迭代反思需要有状态的工作流（LangGraph、ADK LoopAgent）；单步反思可以用更简单的LCEL链或顺序ADK管道实现。
  • Google ADK通过顺序工作流支持反思，一个智能体的输出被另一个智能体批评，从而实现后续的改进步骤。
  • 反思模式最适合质量、准确性和完整性比速度更重要的任务——如生成精致内容、编写和调试代码以及制定详细计划。
