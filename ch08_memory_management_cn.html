记忆管理

有效的记忆管理对智能体至关重要——既提供大语言模型上下文窗口内的短期情境记忆，也提供跨会话的长期持久记忆，使智能体能够学习、个性化响应并随时间保持连贯对话。

模式概览

智能体需要不同类型的记忆——就像人类一样——才能高效运作。记忆是指智能体从过往交互、观察和学习经验中保留并利用信息的能力，使其能够做出明智决策、维护对话上下文并随时间不断进步。

短期记忆（情境记忆）

类似于工作记忆，保存当前正在处理或最近访问的信息。对于基于大语言模型的智能体，短期记忆存在于上下文窗口中——包括最近的消息、回复、工具调用结果和反思。它是短暂的：会话结束时即消失。

长期记忆（持久记忆）

作为智能体需要跨交互、跨任务或长期保留信息的存储库。数据存储在外部——数据库、知识图谱或向量数据库中——需要时通过语义搜索检索。

会话与状态（ADK）

在 Google ADK 中，Session（会话）跟踪带有事件历史和临时 State（状态）数据的单个对话线程。SessionService 管理生命周期：创建、记录活动和终止。状态作为会话的草稿本，用于存储动态的、特定于轮次的数据。

记忆服务（ADK）

MemoryService 提供可搜索的长期知识存储库，来源于多个过去的会话或外部数据。它使用 add_session_to_memory 摄取会话，使用 search_memory 为当前上下文检索相关的过往知识。

核心问题

没有有效的记忆管理，智能体本质上是无状态的——无法提供连贯或个性化的体验。点击展开每个问题。

无状态智能体

没有记忆，智能体无法在交互之间维护上下文，将每次交流都视为全新开始。

无状态智能体无法记住用户刚说过的名字、完成了工作流的第一步或表达过某种偏好。这使得复杂的多轮任务无法完成，并产生令人沮丧的用户体验。

上下文窗口限制

大语言模型的上下文窗口容量有限——长对话最终会溢出，丢失关键的早期信息。

即使有大上下文窗口，每次调用都处理所有令牌也很昂贵。关键的早期信息被截断，导致智能体在对话中途"忘记"关键约束、用户偏好或先前决策。

无跨会话学习

没有持久记忆，智能体无法保留一次会话的知识以应用于未来的会话。

每次新对话都从零开始。智能体无法回忆用户更喜欢 Python 而非 JavaScript、之前在某个主题上遇到过困难，或者上周完成了入门培训。这阻碍了有意义的个性化。

无法个性化

没有用户特定的持久记忆，每次交互都是通用的，无法适应个人需求。

通用响应令期望系统"了解"自己的用户感到沮丧。个性化需要记住用户偏好、过往行为和以前会话的上下文——这一切都需要持久记忆。

昂贵的重复处理

没有摘要或选择性检索，智能体必须在每轮都重新处理所有先前上下文——代价高昂且缓慢。

每次 API 调用都传递完整对话历史会大幅增加延迟和令牌成本。高效的记忆管理使用摘要和向量搜索，只注入最相关的上下文。

双组件记忆解决方案

标准化解决方案实现了区分短期和长期存储的双组件记忆系统。

1
短期上下文窗口
大语言模型上下文中的近期交互数据
最近的消息、智能体回复、工具调用结果和反思都保存在上下文窗口中。这确保了单次会话内的对话连贯性。摘要技术可以压缩较旧的段落，为新信息腾出空间。
2
长期外部存储
向量数据库或数据库中的持久知识
必须超越单次会话持久保存的信息存储在外部数据库中——通常是用于语义搜索的向量存储。在后续会话中，智能体查询此存储以检索相关的过往知识，并将其整合到当前上下文窗口中。
3
会话状态管理
活跃对话的动态键值草稿本
会话状态存储与活跃对话相关的动态数据点：用户偏好、任务进度、增量数据收集和条件标志。键可以有作用域：会话特定（无前缀）、用户范围（user:）、应用范围（app:）或仅限当前轮次（temp:）。
4
记忆检索与注入
语义搜索将相关的过去上下文带入当前
开始新会话时，智能体使用语义相似性搜索查询长期记忆。最相关的过往知识被检索并注入当前上下文窗口——将多次交互桥接为连贯、连续的体验。
核心概念

Google ADK 和 LangChain 中智能体记忆系统的技术构建模块。

ADK 会话对象

Session（会话）封装了唯一的对话线程：标识符（id、app_name、user_id）、按时间顺序排列的事件日志、会话状态字典和最后更新时间戳。由 SessionService 管理——不要直接操作。

状态键前缀

状态键使用前缀定义数据范围和持久性。无前缀的键是会话特定的。user: 将数据与跨所有会话的用户 ID 关联。app: 指定所有用户共享的数据。temp: 仅对当前处理轮次有效，永不持久化。

ADK 记忆架构

{ "Session": "单个对话线程——事件 + 状态", "State": "会话内的轮次作用域草稿本", "Memory": "可搜索的跨会话知识存储库", "Services": { "SessionService": "管理 Session 生命周期", "MemoryService": "管理长期 Memory 存储" }, "Backends": { "InMemory": "测试用——非持久化", "Database": "通过 SQLAlchemy 使用 SQLite/PostgreSQL", "VertexAI": "可扩展的 GCP 生产环境" } }

长期记忆类型（LangChain）

类比人类记忆的三种类型：语义记忆（记住事实——用户档案、领域知识）；情节记忆（记住经历——过去的任务序列、少样本示例）；程序记忆（记住规则——智能体指令、可通过反思自我更新的系统提示）。

实际应用场景

记忆对于需要完成不止单一问题回答的智能体至关重要。

聊天机器人和对话式 AI

对话流程依赖短期记忆跟踪正在进行的对话。长期记忆使聊天机器人能够回忆用户偏好、过往问题和之前的讨论——在多个会话中提供个性化、连续的交互，而用户无需重复说明。

1
短期记忆维护当前对话线程
2
会话开始时检索相关的长期记忆
3
智能体根据召回的偏好个性化响应
4
会话结束时保存到长期记忆

任务型智能体

管理多步骤任务的智能体需要短期记忆来跟踪之前的步骤、当前进度和总体目标——保存在会话状态中。长期记忆对于访问不在直接上下文中的特定用户相关数据至关重要：过往配置、之前的决策、随时间积累的领域专业知识。

1
会话状态跟踪任务进度和中间结果
2
长期记忆提供历史上下文和用户偏好
3
任务完成时更新状态和长期知识

个性化体验

提供定制化交互的智能体利用长期记忆存储和检索用户偏好、过往行为和个人信息。这使智能体能够为每个人调整响应和建议——将通用助手转变为真正的个人助手。

1
用户档案以 user: 前缀键存储在长期记忆中
2
会话开始时检索并注入上下文
3
随着用户行为演变，偏好逐步更新

检索增强生成（RAG）

专为知识密集型问答设计的智能体将知识库作为其长期记忆。智能体使用语义相似性搜索检索相关文档或数据，并整合检索到的上下文以产生准确、有依据的响应。

1
查询被嵌入并用于语义相似性搜索
2
从向量存储检索 Top-K 相关文档
3
检索到的上下文注入提示以生成有依据的回答

学习与自我改进

智能体可以通过将成功策略、错误和新信息存储到长期记忆中来提升性能。强化学习智能体以这种方式存储学到的策略。反思技术提示智能体根据过去表现更新自己的程序记忆（系统提示）。

1
评估每次交互的结果
2
成功和失败的策略存储到长期记忆
3
智能体反思并根据历史更新自身指令

自主系统

机器人和自动驾驶汽车需要地图、路线、物体位置和习得行为的记忆。短期记忆处理即时周围环境和当前导航状态；长期记忆提供一般环境知识和习得的安全行为。

1
传感器数据填充短期情境感知
2
长期记忆提供地图和习得的环境模型
3
新经历更新持久的环境知识库
代码实战

使用 Google ADK 和 LangChain 的实际记忆实现示例。

ADK：通过 output_key 更新会话状态

from google.adk.agents import LlmAgent from google.adk.sessions import InMemorySessionService from google.adk.runners import Runner from google.genai.types import Content, Part # 智能体自动将最终文本响应保存到会话状态 greeting_agent = LlmAgent( name="Greeter", model="gemini-2.0-flash", instruction="生成一句简短友好的问候语。", output_key="last_greeting" # 保存到 session.state ) app_name, user_id, session_id = "state_app", "user1", "session1" session_service = InMemorySessionService() runner = Runner( agent=greeting_agent, app_name=app_name, session_service=session_service ) session = session_service.create_session( app_name=app_name, user_id=user_id, session_id=session_id ) # 运行智能体 user_message = Content(parts=[Part(text="你好")]) for event in runner.run( user_id=user_id, session_id=session_id, new_message=user_message ): if event.is_final_response(): print("智能体已响应。") # 运行后检查状态——状态已正确更新 updated_session = session_service.get_session(app_name, user_id, session_id) print(f"运行后状态：{updated_session.state}")

LangChain：ConversationBufferMemory 与 LLMChain

from langchain_openai import OpenAI from langchain.chains import LLMChain from langchain.prompts import PromptTemplate from langchain.memory import ConversationBufferMemory llm = OpenAI(temperature=0) # 提示模板中的 {history} 占位符 template = """你是一位有帮助的旅行顾问。 之前的对话： {history} 新问题：{question} 回答：""" prompt = PromptTemplate.from_template(template) # memory_key 与提示中的 {history} 匹配 memory = ConversationBufferMemory(memory_key="history") conversation = LLMChain(llm=llm, prompt=prompt, memory=memory) # 多轮对话——历史自动累积 r1 = conversation.predict(question="我想预订一张机票。") r2 = conversation.predict(question="顺便说一下，我叫小明。") r3 = conversation.predict(question="我叫什么名字？") print(r3) # 智能体从记忆中回忆起"小明"

LangGraph：长期程序记忆——反思更新指令

# 更新智能体指令的节点（程序记忆通过反思更新） def update_instructions(state: State, store: BaseStore): namespace = ("instructions",) # 从存储中获取当前指令 current_instructions = store.search(namespace)[0] # 提示大语言模型反思并生成改进的指令 prompt = prompt_template.format( instructions=current_instructions.value["instructions"], conversation=state["messages"] ) output = llm.invoke(prompt) new_instructions = output['new_instructions'] # 将更新后的指令存储回存储 store.put(("agent_instructions",), "agent_a", {"instructions": new_instructions})
记忆架构深度解析

从短暂的上下文到持久的知识——智能体记忆的各个层次。点击每一层进行探索。

上下文窗口（工作记忆）
短暂——会话结束时丢失
大语言模型的上下文窗口是智能体的工作记忆。它保存当前对话、工具输出和智能体反思。对于大上下文模型，这可以相当可观——但始终是短暂的。反复处理它代价高昂，使摘要策略对于长对话至关重要。
会话状态（草稿本）
会话内的轮次作用域键值存储
会话状态是跟踪与活跃对话相关的动态数据的键值对字典：任务进度、条件标志、中间结果。它在会话中的轮次之间持久保存，但作用域限于该会话。键前缀（user:、app:、temp:）可以扩展范围。始终通过 append_event 更新——不要直接修改。
语义记忆（事实）
用户档案、偏好、领域知识
语义记忆保留特定事实和概念——用户偏好、领域知识、实体信息。作为持续更新的用户档案（JSON 文档）或向量存储中的事实文档集合存储。通过语义相似性检索，将相关事实注入上下文窗口。
情节记忆（经历）
过去的交互序列和任务轨迹
情节记忆回忆过去的事件或行动——完成任务的交互序列，或需要避免的失败尝试。在实践中，通过少样本示例提示实现，智能体从过去成功的交互序列中学习。LangGraph BaseStore 支持跨会话的情节记忆存储。
程序记忆（规则与指令）
系统提示、行为规则——可自我更新
程序记忆是智能体的核心指令和行为规则——通常包含在系统提示中。智能体可以修改自己的提示以适应和改进。反思技术提示智能体审视当前指令和近期交互，然后生成存储回 BaseStore 的精炼指令。
一览概要

问题

没有记忆机制，智能体是无状态的——无法维护对话上下文、从经验中学习或个性化响应。这将它们限制为简单的单次交互。

解决方案

双组件记忆系统区分短期（上下文窗口）和长期（外部向量存储）。ADK 提供 Session + State 用于短期记忆，MemoryService 用于长期记忆。两者共同实现连贯、个性化和持续学习的智能体。

适用时机

每当智能体需要完成不止单一问题回答时使用：多轮对话、多步骤任务、个性化、从反馈中学习，或任何需要跨轮次或会话保持上下文的场景。

经验法则

当智能体预期需要根据过去的成功、失败或新获得的信息进行学习或适应时，实施记忆管理——或者当用户满意度取决于系统记住他们是谁时。

核心要点
  • 记忆使智能体能够维护历史记录、从经验中学习、个性化交互并管理复杂的时间依赖任务。
  • 短期记忆（上下文窗口）是短暂的，受大语言模型容量限制，会话结束时消失——但可以立即访问。
  • 长期记忆通过外部数据库（尤其是通过语义搜索访问的向量存储）跨会话持久保存。
  • 在 Google ADK 中，三种核心记忆抽象是：Session（对话线程）、State（会话草稿本）和 Memory（可搜索的跨会话存储库）。
  • 状态键前缀定义作用域：无前缀 = 会话级，user: = 用户级，app: = 应用级，temp: = 仅当前轮次。
  • 强烈不建议直接修改 session.state——始终通过 append_event 机制更新，以确保正确的持久化和事件跟踪。
  • 长期记忆有三种类比人类记忆的类型：语义记忆（事实）、情节记忆（经历）和程序记忆（可自我更新的规则/指令）。
  • Vertex AI Memory Bank 和 VertexAiRagMemoryService 提供具有语义搜索能力的生产级可扩展持久记忆。
