路由（Routing）

路由模式为智能体引入条件逻辑，使其能够根据输入内容、系统状态或前序操作的结果，动态地将请求分发至最合适的子流程、工具或子智能体。

路由模式概览

提示链提供了确定性的线性工作流，但现实世界的智能体系统往往需要根据情境做出自适应决策。路由正是那个让智能体具备动态分叉能力的机制。

什么是路由？

路由在智能体的操作框架中引入条件逻辑，使其从固定执行路径转变为动态评估特定标准、从一组可能的后续动作中进行选择的模型。这赋予系统更灵活、更具上下文感知能力的行为。

经典示例：客户服务智能体

一个配备路由功能的客服智能体，可先对用户查询意图进行分类，然后将其定向至：用于直接问答的专用子智能体、用于账户信息的数据库检索工具，或用于复杂问题的人工升级流程——而非使用单一固定的响应路径。

框架支持

LangChain、LangGraph 和 Google ADK 均提供了定义和管理条件逻辑的专用构件。LangGraph 的状态图架构尤其适合需要根据系统整体状态做出决策的复杂路由场景。

与提示链的区别

提示链是固定的线性序列；路由则在节点之间引入条件分支。二者往往结合使用——路由决定走哪条链，每条链各自完成其线性子任务。

问题所在

当智能体只能按照单一预设路径执行时，它在面对多样化的真实用户需求时将无能为力。点击每张卡片查看具体痛点。

线性流程的局限

顺序处理在需要自适应响应的场景中能力有限。

真实世界的智能体系统必须在多个潜在动作之间进行仲裁，考量因素包括环境状态、用户输入或前序操作的结果。固定的线性流程根本无法应对这种多样性。

单一响应路径

所有请求走同一条路，忽视了意图的差异性。

如果系统只有一个响应路径，"查询订单状态"和"请求技术支持"会得到相同的处理流程，这既低效又不准确——不同的意图本应触发截然不同的工具和子流程。

缺乏上下文感知

系统无法根据输入内容或状态动态调整行为。

没有路由机制，智能体就不具备"读懂"用户意图的能力。它无法区分"产品咨询"与"投诉升级"，更无法针对不清晰的请求触发澄清子流程。

工具选择的困境

面对多个专用工具时，系统不知该调用哪一个。

复杂系统中往往有针对不同任务的多个专用工具或子智能体。没有路由机制，系统无法充当高层级的调度器，将任务分配给最合适的处理单元。

静态而非动态

无法应对状态变化带来的新需求。

路由机制可以在智能体操作周期的多个节点介入：在开始时对主任务分类，在处理链中间确定后续动作，或在子程序中从给定工具集选择最合适的工具。缺乏这一能力，系统就只能静态运行。

解决方案：路由模式

路由通过引入一个评估机制来解决上述问题，该机制分析输入并将控制流导向最合适的处理路径。点击每个节点了解详情。

1
接收输入
用户查询 / 系统状态
智能体接收来自用户的查询或来自系统的状态信号。这可以是自然语言文本、结构化数据，或是前序操作的输出结果。
2
路由评估
LLM / 规则 / 嵌入向量
核心路由组件对输入进行分析，通过 LLM 推断意图、基于关键词的规则匹配，或向量嵌入相似度计算，输出一个路由决策标识符（如 "booker"、"info"、"unclear"）。
3
分支分发
RunnableBranch / ADK Auto-Flow
根据路由决策，系统将请求分发至对应的处理分支。在 LangChain 中使用 RunnableBranch；在 Google ADK 中，定义了 sub_agents 的协调者智能体会自动触发 LLM 驱动的委派（Auto-Flow）。
4
专用处理器执行
子智能体 / 工具链 / 人工升级
被选中的专用处理器（如预订处理器、信息检索器或澄清子流程）执行其专属逻辑，产生最终结果并返回给用户或上层流程。

核心概念

路由的核心组件是执行评估并引导控制流的机制。该机制可以通过多种方式实现。

基于 LLM 的路由

语言模型本身被提示分析输入并输出一个特定标识符或指令，以指示下一步骤或目标。例如，提示可要求 LLM 分析用户查询并仅输出类别：'订单状态'、'产品信息'、'技术支持' 或 '其他'。智能体系统读取此输出后相应地调度工作流。

基于嵌入向量的路由

输入查询被转换为向量嵌入（参见第14章 RAG），然后与代表不同路由或功能的嵌入向量进行比较，将查询路由至嵌入最相似的目标。这对于语义路由非常有用——决策基于输入的语义含义，而非仅仅是关键词。

基于规则的路由

使用基于从输入中提取的关键词、模式或结构化数据的预定义规则或逻辑（例如 if-else 语句、switch 分支）。这比基于 LLM 的路由更快、更具确定性，但在处理细微差别或新颖输入时灵活性较低。

基于机器学习模型的路由

使用专门在小型标注数据集上训练的判别模型（如分类器）来执行路由任务。与基于嵌入的方法共享概念相似性，但其关键特征是监督微调过程，将路由逻辑编码到模型权重中，而非在推理时执行提示。

# 路由决策的结构示意 { "input": "我想预订一张去伦敦的机票", "routing_decision": "booker", "confidence": 0.97, "handler": "booking_handler", "method": "llm_based" }

应用场景

路由模式是设计自适应智能体系统的关键控制机制，其实用性横跨多个领域。选择场景查看详情。

虚拟助手 / AI 客服

在人机交互中，路由用于解析用户意图。对自然语言查询的初步分析决定了最适当的后续动作。

1
接收用户自然语言查询（如"我的订单到哪了？"）
2
LLM 路由器对意图进行分类：订单查询 / 产品咨询 / 投诉升级
3
将请求转发至对应子智能体或工具链
4
专用处理器返回个性化、情境化的响应

允许系统超越线性对话流，实现真正的上下文感知响应。

自动化数据与文档处理

路由充当分类与分发功能。收到的数据（电子邮件、工单、API 载荷）根据内容、元数据或格式进行分析，然后被引导至对应的处理工作流。

1
接收传入数据（邮件、支持工单、API 载荷）
2
分析内容、元数据或格式，确定类别
3
路由至：销售线索流程 / JSON 数据转换 / 紧急问题升级路径
4
对应工作流处理并生成输出

多专用工具的研究系统

路由充当高层级调度器。由搜索、摘要和分析等不同专用子智能体组成的研究系统，使用路由器根据当前目标将任务分配给最合适的智能体。

1
接收研究任务或查询
2
分析当前目标：需要搜索新信息？摘要已有内容？还是数据分析？
3
调度至：搜索子智能体 / 摘要子智能体 / 分析子智能体
4
聚合各子智能体结果，输出综合报告

AI 编程助手

AI 编程助手通过路由识别编程语言和用户意图（调试、解释或翻译），再将代码片段传递给正确的专用工具。

1
接收用户代码请求（如"帮我调试这段 Python 代码"）
2
识别编程语言 + 意图：调试 / 解释 / 翻译
3
路由至对应专用处理器，附带代码上下文
4
返回针对性的代码建议或解释

AI 驱动的智能教学系统

在 AI 辅导系统中，路由根据学生表现选择课程中的下一个模块，允许系统超越线性对话流，实现真正的个性化学习路径。

1
评估学生当前表现和理解程度
2
路由决策：需要复习？进入下一模块？还是补充练习？
3
选取最合适的学习内容或练习
4
提供个性化的下一步学习体验

代码实战

以下代码展示了使用 LangChain 和 Google ADK 实现路由模式的核心逻辑。

LangChain 实现：RunnableBranch

使用 LangChain 构建协调者路由链，通过 RunnableBranch 将请求委派至对应的处理函数。

from langchain_google_genai import ChatGoogleGenerativeAI from langchain_core.prompts import ChatPromptTemplate from langchain_core.output_parsers import StrOutputParser from langchain_core.runnables import RunnablePassthrough, RunnableBranch # --- 定义模拟子智能体处理函数 --- def booking_handler(request: str) -> str: """模拟预订智能体处理请求。""" return f"预订处理器已处理请求：'{request}'。结果：模拟预订操作。" def info_handler(request: str) -> str: """模拟信息智能体处理请求。""" return f"信息处理器已处理请求：'{request}'。结果：模拟信息检索。" # --- 定义协调者路由链 --- coordinator_router_prompt = ChatPromptTemplate.from_messages([ ("system", """分析用户请求，判断应由哪个专用处理器处理。 - 如果请求与预订航班或酒店相关，输出 'booker'。 - 对于其他通用信息问题，输出 'info'。 - 如果请求不明确，输出 'unclear'。 只输出一个词：'booker'、'info' 或 'unclear'。"""), ("user", "{request}") ]) coordinator_router_chain = coordinator_router_prompt | llm | StrOutputParser() # --- 定义委派逻辑（使用 RunnableBranch）--- branches = { "booker": RunnablePassthrough.assign( output=lambda x: booking_handler(x['request']['request'])), "info": RunnablePassthrough.assign( output=lambda x: info_handler(x['request']['request'])), } delegation_branch = RunnableBranch( (lambda x: x['decision'].strip() == 'booker', branches["booker"]), (lambda x: x['decision'].strip() == 'info', branches["info"]), branches["unclear"] # 默认分支 ) coordinator_agent = { "decision": coordinator_router_chain, "request": RunnablePassthrough() } | delegation_branch | (lambda x: x['output'])

Google ADK 实现：Auto-Flow 自动委派

在 ADK 范式中，路由通过定义离散的工具集来实现。框架的内部逻辑（Auto-Flow）自动根据用户查询将请求路由至正确的功能处理器。

from google.adk.agents import Agent from google.adk.runners import InMemoryRunner from google.adk.tools import FunctionTool # --- 定义专用子智能体 --- booking_agent = Agent( name="Booker", model="gemini-2.0-flash", description="处理所有航班和酒店预订请求的专用智能体。", tools=[booking_tool] ) info_agent = Agent( name="Info", model="gemini-2.0-flash", description="提供通用信息并回答用户问题的专用智能体。", tools=[info_tool] ) # --- 定义带有明确委派指令的协调者智能体 --- coordinator = Agent( name="Coordinator", model="gemini-2.0-flash", instruction="""你是主协调者。你唯一的任务是分析 传入的用户请求并将其委派给适当的专用智能体。 不要直接回答用户。 - 对于任何与预订航班或酒店相关的请求，委派给 'Booker' 智能体。 - 对于所有其他通用信息问题，委派给 'Info' 智能体。""", description="将用户请求路由到正确专用智能体的协调者。", # sub_agents 的存在默认启用 LLM 驱动委派（Auto-Flow） sub_agents=[booking_agent, info_agent] )

深入解析：路由实现层次

路由机制可以在智能体操作周期的多个节点介入。点击每一层深入了解其工作原理与适用场景。

任务初始分类层
在起点对主任务进行路由
在智能体操作周期的起始阶段，路由机制对传入的主任务进行分类。这是最常见的路由场景：分析用户的初始请求，将其整体分派至最合适的处理流程。例如客户服务系统在接收到查询时，首先判断是售前咨询、售后支持还是投诉处理。
处理链中间节点层
在链式处理流程中动态决定下一步
路由可以在处理链的中间节点介入，根据前序步骤的输出结果动态决定后续动作。例如在文档处理流程中，初步分析完成后，路由器根据文档类型（合同、报告、邮件）决定调用哪种专用解析器。LangGraph 的状态图架构特别适合这种场景。
工具选择子程序层
从工具集中选择最合适的工具
在子程序级别，路由机制从给定的工具集中为特定子任务选择最合适的工具。Google ADK 中每个工具都有描述，框架通过 LLM 理解工具描述来匹配用户意图，自动调用最恰当的工具函数——这本质上是一种内置的工具级路由。
LangGraph 状态图层
基于累积状态的复杂条件路由
LangGraph 的状态图架构特别适合复杂路由场景——决策依赖于整个系统的累积状态。开发者在图中定义可能的操作路径和节点间的转换条件（函数或模型评估），从而构建能够根据动态状态做出精确决策的智能体工作流。
ADK Auto-Flow 委派层
LLM 驱动的自动子智能体委派
Google ADK 的 Auto-Flow 机制通过让 LLM 理解各子智能体的描述来实现自动路由委派。协调者智能体配置了 sub_agents 列表后，其 LLM 会自动分析传入请求的意图，并将任务委派给描述最匹配的子智能体——无需显式编写路由规则。

一览概要

是什么（WHAT）

智能体系统必须能够响应多种多样的输入和情况，而这些无法由单一线性流程处理。没有选择正确工具或子流程的机制，系统将保持僵化且缺乏自适应能力。

为什么（WHY）

路由模式通过将条件逻辑引入智能体的操作框架，提供了标准化解决方案。它将静态的预定义执行路径转变为灵活的、能够选择最佳可能动作的上下文感知工作流。

经验法则

当智能体必须根据用户输入或当前状态在多个不同工作流、工具或子智能体之间做出选择时，使用路由模式。对于需要对传入请求进行分诊或分类以处理不同类型任务的应用，路由必不可少。

三种路由机制对比

LLM 路由：灵活、语义理解强，但有延迟。规则路由：快速确定，但无法处理新颖输入。嵌入路由：基于语义相似度，适合意图匹配。ML 模型路由：经过专项训练，决策逻辑固化在权重中。

核心要点

掌握路由模式的关键洞见。

  • 路由使智能体能够根据条件对工作流中的下一步骤做出动态决策，而不仅仅是线性执行。
  • 它允许智能体处理多样化的输入并自适应其行为，从而超越线性执行的局限。
  • 路由逻辑可以使用 LLM（语义理解）、基于规则的系统（确定快速）或嵌入相似度（语义匹配）来实现——各有其适用场景。
  • LangGraph 和 Google ADK 等框架提供了在智能体工作流中定义和管理路由的结构化方式，尽管架构方法不同。
  • LangGraph 的图结构提供了定义状态和转换的可视化和显式方式，非常适合具有复杂路由逻辑的多步骤工作流。
  • Google ADK 中，定义了 sub_agents 的协调者智能体自动启用 LLM 驱动的委派（Auto-Flow），无需显式编写路由规则。
  • 路由机制可在智能体操作周期的多个节点介入：初始任务分类、处理链中间节点，或工具选择子程序。
  • 掌握路由模式是构建能够智能导航不同场景并基于上下文提供定制化响应的多功能、强健智能体应用的关键。

💡 从第一性原理看AI Agent路由（Routing）设计模式：打破顺序处理的枷锁

1. 引言：一个不可避免的痛点

想象一下，你刚刚利用大语言模型（LLM）构建了一个客服AI助手。最开始，你的设计非常简单直接：用户输入问题，系统将问题塞进一个精心设计的Prompt模板中，然后调用LLM生成回复。

第一天，用户问："你们的退货政策是什么？"你的AI完美解答。第二天，用户问："帮我查一下订单号#12345的物流状态。"你的AI陷入了沉思，然后胡编乱造了一个物流状态，或者生硬地回答："作为一个AI，我无法访问您的数据库。"

你突然意识到一个真实的痛点：现实世界中的用户请求是极其多样且不可预测的。复杂的现实场景要求系统能够根据环境变量、用户输入或上一步的操作结果，在多个潜在的动作之间进行仲裁和选择。这就是我们今天要探讨的核心：路由（Routing）设计模式。

2. 问题陈述：为什么顺序处理不够用？

在构建大模型应用的最早期，我们通常使用"提示链"（Prompt Chaining）。这就好比工厂里的流水线：第一步提取关键词，第二步总结摘要，第三步翻译成目标语言。这种顺序处理在执行确定性的、线性的工作流时非常完美。

但问题在于：真实世界的任务往往不是线性的。如果你的业务同时涉及"查询订单"、"产品咨询"和"技术支持"，强制使用单一的线性链条会导致灾难。我们需要系统从"固定执行路径"转变为一种动态评估特定标准以选择后续行动的新模式。

3. 第一性原理推导：从零推导路由的必要性

让我们抛开所有现成的框架，如果你从零开始设计一个智能体，你会如何解决"不同输入需要不同处理"这个问题？

第一步：识别分歧点。你首先会发现，用户的自然语言输入就像一个充满了岔路口的迷宫。有的路通向数据库（查订单），有的路通向知识库（查产品），有的路需要呼叫人类专家（技术支持）。

第二步：引入"大脑"机制。为了在岔路口做出正确的选择，你需要在系统最前端放置一个"分发员"。这个分发员不负责实际解决问题，它只负责一件事：分类与指派。

第三步：形成条件逻辑。这就自然而然地引入了条件逻辑（Conditional Logic）：

  • 如果意图是"查询订单状态"→ 路由到与订单数据库交互的子智能体
  • 如果意图是"产品信息"→ 路由到搜索产品目录的子智能体
  • 如果意图是"技术支持"→ 路由到访问故障排除指南或升级给人类的链
  • 如果意图不明确 → 路由到澄清问题的子智能体

顺理成章地，你就推导出了"路由（Routing）"的概念。路由通过将控制流动态导向不同的专业化函数、工具或子流程，实现了动态决策的能力，让智能体从机械的执行者变成了能够根据上下文灵活应对的智能系统。

4. 核心概念详解：路由的三个组成部分

路由模式包含三个关键组成部分：

  1. 输入接收（Input Reception）：捕获用户的自然语言查询或系统内部的状态变化。
  2. 路由器/分类器（Router/Classifier）：整个模式的灵魂——执行评估、判断当前输入属于哪一种类别。
  3. 处理器选择与分发（Processor Selection）：根据分类结果，将数据负载传递给最适合处理该任务的专业模块。

你可以把它类比为医院的急诊分诊台：病人（输入）来到前台，分诊护士（路由器）根据症状评估紧急程度和科室，然后将病人送往外科、内科或重症监护室（处理器）。

5. 五种路由机制深度对比

5.1 规则路由（Rule-based）
基于关键词、正则表达式或结构化数据的 if-else 逻辑。速度极快、确定性强，但在处理人类细微差别或新颖输入时极度缺乏灵活性。适合：明确包含命令前缀（如 /help）的系统。

5.2 嵌入向量路由（Embedding-based）
将输入转换为向量嵌入，与预先计算的路由向量进行距离相似度比较。即使用词完全不同，语义相近的请求也能正确路由。适合：基于深层含义而非表面关键词的路由决策。

5.3 机器学习分类器路由（ML Model-based）
使用专门训练的判别模型（分类器）执行路由任务。路由逻辑被编码在微调模型的权重中，LLM不参与实时路由决策。适合：需要极高准确率且有足够标注数据的企业级系统。

5.4 LLM路由（LLM-based）
让大语言模型自己充当路由器，通过Prompt指示LLM分析输入并输出特定的分类标识符。利用LLM强大的零样本泛化能力理解复杂长尾意图。适合：意图边界模糊、需要深度推理才能分类的场景。

5.5 LangGraph状态路由（State-based Graph）
路由决策取决于整个系统随时间累积的"状态"，而非仅当前输入。适合：多步骤任务，下一步取决于之前工具调用结果的复杂工作流。

6. 框架实战：LangChain vs Google ADK

LangChain — RunnableBranch 显式委派

coordinator_router_prompt = ChatPromptTemplate.from_messages([
    ("system", """Analyze the user's request and determine which specialist should process it.
     - If booking related, output 'booker'.
     - For general information, output 'info'.
     - If unclear, output 'unclear'.
     ONLY output one word: 'booker', 'info', or 'unclear'."""),
    ("user", "{request}")
])
coordinator_router_chain = coordinator_router_prompt | llm | StrOutputParser()

delegation_branch = RunnableBranch(
    (lambda x: x['decision'].strip() == 'booker', branches["booker"]),
    (lambda x: x['decision'].strip() == 'info',   branches["info"]),
    branches["unclear"]
)

coordinator_agent = {
    "decision": coordinator_router_chain,
    "request":  RunnablePassthrough()
} | delegation_branch | (lambda x: x['output'])

Google ADK — Auto-Flow 隐式委派

coordinator = Agent(
    name="Coordinator",
    model="gemini-2.0-flash",
    instruction="Analyze incoming user requests and delegate to the appropriate specialist agent.",
    sub_agents=[booking_agent, info_agent]  # Auto-Flow 自动路由
)

LangChain 使用显式的 RunnableBranch，类似传统 switch-case，但判断条件由 LLM 生成。ADK 则通过 sub_agents 注册自动委托——框架内部逻辑自动将意图与处理器匹配，代码更简洁。

7. 决策指南：何时用哪种机制？

  1. 输入完全可预测且结构化？ → 规则路由：速度最快，成本为零。
  2. 有大量标注数据且需要极高准确率？ → ML分类器：稳定且推理成本低。
  3. 主要基于语义意图的粗粒度分类？ → 嵌入向量路由：捕捉语义距离。
  4. 智能体动作是高度离散的独立工具/子智能体？ → Google ADK：Auto-Flow 减少样板代码。
  5. 多步骤、复杂状态累积的工作流？ → LangGraph + LLM路由：状态图处理复杂节点转换。

8. 常见陷阱与最佳实践

  • LLM 输出边界陷阱：在 Prompt 中严格约束输出词汇，并在代码中使用 .strip() 消除格式噪音。
  • 缺少兜底逻辑：永远实现一个 unclear_handler，意图不明时路由到澄清提示链。
  • 框架锁定思维：LangGraph 强调显式节点转换，ADK 强调意图到工具的隐式映射。理解底层机制才能在不同场景选对工具。

9. 总结：路由模式对智能体能力的根本性提升

路由提供了一个标准化的解决方案，将条件逻辑注入到智能体的操作框架中。无论是在人机交互中解释用户意图，在数据处理管道中充当分类与分发网络，还是在多智能体系统中担任高层级调度员，路由都提供了不可或缺的逻辑仲裁能力。

掌握路由模式，我们实质上完成了一次范式转移：将智能体从静态的、预先定义序列的执行器，彻底转变成了一个能够在不断变化的条件下智能决定最有效任务完成方法的动态决策系统——这是构建真正具备自适应能力的现代 Agentic 系统的关键一步。
