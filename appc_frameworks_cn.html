智能体框架概览

主流 LLM 应用开发框架实用指南——从简单的线性链到完整的多智能体编排平台，助你找到最适合的技术选型。

框架全景

智能体框架生态涵盖了从基础组件库到高度封装的多智能体平台的宽广谱系。选择合适的框架，取决于你的应用对"控制力"与"开发效率"的权衡需求。

LangChain — 线性管道

LangChain 的核心优势在于其表达式语言（LCEL），允许通过管道符 | 将组件串联成链。适用于有向无环图（DAG）式工作流，流程单向推进，无循环。

LangGraph — 有状态图

基于 LangChain 构建，支持将工作流定义为含节点和边的图结构。核心优势是支持循环——允许智能体在任务完成前持续循环、重试、调用工具。

Google ADK — 多智能体编排

Google 的智能体开发套件提供了现成的多智能体交互模式，内置 SequentialAgent 和 ParallelAgent 等智能体类型，将底层图构造细节完全抽象隐藏。

CrewAI — 角色团队

CrewAI 将多智能体系统类比为一支人类专家团队，每个智能体拥有角色、目标和背景故事。开发者只需定义智能体（Agent）、任务（Task）和团队（Crew）即可。

为什么需要专用框架？

从零开始构建可靠的智能体应用，需要反复解决若干共性难题。点击每张卡片深入了解。

状态管理困难

LLM 本身是无状态的，复杂工作流需要在多个步骤和智能体边界之间传递上下文。

若无显式状态对象，数据必须手动穿透每个函数调用。LangGraph 通过类型化的状态字典解决了这一问题，状态在整个工作流中持续存在。

控制流过于复杂

智能体需要循环、分支、重试和条件决策——这些用简单函数链无法表达。

基于 DAG 的框架（如 LangChain）无法处理循环。基于图的框架（如 LangGraph）引入了带条件逻辑的边，可在节点间动态路由。

多智能体协调难

包含多个专业智能体的系统需要编排逻辑来分配、路由和汇总结果。

Google ADK 和 CrewAI 等高层平台提供了现成的协调模式——监督者智能体、顺序团队、层级委派——开发者无需从零搭建。

工具集成样板代码繁琐

将 LLM 连接到外部 API、搜索工具、数据库和代码执行器，涉及大量重复性脚手架代码。

框架提供了标准化的工具接口和函数调用集成，让开发者专注于定义工具功能，而非如何从 LLM 调用它们。

可观测性与调试困难

多步骤 LLM 工作流不透明——很难追踪为何走了某条路径，或错误发生在哪个环节。

SuperAGI 等面向生产的平台内置了监控、日志记录和故障处理机制，能捕获无限循环等常见问题。

三层抽象架构

智能体框架按抽象层级分为三个层次。理解这一层级结构，有助于为应用场景选择最合适的工具。点击各节点查看详情。

1
基础组件库
LangChain · LlamaIndex · Haystack
这些框架在最基础的层级运作。LangChain 为模型调用、输出解析和操作串联提供标准化接口。LlamaIndex 专注于数据摄取和 RAG 管道。Haystack 聚焦于可扩展的生产级搜索系统。它们提供最大控制力，但样板代码也最多。
2
基于图的智能体框架
LangGraph · Microsoft AutoGen · Semantic Kernel
这些框架在基础库之上引入了有状态的循环控制流。LangGraph 将智能体工作流建模为带显式状态对象的节点与边图。AutoGen 使智能体能通过对话协作完成任务。Semantic Kernel 通过插件和规划器将 LLM 集成到企业代码库中。
3
多智能体编排平台
Google ADK · CrewAI · MetaGPT · SuperAGI · Strands
这些平台完全抽象了图构造细节，提供现成的多智能体团队模式。Google ADK 提供 SequentialAgent 和 ParallelAgent 原语。CrewAI 将协作建模为有角色和任务的团队。MetaGPT 用标准操作流程（SOP）模拟软件公司。它们以牺牲细粒度控制为代价，换取对复杂系统的快速开发能力。
各框架核心概念

每个框架都有其独特的思维模型。在编写任何代码之前，理解核心抽象至关重要。

LangChain：LCEL 与链

核心抽象是用 LangChain 表达式语言（LCEL）构建的链。组件通过管道符 | 组合。工作流是有向无环图——每个步骤的输出成为下一步的输入。每次运行通常无状态。

# LCEL 链：提示词 | 模型 | 输出解析器 chain = prompt | model | output_parser result = chain.invoke({"topic": "机器学习"})

LangGraph：节点、边与状态

核心抽象是 StateGraph（状态图）。你需要定义类型化的状态字典，然后添加节点（Python 函数）和边（控制流）。边可以是条件性的，从而支持循环。状态对象由每个节点更新并传递给下一个节点。

CrewAI：智能体、任务与团队

三个核心构件：智能体（Agent）拥有角色、目标和背景故事；任务（Task）是有明确描述和预期输出的离散工作单元；团队（Crew）将智能体和任务绑定在一起，执行顺序或层级化的流程（Process）。

Google ADK：LlmAgent 与工具

核心原语是配置了模型、名称、指令和工具列表的 LlmAgent。ADK 隐式处理智能体循环。对于多智能体系统，智能体可以组合进 SequentialAgent 或 ParallelAgent 容器，自动管理流程控制。

框架选型指南

根据应用场景选择合适的框架。每个标签页描述一个典型场景及最佳框架选择。

简单 RAG 管道

推荐框架：LangChain（LCEL）

1
用户查询到达
2
检索器从向量数据库中检索相关文档
3
提示词模板将上下文与查询格式化
4
LLM 生成答案；输出解析器格式化响应

LangChain 的 LCEL 在此完美适用。工作流是线性的（A → B → C），没有循环，完全符合管道链的设计：retriever | prompt | model | parser。

带重试逻辑的工具调用智能体

推荐框架：LangGraph

1
智能体接收任务并生成计划
2
智能体调用工具（搜索、计算器、API）
3
条件边判断：若结果不满意，循环回步骤 2
4
任务完成时退出循环，返回最终答案

LangGraph 的循环图天然支持重试逻辑。状态对象跨迭代追踪工具调用历史记录。

使用专业智能体生成研究报告

推荐框架：CrewAI 或 Google ADK

1
研究员智能体搜索网络信息
2
分析师智能体将发现整合为结构化数据
3
写作智能体撰写最终报告
4
编辑智能体审阅并优化草稿

CrewAI 的角色模型非常契合此场景。每个智能体有明确的人物设定和任务；Crew 顺序执行并在任务间传递上下文。

规划与执行智能体

推荐框架：LangGraph

1
规划器节点为目标制定多步骤计划
2
执行器节点执行计划中的下一步
3
评估器节点检查步骤是否成功
4
基于结果循环回规划器更新计划，或在完成时退出

LangGraph 的显式状态传递使计划、已完成步骤和观察结果都可以存储在共享状态对象中，跨循环更新。

企业软件集成

推荐框架：Semantic Kernel

1
现有 .NET 或 Python 代码库将原生函数作为插件暴露
2
Semantic Kernel 规划器根据用户目标决定调用哪些插件
3
LLM 作为推理引擎，编排现有业务逻辑
4
结果通过标准应用层返回

Semantic Kernel 的插件架构专为集成进现有企业代码库而设计，尤其适用于 .NET 和 Python 环境。

知识密集型 RAG 系统

推荐框架：LlamaIndex

1
将大型文档语料库摄取进复杂的索引结构
2
构建带重排序和过滤的自定义检索管道
3
查询引擎处理跨多个数据源的复杂嵌套查询
4
智能体利用检索上下文给出有据可查的回答

LlamaIndex 在复杂数据摄取和检索管道方面表现卓越。其原生的复杂智能体控制流工具相对不足，但在 RAG 领域无可匹敌。

代码实战

三个主要框架的具体 Python 代码示例，来自本附录的核心内容。

LangChain — 简单 LCEL 链

# 简单 LCEL 链：演示管道组合方式 from langchain_core.prompts import ChatPromptTemplate from langchain_openai import ChatOpenAI from langchain_core.output_parsers import StrOutputParser prompt = ChatPromptTemplate.from_template( "用一句话总结以下文本：\n\n{text}" ) model = ChatOpenAI(model="gpt-4o") output_parser = StrOutputParser() # 管道符将各组件串联 chain = prompt | model | output_parser result = chain.invoke({"text": "LangChain 支持可组合的 LLM 应用构建..."}) print(result)

LangGraph — 并行扇出与聚合

from langgraph.graph import StateGraph, START, END from typing import TypedDict # 图状态定义 class State(TypedDict): topic: str joke: str story: str poem: str combined_output: str # 节点函数——每个节点更新状态的一个切片 def call_llm_1(state: State): msg = llm.invoke(f"写一个关于 {state['topic']} 的笑话") return {"joke": msg.content} def call_llm_2(state: State): msg = llm.invoke(f"写一个关于 {state['topic']} 的故事") return {"story": msg.content} def aggregator(state: State): combined = f"笑话：\n{state['joke']}\n\n故事：\n{state['story']}" return {"combined_output": combined} # 构建并编译图 builder = StateGraph(State) builder.add_node("call_llm_1", call_llm_1) builder.add_node("call_llm_2", call_llm_2) builder.add_node("aggregator", aggregator) builder.add_edge(START, "call_llm_1") builder.add_edge(START, "call_llm_2") builder.add_edge("call_llm_1", "aggregator") builder.add_edge("call_llm_2", "aggregator") builder.add_edge("aggregator", END) workflow = builder.compile() state = workflow.invoke({"topic": "猫咪"}) print(state["combined_output"])

Google ADK — 搜索增强智能体

from google.adk.agents import LlmAgent from google.adk.tools import google_search # 定义带工具的智能体——ADK 隐式管理智能体循环 search_agent = LlmAgent( model="gemini-2.0-flash-exp", name="research_agent", description="使用网络搜索回答问题的智能体。", instruction="""使用 google_search 工具查找当前准确的信息， 然后基于搜索结果回答用户问题。""", tools=[google_search], ) # 智能体循环（观察 → 思考 → 行动）由框架隐式处理 response = search_agent.run("LangGraph 的最新进展是什么？")
框架横向对比深度解析

从关键维度对四个主要框架进行结构化比较。点击每个层级展开分析。

LangChain
核心抽象：链（LCEL）· 工作流：线性 DAG · 状态：每次运行无状态
LangChain 在最基础的层级运作，提供组件和标准化接口。LCEL 管道语法简洁易读。最适用场景：简单 RAG 管道、文本摘要链、从文本中提取数据。局限性是没有循环——无法基于结果回退或重试，这需要 LangGraph 来实现。
LangGraph
核心抽象：节点图 · 工作流：循环型 · 状态：显式持久对象
LangGraph 为 LangChain 增加了有状态的图运行时。开发者显式定义节点（函数）、边（转换）和条件边（分支逻辑）。TypedDict 状态是唯一的事实来源，由每个节点更新。这使循环智能体、规划执行模式和人在回路工作流成为可能。最适用：需要推理、使用工具和重试的复杂智能体。
Google ADK
核心抽象：LlmAgent + 子智能体 · 工作流：团队导向 · 状态：隐式管理
Google 的 ADK 通过预置智能体类型抽象了图构造细节。SequentialAgent 顺序运行子智能体；ParallelAgent 并行运行它们。主编排者智能体将任务委派给专家。状态和会话管理更为隐式，以更低的细粒度控制换取系统凝聚力。最适用：需要托管"智能体舰队"的生产级多智能体应用。
CrewAI
核心抽象：智能体 + 任务 + 团队 · 工作流：顺序或层级 · 状态：任务上下文传递
CrewAI 通过将协作建模为人类团队而占据独特位置。智能体拥有丰富的人物设定（角色、目标、背景故事）以指导行为。任务有明确的预期输出。开发者设计"团队章程"而非状态机。CrewAI 专注于智能体协作逻辑。最适用：模仿人类团队工作流的多智能体系统——研究、分析和写作管道。
其他框架：AutoGen、LlamaIndex、Haystack、MetaGPT、SuperAGI、Strands
针对特定场景的专用工具
AutoGen：对话驱动的多智能体协作，支持复杂问题分解，但执行路径可预测性较低。LlamaIndex：数据密集型应用和复杂 RAG 的首选。Haystack：生产就绪的模块化搜索系统。MetaGPT：SOP 驱动的智能体协作，模拟软件公司，擅长代码生成。SuperAGI：提供监控和可观测性的全生命周期管理平台。Strands Agents（AWS）：轻量级模型无关 SDK，原生支持 MCP，定制化智能体循环简洁灵活。

一览概要

LANGCHAIN

线性管道。当工作流是 A → B → C 无循环时使用。适合 RAG、摘要和信息提取。

LANGGRAPH

带循环的有状态图。当智能体需要循环、重试或动态条件控制流时使用。

GOOGLE ADK

预置多智能体模式。当你希望管理一组协作智能体而无需手动连接每条边时使用。

CREWAI

基于角色的团队协作。当应用场景类似人类团队——研究员、分析师、写作者各司其职时使用。

知识测验
核心要点
  • LangChain 的 LCEL 提供了简洁的管道语法，用于构建线性 DAG 式 LLM 工作流。它是可预测、顺序管道的正确选择。
  • LangGraph 在 LangChain 基础上增加了循环图和显式状态管理。当智能体需要循环、重试或基于中间结果做条件决策时使用。
  • LangChain vs. LangGraph 的关键判断：若流程可描述为 A → B → C 无循环，选 LangChain；若智能体需要推理、使用工具并可能重试，选 LangGraph。
  • Google ADK 通过提供预置智能体类型（SequentialAgent、ParallelAgent）抽象了底层图构造，用于编排专业智能体团队。
  • CrewAI 将多智能体协作建模为人类团队，每个成员有明确的角色、目标和背景故事。它聚焦于"谁做什么"，而非"图如何连接"。
  • 专业框架各有侧重：LlamaIndex 适合数据密集型 RAG；Haystack 适合生产级搜索；MetaGPT 适合 SOP 驱动的代码生成；Strands 适合轻量 AWS 原生智能体。
  • 所有框架的核心权衡是：细粒度控制（基于图）vs. 流线型开发（封装平台）。正确的选择取决于应用是否需要简单序列、动态推理循环或托管的专家智能体团队。
  • 智能体框架生态系统正在快速演进。最佳框架是与应用所需抽象层级精确匹配的那个——而未必是最流行的那个。
