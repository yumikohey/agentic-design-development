智能体间通信（A2A）

单个 AI 智能体面对复杂多面问题时往往力不从心。智能体间通信（A2A）协议使不同框架构建的 AI 智能体能够无缝协作，实现任务委派、信息共享与跨框架互操作。

本章概览

Google 的 A2A（Agent2Agent）协议是一个开放标准，专为实现不同 AI 智能体框架之间的通用通信而设计，支持 LangGraph、CrewAI、Google ADK 等框架间的无缝协作。

A2A：跨框架协作的开放标准

A2A 协议由 Google 开发，得到 Atlassian、Box、LangChain、MongoDB、Salesforce、SAP、ServiceNow 等技术公司的广泛支持。Microsoft 计划将 A2A 集成到 Azure AI Foundry 和 Copilot Studio，Auth0 和 SAP 也在将 A2A 支持整合至其平台与智能体产品中。

A2A 与 MCP 的互补关系

A2A 专注于智能体之间的高层任务协调与委派，而 Anthropic 的模型上下文协议（MCP）专注于为智能体提供结构化上下文以及与外部工具和数据的交互。两者相辅相成，共同构建完整的智能体生态。

开放源代码生态

作为开源协议，A2A 欢迎社区贡献，推动协议持续演进与广泛普及。代码仓库提供 Java、Go、Python 的示例实现，涵盖 LangGraph、CrewAI、Azure AI Foundry 等多种框架集成方案。

核心挑战

单个 AI 智能体在面对复杂、多面向问题时存在固有局限。点击各卡片了解具体挑战。

框架隔离问题

不同框架（LangGraph、CrewAI、ADK）构建的智能体无法直接通信。

每个框架都有专属通信格式与接口，使跨框架集成需要大量定制开发工作，严重阻碍多智能体系统的构建效率。

缺乏统一发现机制

智能体无法自动找到并了解其他智能体的能力边界。

没有标准化的能力描述格式，智能体必须依赖硬编码配置才能知道"哪个智能体能做什么"，极大限制了系统的灵活性与可扩展性。

集成成本高昂

为每对智能体定制集成接口，工程成本随规模指数级增长。

N 个智能体之间若两两定制集成，需要 N×(N-1)/2 个独立接口。这使得大规模多智能体系统在工程上几乎不可维护。

异步任务管理困难

长时间运行的任务缺乏标准化的状态追踪与中断恢复机制。

复杂工作流中，任务可能耗时数分钟乃至数小时。没有统一的任务状态模型，协调智能体无法可靠地追踪进度或处理失败情况。

A2A 协议架构

A2A 通过一套标准化的组件体系解决跨框架协作难题。点击各节点了解详细设计。

1
智能体名片（Agent Card）：数字身份标识
JSON 格式的能力声明文件
每个智能体都拥有一张名片（JSON 文件），定义其身份、服务端点 URL、版本号、支持的能力（流式传输、推送通知）、具体技能列表、默认输入/输出模式以及身份验证要求。其他智能体通过读取名片即可自动发现并理解该智能体的能力。
2
智能体发现：三种定位策略
Well-Known URI / 注册表 / 直接配置
客户端可通过三种方式发现 A2A 服务端：(1) 知名 URI——智能体在标准路径（如 /.well-known/agent.json）托管名片，适合公开或域级服务；(2) 集中注册表——企业环境下发布和查询名片的中心目录，支持访问控制；(3) 直接配置——将名片信息嵌入或私下共享，适合紧耦合私有系统。
3
任务与消息：通信的基本单元
异步任务 + JSON-RPC 2.0
A2A 的通信围绕异步任务构建——每个任务获得唯一标识符，经历"已提交→处理中→已完成"等状态转换，支持复杂操作的并行处理。消息包含键值元数据属性和承载实际内容的消息段（文本、文件或 JSON 数据）。任务产出的有形结果称为制品（Artifact），可增量流式传输。所有通信基于 HTTP(S) + JSON-RPC 2.0，通过服务端生成的 contextId 维护跨任务的会话连续性。
4
安全机制：多层防护体系
mTLS + 审计日志 + OAuth 2.0
A2A 内置多层安全机制：相互传输层安全（mTLS）加密认证连接，防止未授权访问与数据截获；全面审计日志记录所有智能体间通信细节，支持问责与安全分析；名片声明认证需求（OAuth 2.0、API 密钥等），通过 HTTP 请求头传递凭证，避免在 URL 或消息体中暴露敏感信息。

核心概念详解

深入理解 A2A 协议的关键组成要素与交互机制。

三类核心角色

  • 用户（User）：发起智能体辅助请求的人或系统
  • A2A 客户端（Client Agent）：代表用户行事的应用程序或 AI 智能体，负责发出请求或获取信息
  • A2A 服务端（Remote Agent）：提供 HTTP 端点处理客户端请求并返回结果的 AI 智能体或系统。服务端作为"不透明"系统运行，客户端无需了解其内部实现细节

四种交互机制

  • 同步请求/响应：适用于快速即时操作。客户端发送请求并等待服务端在单次同步交换中返回完整响应（sendTask 方法）
  • 异步轮询：适用于耗时较长的任务。服务端立即返回"处理中"状态和任务 ID，客户端定期轮询检查状态直至完成或失败
  • 流式更新（SSE）：通过服务器推送事件建立持久单向连接，服务端持续推送状态变更或部分结果，无需客户端多次请求（sendTaskSubscribe 方法）
  • 推送通知（Webhook）：适用于极长时间或资源密集型任务。客户端注册 Webhook URL，任务状态发生重大变化时服务端主动通知

智能体名片示例（WeatherBot）

名片完整声明了智能体的身份、端点、能力与技能：

实际应用场景

智能体间通信是构建跨领域复杂 AI 解决方案的基础，支撑模块化、可扩展的系统架构。

多框架智能体协作

A2A 的核心使用场景：使不同底层框架（ADK、LangGraph、CrewAI）构建的独立 AI 智能体能够无障碍通信与协作。

1
各智能体发布各自的名片，声明技能与能力
2
主控智能体通过 Well-Known URI 发现可用的专业智能体
3
将任务的不同部分委派给最适合的专业智能体处理
4
通过标准 A2A 消息格式汇聚各智能体的处理结果

企业自动化工作流编排

在企业环境中，A2A 可支持复杂工作流的智能体协作：一个智能体负责初始数据收集，将分析任务委派给另一个智能体，最终由第三个智能体生成报告，全程通过 A2A 协议通信。

1
数据采集智能体通过 API 收集原始业务数据
2
通过 A2A 将数据集及分析指令传递给数据分析智能体
3
分析智能体处理完成后，结果通过流式 SSE 实时推送
4
报告生成智能体接收分析结果，输出最终格式化报告

动态实时信息检索

智能体之间通过通信实时检索和交换信息。主控智能体可向专业的"数据获取智能体"请求实时市场行情，该智能体调用外部 API 获取数据后回传。

1
主控智能体识别需要外部实时数据的子任务
2
查找注册表中专精于特定数据源的智能体
3
通过 A2A 同步请求获取即时数据（适合低延迟场景）
4
接收结构化 JSON 数据并融入当前推理上下文

Google 日历管理智能体

A2A 官方示例：基于 Google ADK 的日历智能体，通过 A2A 协议对外暴露日历管理服务，支持检查用户可用时间、创建与修改日历事件。

1
智能体名片声明 check_availability 等技能及 Google OAuth 认证要求
2
客户端通过 A2A 任务请求用户某时段的日历可用性
3
ADK 智能体调用 Google Calendar API 获取实时数据
4
结果通过流式 SSE 或同步响应返回客户端智能体

分布式推理与专业智能体网络

A2A 支持构建由专业智能体组成的分布式网络，各智能体独立运行在不同端口，系统整体实现高可扩展性与分布式部署。

1
部署专注于不同领域的专业智能体（法律、财务、技术等）
2
路由智能体根据查询内容动态选择最适合的专业智能体
3
多个专业智能体并行处理复杂问题的不同子任务
4
聚合智能体整合多路响应，生成统一的综合答复

ADK 日历智能体：名片与服务器配置

下方代码演示构建 A2A 兼容的"日历智能体"：定义名片声明技能，配置 ADK 运行器，初始化 Starlette 应用并通过 Uvicorn 暴露 HTTP 服务。

import datetime from google.adk.agents import LlmAgent from google.adk.tools.google_api_tool import CalendarToolset async def create_agent(client_id, client_secret) -> LlmAgent: """构建 ADK 日历智能体""" toolset = CalendarToolset( client_id=client_id, client_secret=client_secret ) return LlmAgent( model='gemini-2.0-flash-001', name='calendar_agent', description="能够管理用户日历的智能体", instruction=f"""你是一个能够帮助管理用户日历的智能体。 用户将查询其日历状态或请求变更。 请使用提供的工具与 Calendar API 交互。 今天是 {datetime.datetime.now()}。""", tools=await toolset.get_tools(), ) def main(host: str, port: int): # 定义技能与名片 skill = AgentSkill( id='check_availability', name='检查日历可用性', description="使用 Google 日历检查用户特定时段的可用性", tags=['calendar'], examples=['明天上午 10 点到 11 点我有空吗？'], ) agent_card = AgentCard( name='Calendar Agent', url=f'http://{host}:{port}/', version='1.0.0', capabilities=AgentCapabilities(streaming=True), skills=[skill], ) # 初始化运行器并启动 A2A 服务 runner = Runner( app_name=agent_card.name, agent=adk_agent, artifact_service=InMemoryArtifactService(), session_service=InMemorySessionService(), memory_service=InMemoryMemoryService(), ) a2a_app = A2AStarletteApplication( agent_card=agent_card, http_handler=request_handler ) uvicorn.run(a2a_app, host=host, port=port)

深入解析：A2A 协议五大支柱

A2A 协议建立在五个相互支撑的核心概念之上。点击各层深入了解。

支柱一：核心角色体系
用户 → A2A 客户端 → A2A 服务端
A2A 的角色体系清晰划分职责：用户（真实人类或触发系统）→ A2A 客户端（代理用户行动，发出请求）→ A2A 服务端（提供 HTTP 端点，作为不透明系统处理请求）。这种分层设计确保了松耦合，服务端可自由升级内部实现而无需通知客户端。
支柱二：智能体名片机制
能力声明与自动发现的核心载体
名片是 A2A 生态的"通行证"，以标准化 JSON 格式声明智能体的完整数字身份：名称与描述、服务端点 URL、版本号、流式传输与推送通知能力、具体技能（含示例查询与标签）、默认 I/O 模式、认证方案。无论采用 Well-Known URI、集中注册表还是直接配置，发现过程的核心始终是读取与解析名片。
支柱三：任务生命周期管理
异步任务状态机设计
A2A 的任务是一级公民，拥有完整的生命周期管理：已提交（submitted）→ 处理中（working）→ 需要输入（input-required，支持多轮对话）→ 已完成（completed）/ 已失败（failed）/ 已取消（cancelled）。每个任务有唯一 ID，通过服务端生成的 contextId 将相关任务分组，维护跨多次交互的会话连续性。制品（Artifact）作为任务产出，可随处理进度增量流式传输。
支柱四：多模态交互机制
A2A 的模态无关设计
A2A 采用模态无关设计，不仅支持文本，还支持音频、视频等多种数据类型，为构建富媒体多模态 AI 应用奠定基础。四种交互机制（同步、异步轮询、SSE 流式、Webhook 推送）分别针对不同的响应时延与资源特性场景，由名片中的 capabilities 字段声明支持情况。
支柱五：A2A 与 MCP 的协同关系
互补而非竞争的两大协议
A2A 与 Anthropic 的 MCP（模型上下文协议）各司其职：MCP 专注于为单个智能体提供结构化的工具与外部数据访问接口（浏览器、文件系统、数据库、API）；A2A 专注于智能体之间的高层协调与任务委派。两者协同，MCP 解决"智能体如何使用工具"，A2A 解决"智能体如何合作"，共同构建完整的分层智能体生态系统。

一览概要

本章核心要义速查：问题、解决方案与适用场景一览。

问题（WHAT）

不同框架构建的 AI 智能体孤立运作，缺乏通用语言进行有效协作。这种隔离阻碍了多个专业智能体协同解决更大规模任务的系统架构。

解决方案（WHY）

A2A 协议提供基于 HTTP 的开放标准，通过智能体名片实现能力发现，通过标准任务模型支持异步协作，通过 mTLS 保障通信安全，使不同框架的智能体能够无缝互操作。

适用场景（WHEN）

当需要协调两个或更多 AI 智能体的协作时，尤其是当它们基于不同框架构建。特别适合构建模块化复杂应用，如将数据分析委派给一个智能体、报告生成委派给另一个。

关键洞察（KEY INSIGHT）

A2A 通过模块化与可扩展的生态为开发者赋能：智能体作为独立服务部署，通过标准协议即插即用，大幅降低集成成本，推动 AI 系统向真正的分布式、协作式架构演进。

核心要点

本章最重要的八大洞察与结论。

  • Google A2A 协议是一个基于 HTTP 的开放标准，专为实现不同框架（ADK、LangGraph、CrewAI 等）构建的 AI 智能体之间的通信与协作而设计。
  • 智能体名片（Agent Card）是 A2A 生态的核心标识，以 JSON 文件描述智能体的身份、能力、技能与认证要求，支持自动发现与交互理解。
  • A2A 同时提供同步请求/响应（sendTask）和流式更新（sendTaskSubscribe）两种交互模式，满足从即时查询到长时间运行任务的多样化通信需求。
  • 协议支持多轮对话的 input-required 状态，使智能体在执行任务过程中能够请求补充信息，保持交互上下文的连续性。
  • A2A 推崇模块化架构，各专业智能体可独立部署在不同端口，通过标准协议即插即用，实现系统的高可扩展性与分布式部署。
  • A2A 内置多层安全机制，包括 mTLS 加密认证连接、全面审计日志以及通过名片显式声明的认证要求，确保智能体间通信的安全性与可追溯性。
  • A2A 与 MCP 是互补关系而非竞争：MCP 解决智能体如何访问外部工具与数据，A2A 解决智能体之间如何协作与委派任务，两者共同构建完整的分层智能体生态。
  • Trickle AI 等工具提供 A2A 通信的可视化与追踪能力，帮助开发者监控、调试和优化多智能体系统的协作行为。
