第十九章

评估与监控

本章研究智能代理系统地评估其性能、监控目标进展并检测异常行为的方法论——通过指标定义、反馈回路与报告系统确保代理在生产环境中的持续可靠性。

模式概览

智能代理系统在复杂、动态的环境中运行，其性能会随时间推移而下降。传统软件测试不足以保障可靠性，因此需要持续的、多维度的评估框架来衡量代理的有效性、效率与合规性。

本章核心问题

如何在生产环境中系统性地评估 AI 代理的准确率、延迟与资源消耗？如何检测性能漂移、异常行为，并对代理轨迹进行定性评估？

评估的特殊挑战

AI 代理的概率性与非确定性特质意味着：同一输入可能产生不同输出。传统的通过/失败测试无法捕捉代理行为的细微之处，必须同时评估最终结果与决策轨迹。

与第 11、17 章的关系

第 11 章介绍目标设定与监控，第 17 章讨论推理机制，本章则专注于持续的、外部视角的性能测量——定义指标、建立反馈回路、实施报告系统。

最佳实践五层金字塔

定义清晰可测的目标与指标 → 结合定量与定性数据 → 定期一致地收集数据 → 激励与奖励代理 → 提供反馈与指导。

问题所在

在复杂的生产环境中部署 AI 代理时，若缺乏系统性评估框架，将面临多重风险。点击各卡片了解详情。

性能静默退化

代理在部署后性能逐渐下降，但无告警机制。

输入数据分布发生变化（概念漂移）或环境改变时，代理精度可能悄然下滑。若无持续监控，直到用户投诉才能察觉，造成重大业务损失。

无法量化实际价值

缺乏指标体系，难以证明代理创造了真实价值。

若不追踪准确率、响应时间、任务完成率等关键指标，则无法对比不同版本、策略或模型的优劣，A/B 测试也无从开展。

安全与合规盲区

代理行为偏离伦理准则或法规要求而未被发现。

在金融、医疗、法律等受监管行业，代理必须符合特定合规要求。若无自动审计报告，合规违规可能积累至严重程度才被发现。

异常行为未被识别

恶意攻击或涌现的不期望行为无法被及时检测。

代理可能受到提示注入攻击，或因训练数据问题产生偏见性输出。若无异常检测机制，这些行为将持续对真实用户产生影响。

多代理系统评估失效

传统单元测试无法评估多代理协作的整体质量。

当多个代理协同完成任务时，需要超越个体性能度量的协作指标——代理间通信质量、任务分配合理性、系统整体效率等都需要专门的评估方法。

解决方案：系统化评估框架

建立一套覆盖指标定义、数据收集、分析方法、反馈回路的完整评估体系。点击各节点查看详细说明。

1
定义评估指标
明确可测量的目标与 KPI
根据业务目标选择指标：准确率（任务完成质量）、延迟（响应速度）、Token 用量（成本控制）、目标达成率（战略对齐度）。指标需同时涵盖定量（数值）与定性（主观质量）两个维度。
2
构建数据收集管道
持续、一致地记录运行数据
将监控数据持久化存储至结构化日志（JSON）、时序数据库（InfluxDB、Prometheus）、数据仓库（BigQuery）或可观测性平台（Datadog、Grafana Cloud）。避免仅打印至控制台。
3
实施多维度评估方法
结合自动化指标与 LLM-as-a-Judge
基础评估：精确字符串匹配；进阶方法：编辑距离、Jaccard 相似度、关键词分析、余弦相似度语义匹配；主观质量：LLM 作为裁判对帮助性、相关性进行打分；RAG 指标：忠实度与相关性评估。
4
建立反馈与告警机制
驱动持续改进与异常响应
设定阈值触发告警，识别漂移与异常。A/B 测试不同策略，为代理学习提供反馈信号。构建人在回路的审核流程，处理边缘案例。报告系统支持合规审计与透明度要求。
核心概念

深入理解评估与监控的关键技术概念。

响应准确率评估

最基础的评估核心是判断代理输出是否与预期输出匹配。简单的精确匹配（去除空白、大小写不敏感比较）适合结构化输出，但对自然语言响应会产生误判——语义等价但措辞不同的答案会被错误标记为失败。

语义相似度度量

超越字符串匹配，使用 NLP 技术理解句子间的含义。方法包括：Levenshtein 编辑距离（字符级差异）、Jaccard 相似度（词集合重叠度）、嵌入模型的余弦相似度（语义层面匹配）。

LLM-as-a-Judge（LLM 作为裁判）

用另一个 LLM 评估代理输出的主观质量，如帮助性、中立性、相关性、完整性、适当性。裁判 LLM 根据预定义评分标准（Rubric）给出 1-5 分及详细理由。该方法一致、高效、可扩展，但受裁判 LLM 自身能力限制。

{ "overall_score": 4, "rationale": "回答准确且完整，措辞中立", "detailed_feedback": { "clarity": 5, "neutrality": 4, "relevance": 4, "completeness": 3 }, "recommended_action": "按原样批准" }

延迟监控与 Token 追踪

延迟是实时应用的关键指标——测量代理处理请求到生成输出的耗时。Token 追踪对 LLM 驱动的代理至关重要：输入/输出 Token 数量直接决定 API 调用成本，优化 Token 使用可直接降低运营支出。

代理轨迹评估

轨迹是代理达成目标所采取的步骤序列。评估轨迹需要将实际行动与理想的"真值轨迹"对比，匹配策略包括：精确匹配（完美序列）、顺序匹配（允许额外步骤）、任意顺序匹配、精确率（预测相关性）、召回率（关键动作覆盖率）。

应用场景

评估与监控模式在多种真实场景中不可或缺。选择场景查看详细说明。

生产环境实时性能追踪

在生产环境中持续监控代理的准确率、延迟和资源消耗。例如，客服聊天机器人需要实时追踪问题解决率和平均响应时间。当指标低于阈值时自动触发告警，运维团队可立即响应。

1
部署采集探针，记录每次代理交互的输入、输出、耗时
2
将数据写入时序数据库（Prometheus / InfluxDB）
3
Grafana 仪表板可视化关键指标，设置异常告警
4
定期回顾报告，识别改进机会

代理改进的 A/B 测试

系统性地并行比较不同代理版本或策略，识别最优方案。例如，为物流代理测试两种不同的规划算法，根据完成率、成本效益和延迟来选择更优版本。

1
定义对照组（A）和实验组（B），明确评估指标
2
随机将请求路由至两组，保证流量均等
3
收集足够样本后进行统计显著性检验
4
根据结果决定是否全量推广新版本

合规与安全审计

自动生成追踪代理合规情况的审计报告，涵盖伦理准则、监管要求和安全协议。报告可由人工审核员或另一个代理验证，并在发现问题时生成 KPI 或触发告警。企业系统可使用 AI"合同"机制明确代理的目标、规则与控制权限。

漂移检测

随时间推移监控代理输出的相关性与准确率，检测因输入数据分布变化（概念漂移）或环境变化导致的性能退化。例如，当新闻摘要代理的质量评分持续下滑时，系统自动标记并启动模型更新流程。

1
建立基线性能指标（初始部署时）
2
设置滑动窗口监控，检测统计分布变化
3
超出预设阈值时自动触发重新评估或再训练流程

多代理系统评估

评估复杂多代理系统如同评估团队项目，需要在各阶段检查工作质量，同时评估整体系统性能。关键问题：代理间是否有效协作？任务分配是否合理？添加新代理是否提升整体性能还是引入冲突？

  • 检验航班代理与酒店代理的日期/目的地传递是否正确
  • 评估计划坚守度——代理是否按预定序列执行
  • 验证任务路由——是否为每类任务选择了最合适的代理
  • 测试可扩展性——增加代理数量是否线性提升性能

学习进度评估

对具有学习能力的代理，追踪其学习曲线、特定技能提升情况和跨任务泛化能力。记录代理在不同时间点的性能快照，生成学习进度报告，识别需要额外训练数据的薄弱环节。

def evaluate_response_accuracy(agent_output: str, expected_output: str) -> float: """计算代理响应的基础准确率分数""" # 精确匹配（大小写不敏感，去除首尾空白） return 1.0 if agent_output.strip().lower() == expected_output.strip().lower() else 0.0 # 示例使用 agent_response = "法国的首都是巴黎。" ground_truth = "巴黎是法国的首都。" score = evaluate_response_accuracy(agent_response, ground_truth) # score = 0.0（尽管语义相同，字符串不匹配） # 局限性：无法处理语义等价但措辞不同的情况 # 改进方向：使用嵌入模型计算余弦相似度

示例二：LLM 交互的 Token 用量追踪

class LLMInteractionMonitor: """追踪 LLM 交互中的 Token 用量""" def __init__(self): self.total_input_tokens = 0 self.total_output_tokens = 0 def record_interaction(self, prompt: str, response: str): # 实际生产中应使用 LLM API 的 token 计数器或 tokenizer input_tokens = len(prompt.split()) # 占位实现 output_tokens = len(response.split()) self.total_input_tokens += input_tokens self.total_output_tokens += output_tokens def get_total_tokens(self): return self.total_input_tokens, self.total_output_tokens # 使用示例 monitor = LLMInteractionMonitor() monitor.record_interaction("法国的首都是哪里？", "法国的首都是巴黎。") monitor.record_interaction("讲个笑话", "为什么科学家不信任原子？因为它们构成了一切！") input_t, output_t = monitor.get_total_tokens() # 成本控制：Token 总量 × 每千 Token 单价 = 运营成本

示例三：LLM 作为裁判——评估"帮助性"

import google.generativeai as genai import json, logging from typing import Optional # 评分标准 Rubric（法律调查质量示例） EVAL_RUBRIC = """你是专业的质量评估员。 按以下五个维度对输出评分（1-5分）： 1. 清晰度与精确性 2. 中立性与无偏见 3. 相关性与聚焦度 4. 完整性 5. 受众适当性 以 JSON 格式返回：overall_score, rationale, detailed_feedback""" class LLMJudge: def __init__(self, model_name='gemini-1.5-flash-latest', temperature=0.2): self.model = genai.GenerativeModel(model_name) self.temperature = temperature def judge(self, content: str) -> Optional[dict]: prompt = f"{EVAL_RUBRIC}\n\n待评估内容：\n{content}" try: response = self.model.generate_content( prompt, generation_config=genai.types.GenerationConfig( temperature=self.temperature, response_mime_type="application/json" ) ) return json.loads(response.text) except Exception as e: logging.error(f"裁判 LLM 评估失败：{e}") return None
深入解析：评估方法体系

评估方法从简单到复杂形成体系。点击各层展开详细分析。

第一层：精确字符串匹配
最简单，适用于结构化输出
精确匹配（区分大小写）或规范化后匹配（去除空白、统一大小写）。返回二值结果（1.0 正确 / 0.0 错误）。优点：快速、无歧义；缺点：无法处理同义表达，对自然语言高度敏感。
第二层：字符串相似度度量
允许部分匹配
Levenshtein 编辑距离：计算将一个字符串转换为另一个字符串所需的最少编辑操作数（插入、删除、替换）。Jaccard 相似度：两个词集合的交集除以并集。这些方法可捕捉拼写变体和词序调整。
第三层：关键词分析
检查特定词汇的存在性
检测响应中是否存在或缺失特定关键词。适用于需要验证特定专业术语、实体名称或数值的场景。可以配合正则表达式使用，支持模式匹配。局限：忽略词语在不同语境中的语义差异。
第四层：语义相似度（嵌入模型）
理解语义层面的含义
将文本编码为高维向量，计算余弦相似度。使用 sentence-transformers 等模型。能够识别"法国的首都是巴黎"与"巴黎是法国的首都"语义等价。计算成本较高，但对自然语言评估更准确。
第五层：LLM-as-a-Judge + RAG 指标
评估主观质量与检索增强指标
LLM 裁判方法对主观质量进行细粒度评估，包括帮助性、礼貌性、安全性等。RAG 特定指标包括：忠实度（生成内容是否基于检索上下文）和相关性（检索内容是否与问题相关）。这是目前最强大但计算成本最高的评估层级。

三种主要评估方法对比

人工评估：能捕捉细微行为，但难以扩展、成本高昂、耗时长，且受主观因素影响。

LLM-as-a-Judge：一致、高效、可扩展，但可能忽略中间步骤，受 LLM 自身能力限制。

自动化指标：可扩展、高效、客观，但在捕捉完整能力方面存在潜在局限。

一览概要

是什么（WHAT）

智能代理在复杂动态环境中运行，性能随时间退化。概率性与非确定性特质使传统测试失效。需要持续评估框架来衡量有效性、效率与目标对齐度。

为什么（WHY）

标准化评估框架提供系统性方法：定义准确率、延迟、Token 用量等指标；分析轨迹理解推理过程；LLM 裁判进行细粒度评估；反馈回路实现持续改进。

经验法则

在以下情况使用此模式：生产部署需要实时可靠性保障；需要系统性 A/B 测试；在合规/安全敏感领域运营；性能可能随数据漂移而退化；需要评估复杂代理轨迹或主观质量。

Google ADK 支持

三种评估方式：Web UI（adk web）交互式评估；pytest 集成用于 CI/CD 管道；命令行（adk eval）自动化评估。测试文件（单次交互）适合单元测试；Evalset 文件适合集成测试。

知识测验
核心要点
  • 评估智能代理需要超越传统测试，在真实环境中持续衡量有效性、效率与合规性，而非简单的通过/失败判断。
  • 代理评估的实际应用包括：生产环境性能追踪、A/B 测试改进、合规审计，以及行为漂移与异常检测。
  • 基础评估涉及响应准确率，真实场景还需要延迟监控和 Token 用量追踪等更复杂的指标来驱动成本优化。
  • 代理轨迹——达成目标所采取的步骤序列——对评估至关重要，需将实际行动与理想真值路径对比以识别错误与低效。
  • Google ADK 通过测试文件（单元测试）和 Evalset 文件（集成测试）提供结构化评估方法，均可定义预期代理行为。
  • 代理评估可通过 Web UI 交互式测试、pytest CI/CD 集成或命令行自动化工作流三种方式执行。
  • 要使 AI 在高风险任务中可靠，必须从简单提示迁移到精确定义可验证交付物的正式"合同"，使代理能够协商、澄清歧义并迭代验证自身工作。
  • 多代理系统的评估需要超越个体性能，关注协作质量、任务分配合理性与系统整体效率。
