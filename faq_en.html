What is an "agentic design pattern"?+
An agentic design pattern is a reusable, high-level solution to a common problem encountered when building intelligent, autonomous systems (agents). These patterns provide a structured framework for designing agent behaviors, much like software design patterns do for traditional programming. They help developers build more robust, predictable, and effective AI agents.
What is the main goal of this guide?+
The guide aims to provide a practical, hands-on introduction to designing and building agentic systems. It moves beyond theoretical discussions to offer concrete architectural blueprints that developers can use to create agents capable of complex, goal-oriented behavior in a reliable way.
Who is the intended audience?+
The guide is written for AI developers, software engineers, and system architects who are building applications with large language models (LLMs) and other AI components. It is for those who want to move from simple prompt-response interactions to creating sophisticated, autonomous agents.
What are the key agentic patterns covered?+
The guide covers 21 patterns including: Reflection (self-critique), Planning (goal decomposition), Tool Use (external APIs), Multi-Agent Collaboration (specialist teams), Human-in-the-Loop (oversight), Memory Management, Knowledge Retrieval (RAG), Routing, Parallelization, Exception Handling, Guardrails/Safety, and more.
Common Misconceptions
These are the most frequent misunderstandings developers have when first approaching agentic systems. Click each to reveal the clarification.

"Agents are just chatbots"

Wrong—agents actively plan and act

A chatbot responds to prompts. An agent perceives its environment, makes decisions, uses tools, maintains state, and takes multi-step actions to achieve goals—far beyond single-turn question-answering.

"More autonomy is always better"

Wrong—autonomy requires proportional safety

Greater autonomy demands greater safety infrastructure. Without Guardrails and Human-in-the-Loop patterns, a highly autonomous agent can take harmful or irreversible real-world actions. The HITL pattern is especially critical for high-stakes tasks.

"Prompt chaining = full agent"

Wrong—it's one foundational pattern of many

Prompt chaining is the simplest agentic pattern—a linear pipeline. A production-grade agent requires tool use, memory, reflection, and error handling layered on top. Chaining alone cannot handle dynamic, stateful, or multi-turn scenarios well.

"Reflection is just re-prompting"

Wrong—it requires structured critique

Simply asking the model to "check your work" often produces rephrasing without improvement. True Reflection requires a structured critique against specific criteria—ideally by a separate Critic agent—leading to genuine quality improvements, not just stylistic edits.

Is constant human intervention inefficient?+
It can be, which is why the key is finding the right balance. HITL should be implemented at critical checkpoints—plan approval, tool use confirmation, ambiguity resolution, final output review—not for every single action. The goal is a collaborative partnership where the agent handles the bulk of work and the human provides strategic guidance.
Pattern-Specific FAQs
Detailed questions and answers about the key patterns in the book. Click any question to expand.
Reflection Pattern
How does Reflection reduce hallucinations?+
By prompting an agent to review its work—especially comparing statements against a known source or checking reasoning steps—the Reflection pattern forces the agent to be more grounded in provided context and less likely to generate unsupported information. The agent is explicitly prompted to verify its claims rather than accepting its first output as final.
What are the two main types of Reflection?+
"Check your work" Reflection — a basic form where the agent is asked to review and fix its previous output. Good for catching simple errors. "Internal Critic" Reflection — a more advanced form using a separate "critic" agent or dedicated prompt to evaluate the "worker" agent's output against specific criteria, leading to more rigorous and targeted improvements.
Planning Pattern
Why is planning necessary for complex tasks?+
LLMs can struggle with tasks requiring multiple steps or dependencies. Without a plan, an agent might lose track of the overall objective, miss crucial steps, or fail to handle the output of one step as the input for the next. A plan provides a clear roadmap, ensuring all requirements of the original request are met in a logical order.
How does an agent handle errors during plan execution?+
A robust planning pattern allows for dynamic adjustments. If a step fails or the situation changes, the agent can be prompted to "re-plan" from the current state. It analyzes the error, modifies the remaining steps, or adds new ones to overcome the obstacle. This is complemented by the Exception Handling and Recovery pattern (Chapter 12).
Multi-Agent Collaboration
What is the Orchestrator Agent?+
A common multi-agent architecture involves an Orchestrator Agent (sometimes called "manager" or "conductor"). The orchestrator understands the overall goal, breaks it down, and delegates sub-tasks to appropriate specialist agents. It then collects results from specialists and synthesizes them into a final output. This is the primary architectural pattern explored in Chapter 7.
How do agents communicate with each other?+
Communication is often managed by the orchestrator—passing the output of one agent as context to the next. A shared "scratchpad" or message bus where agents post their findings is another common method. The Inter-Agent Communication (A2A) and Model Context Protocol (MCP) patterns provide standardized approaches for this communication.
Conceptual FAQs
Questions about the underlying concepts, terminology, and mental models for agentic systems.
What is the difference between a "tool" and a "skill"?+
A tool generally refers to an external resource the agent can call upon (e.g., a weather API, a calculator). A skill is a more integrated capability combining tool use with internal reasoning to perform a specific function (e.g., "booking a flight" combines calendar and airline APIs with planning logic). Skills are higher-order capabilities built from tools.
What is "ReAct" (Reason and Act)?+
ReAct is a popular framework that integrates reasoning and acting. The agent follows a loop of Thought (reasoning about what to do), Action (deciding which tool to use and with what inputs), and Observation (seeing the tool result). This loop continues until enough information is gathered. It is foundational to the Tool Use and Planning patterns.
What is an "agent trajectory"?+
An agent trajectory is the complete log of an agent's steps while performing a task—all its thoughts, actions (tool calls), and observations. Analyzing trajectories is a key part of debugging and understanding agent behavior. It is the primary artifact used in the Evaluation and Monitoring pattern (Chapter 19).
What is "prompt leakage"?+
Prompt leakage occurs when parts of the system prompt (tool definitions, internal instructions) are inadvertently revealed in the agent's final response to the user. This can expose implementation details and confuse users. Prevention techniques include using separate prompts for reasoning vs. final answer generation, and explicit instructions to not repeat system prompt content.
What are the four levels of agent complexity?+
Level 0 (Core Reasoning Engine): LLM with no tools, memory, or environment interaction. Level 1 (Connected Problem-Solver): LLM with tool use and RAG for live information. Level 2 (Strategic Problem-Solver): Tool use + planning + context engineering + self-improvement. Level 3 (Collaborative Multi-Agent): Multiple specialized agents orchestrated to tackle complex workflows.
Practical FAQs
Questions about applying these patterns in real-world systems. Select a use-case area.
Which pattern should I start with?+
Start with Prompt Chaining (Chapter 1) for any multi-step task—it is the simplest foundational pattern. Add Tool Use (Chapter 5) when your agent needs live information or external actions. Add Memory Management (Chapter 8) when context must persist across turns. Add Reflection (Chapter 4) when output quality is insufficient. Layer in multi-agent only when single-agent approaches hit a ceiling.
What frameworks does the book use?+
The book uses three primary frameworks: LangChain (with LangGraph) for flexible chaining and graph-based agent workflows; CrewAI for multi-agent orchestration with roles and tasks; and Google Agent Developer Kit (ADK) for production-grade agents integrated with Google's AI infrastructure. Each represents a different "canvas" on which patterns are applied.
How does an agent decide which tool to use?+
The agent is given a list of available tools with descriptions of what each does and what parameters it requires. When faced with a request it cannot handle with internal knowledge alone, the agent's reasoning ability allows it to select the most appropriate tool. The Routing pattern (Chapter 2) can add explicit conditional logic on top of this selection process.
What are the key challenges in implementing Tool Use?+
Error Handling: Tools can fail, return unexpected data, or time out—the agent must recognize these and decide to retry, use a different tool, or ask for help. Security: Giving tool access has serious implications; safeguards, permissions, and human approval for sensitive operations are essential. Prompting: The agent must be prompted to generate correctly formatted tool calls with the right function name and parameters.
When should I use a multi-agent system vs. a single agent?+
Use multi-agent when: (1) the task requires genuinely different expertise that cannot be captured in a single system prompt; (2) the workflow is too complex for one agent to track without losing context; (3) parallel execution of independent subtasks would significantly speed things up; (4) quality control benefits from a separate critic perspective. Start single-agent and add complexity only when needed.
What are the benefits of multi-agent systems?+
Modularity and Specialization: Each agent can be fine-tuned for its specific task (researcher, writer, critic), leading to higher quality. Reduced Complexity: Breaking complex workflows into specialized roles makes the system easier to design, debug, and maintain. Simulated Brainstorming: Different agents offering different perspectives can produce more creative and robust solutions, similar to how human teams work.
Why is evaluating an agent harder than evaluating traditional software?+
Traditional software has deterministic outputs (same input → same output). Agents using LLMs are non-deterministic and their performance is often subjective. Evaluating them requires assessing the quality and relevance of output, not just whether it's technically "correct." This requires new evaluation methodologies: outcome-based, process-based, and human evaluation working together.
How do you create reliable tests for a non-deterministic system?+
While you cannot guarantee exact output wording, you can write tests that check for key elements: does the final response contain specific required information? Did the agent successfully call a certain tool with correct parameters? Did it complete the required steps? This is often done using mock tools in a dedicated testing environment. LLM-as-a-Judge (Chapter 19) provides an automated scoring approach.
Implementation FAQs
Technical questions about building agents, including key code patterns.
How is prompting an agent different from a simple ChatGPT prompt?+
Prompting an agent involves creating a detailed "system prompt" or constitution that acts as its operating instructions. This goes beyond a single user query—it defines the agent's role, its available tools, the patterns it should follow (like ReAct or Planning), its constraints, and its personality. The system prompt is effectively the agent's "programming."
What does a good system prompt include?+
A strong system prompt typically includes: Role and Goal (who the agent is and its primary purpose); Tool Definitions (available tools with descriptions and usage format); Constraints and Rules (what the agent should NOT do); Process Instructions (which patterns to follow—"First create a plan, then execute step-by-step"); Example Trajectories (a few successful thought-action-observation loops significantly improve reliability).

Minimal System Prompt Structure

The structure below illustrates the key components of a production agent system prompt.

# Minimal agent system prompt structure SYSTEM_PROMPT = """ You are a research assistant agent. Your goal is to answer user questions by gathering information from multiple sources. ## Available Tools - search(query: str) -> str: Search the web for information - fetch_doc(url: str) -> str: Retrieve content from a URL - summarize(text: str) -> str: Summarize a long document ## Process 1. Analyze the user's request 2. Create a brief plan (numbered steps) 3. Execute each step using the appropriate tool 4. Synthesize your findings into a final answer ## Constraints - Always cite your sources - Do not fabricate statistics or quotes - If a tool fails, try an alternative approach - Request clarification if the goal is ambiguous ## Format Think through your reasoning step-by-step before answering. """
How do I structure tool responses back to the agent?+
Tool responses should be formatted consistently—structured JSON when possible. Include the tool name, the input parameters used, the output, and any error status. Clear structure helps the agent correctly interpret the observation before deciding its next thought/action. Schemas should be defined in the system prompt or provided via function-calling APIs like OpenAI function calling or Google function declarations.
Advanced FAQs
Questions for developers building production-grade agentic systems at scale.
What are future trends in agentic systems?+
The guide points toward: More Autonomous Agents requiring less human intervention and learning to adapt on their own; Highly Specialized Agents available as a service in open ecosystems (the "agent economy"); Better Tools and Platforms with more sophisticated frameworks for building, testing, and deploying robust multi-agent systems; and New Safety Patterns addressing alignment, robustness, and adversarial scenarios.
What is the Model Context Protocol (MCP)?+
MCP is an emerging standard (developed by Anthropic) for how agents and tools exchange information. Rather than each integration requiring custom code, MCP defines a standardized protocol for tool discovery, invocation, and response. This is foundational to the "agentic ecosystems" vision—enabling agents to discover and use tools they were not explicitly programmed to work with.
How does the Guardrails/Safety Patterns chapter address safety?+
Chapter 18 covers: Input Validation/Sanitization to prevent injection attacks; Output Filtering/Post-processing to block harmful content; Behavioral Constraints via explicit system prompt rules; Callbacks for monitoring each agent action; Separation of Concerns to limit agent permissions to what is needed; Checkpoint and Rollback for recoverable failures; and External Moderation APIs as a final safety layer.
What is "Context Engineering" and why does it matter?+
Context engineering is the strategic process of selecting, packaging, and managing the most relevant information for each step in an agent's workflow. Because LLMs have limited context windows, feeding irrelevant information wastes capacity and degrades performance. Context engineering ensures each agent step gets a short, focused, and powerful context—maximizing accuracy and efficiency. It is covered in Chapter 1 and Appendix A.
What is Resource-Aware Optimization (Chapter 16)?+
This pattern covers techniques for managing computational costs in complex agent systems: Dynamic Model Switching (using smaller/cheaper models for simpler subtasks), Cost-Sensitive Exploration (planning with cost as a factor), Parallelization & Distributed Computing Awareness, Adaptive Tool Use & Selection, and Contextual Pruning & Summarization to reduce unnecessary token consumption across long-running workflows.

Key Clarifications
The most important clarifications to keep in mind when building agentic systems.
  • An agent is NOT just a chatbot. Agents perceive, plan, use tools, maintain state, and take multi-step actions toward goals. The difference is architectural, not just a matter of model capability.
  • Reflection requires structured critique, NOT just re-prompting. Simply asking "check your work" rarely produces meaningful improvement. A Critic agent or structured evaluation rubric is necessary for genuine quality gains.
  • Planning and planning execution are separate concerns. An agent creates a plan first (Planning pattern), then executes it step-by-step (Prompt Chaining / Tool Use). The ability to re-plan on failure is what makes it robust.
  • Human-in-the-Loop is not an admission of failure. It is a deliberate design choice for safety and quality at critical decision points. The goal is collaborative intelligence, not just autonomous intelligence.
  • Start simple. Add one pattern at a time. Prompt Chaining → add Tool Use → add Memory → add Reflection → add Multi-Agent. Each addition should solve a specific, observed limitation in the simpler system.
  • Non-determinism requires new testing strategies. Test for key outcomes and successful tool calls, not exact output strings. LLM-as-a-Judge and outcome-based evaluation are your primary tools.
