Parallelization

Parallelization executes multiple independent sub-tasks — LLM calls, tool usages, or entire sub-agents — concurrently, drastically reducing total latency for complex agentic workflows.

Parallelization Pattern Overview

While Prompt Chaining handles sequential workflows and Routing enables dynamic decision-making, many complex agentic tasks involve multiple sub-tasks that can be executed simultaneously rather than one after another. This is where Parallelization becomes crucial.

What is Parallelization?

Parallelization involves executing multiple components — such as LLM calls, tool usages, or entire sub-agents — concurrently. Instead of waiting for one step to complete before starting the next, parallel execution allows independent tasks to run at the same time, significantly reducing overall execution time for tasks that can be broken down into independent parts.

Sequential vs. Parallel: A Clear Example

An agent researching a topic sequentially must: Search A → Summarize A → Search B → Summarize B → Synthesize. In parallel: Search A and B simultaneously → Summarize A and B simultaneously → Synthesize. The final synthesis step waits for all parallel steps, but the total time is dramatically reduced.

Framework Support

Frameworks like LangChain, LangGraph, and Google ADK provide mechanisms for parallel execution. In LangChain LCEL, RunnableParallel bundles multiple chains to execute concurrently. Google ADK provides native ParallelAgent and SequentialAgent primitives for building multi-agent workflows with concurrent execution.

Concurrency vs. True Parallelism

In Python, asyncio provides concurrency — not true parallelism. It achieves this on a single thread using an event loop that switches between tasks when one is idle (e.g., waiting for a network request). This is especially effective for I/O-bound tasks like API calls, where most time is spent waiting for responses.

The Problem with Sequential Execution

When every step must wait for the previous one, agentic workflows become unnecessarily slow. Click each card to explore the specific bottlenecks.

Additive Latency

Total time equals the sum of all individual task durations.

A purely sequential execution, where each task waits for the previous one to finish, is often inefficient and slow. If querying three APIs each take 2 seconds, a sequential approach takes 6 seconds minimum — yet none of those calls depend on each other's results.

External I/O Bottlenecks

API calls and database queries force unnecessary idle time.

Latency becomes a significant bottleneck when tasks depend on external I/O operations, such as calling different APIs or querying multiple databases. While waiting for one response, the system is idle — all other independent operations are blocked unnecessarily.

Under-utilized Resources

Compute capacity sits idle waiting for network responses.

Without a mechanism for concurrent execution, the system's compute resources are under-utilized during I/O wait periods. This is particularly wasteful in multi-agent systems where several specialized agents could be doing meaningful work simultaneously.

Poor User Experience

Users wait unnecessarily for results that could come faster.

In user-facing applications, slow sequential workflows directly translate to poor UX. A travel planning agent that sequentially checks flights, then hotels, then local events — rather than querying all three simultaneously — makes users wait far longer than necessary for their complete itinerary.

Debugging and Complexity Costs

Concurrent architectures introduce new design challenges.

While parallelization solves latency issues, adopting a concurrent architecture introduces substantial complexity and cost, impacting key development phases such as design, debugging, and system logging. Race conditions, partial failures, and result aggregation all require careful handling.

Solution: The Parallelization Pattern

Parallelization identifies independent workflow components and executes them concurrently, collecting all results before proceeding to a synthesis step. Click each node to explore the flow.

1
Receive Input
Single query or task initiates the workflow
A single input — such as a user query or a task request — enters the system. The parallelization layer analyzes this input and fans it out to multiple independent processing branches simultaneously.
2
Concurrent Execution
RunnableParallel / ParallelAgent — all branches run simultaneously
Independent tasks are dispatched concurrently. In LangChain LCEL, a RunnableParallel block bundles multiple chains (e.g., summarize_chain, questions_chain, terms_chain) and invokes them at the same time. In Google ADK, a ParallelAgent orchestrates multiple LlmAgent sub-agents running in parallel.
3
Await Completion
Barrier synchronization — wait for all parallel branches
The system waits for all parallel branches to complete before proceeding. This is the synchronization barrier. In asyncio-based systems, this corresponds to awaiting all tasks. The total wait time equals the slowest individual branch, not the sum of all branches.
4
Aggregation and Synthesis
Merger agent or synthesis chain combines all results
All parallel outputs are collected and passed to a synthesis step — typically a final LLM call that combines the results into a coherent unified response. In ADK, a dedicated MergerAgent (LlmAgent) synthesizes the summaries stored in session state by the parallel researcher agents.

Core Concepts

Understanding the key building blocks that make parallelization work in modern agentic frameworks.

RunnableParallel (LangChain LCEL)

The primary LangChain construct for parallel execution. A RunnableParallel block bundles multiple runnable components within a dictionary. When passed as input to a subsequent component in the chain, the LCEL runtime executes all contained runnables concurrently and collects their outputs as a dictionary keyed by name.

ParallelAgent and SequentialAgent (Google ADK)

ADK provides first-class primitives for parallel multi-agent workflows. A ParallelAgent orchestrates concurrent execution of its sub-agents, completing once all sub-agents have finished and stored their results in the shared session state. A SequentialAgent then runs subsequent steps (like a MergerAgent) in order.

output_key and Session State

In Google ADK parallel workflows, each LlmAgent sub-agent uses an output_key parameter to store its result in the shared session state (e.g., output_key="renewable_energy_result"). The downstream merger agent reads all these keys from state to synthesize a combined report.

asyncio and Concurrency Model

Python's asyncio provides concurrency through an event loop that switches between tasks when one is idle (awaiting a network request). This creates the effect of parallel progress while executing on a single thread, constrained by Python's GIL. True CPU parallelism requires multiprocessing.

Use Cases

Information Gathering and Research

Collecting information from multiple sources simultaneously is a classic parallelization use case. An agent researching a company can fire off multiple lookups at once.

Applying different analysis techniques or processing different data segments concurrently. An agent analyzing customer feedback can run multiple analysis passes in parallel.

Run in parallel: sentiment analysis, keyword extraction, feedback categorization, urgent issue identification

Provide a multi-faceted analysis report quickly

Multi-API or Tool Interaction

Calling multiple independent APIs or tools simultaneously to gather different types of information. A travel planning agent benefits greatly from this pattern.

Generating different parts of a complex piece of content in parallel. A marketing email agent can produce all components simultaneously.

Code Examples

Concrete implementations demonstrating parallelization in both LangChain and Google ADK.

LangChain: RunnableParallel

Three independent LLM chains (summarize, generate questions, extract key terms) execute concurrently on the same topic input, then their outputs are merged by a synthesis prompt.

from langchain_openai import ChatOpenAI from langchain_core.prompts import ChatPromptTemplate from langchain_core.runnables import RunnableParallel, RunnablePassthrough # --- Three independent chains that run concurrently --- summarize_chain = ( ChatPromptTemplate.from_messages([ ("system", "Summarize the following topic concisely:"), ("user", "{topic}") ]) | llm | StrOutputParser() ) questions_chain = ( ChatPromptTemplate.from_messages([ ("system", "Generate three interesting questions about the following topic:"), ("user", "{topic}") ]) | llm | StrOutputParser() ) terms_chain = ( ChatPromptTemplate.from_messages([ ("system", "Identify 5-10 key terms from the following topic, separated by commas:"), ("user", "{topic}") ]) | llm | StrOutputParser() ) # --- Bundle into a parallel block --- map_chain = RunnableParallel( { "summary": summarize_chain, "questions": questions_chain, "key_terms": terms_chain, "topic": RunnablePassthrough(), # pass original topic through } ) # --- Synthesis step runs after all parallel chains complete --- synthesis_prompt = ChatPromptTemplate.from_messages([ ("system", """Based on the following information: Summary: {summary} Related Questions: {questions} Key Terms: {key_terms} Synthesize a comprehensive answer."""), ("user", "Original topic: {topic}") ]) full_parallel_chain = map_chain | synthesis_prompt | llm | StrOutputParser() # --- Async invocation --- async def run_parallel_example(topic: str) -> None: response = await full_parallel_chain.ainvoke(topic) print(response)

Google ADK: ParallelAgent + SequentialAgent

Three specialist researcher LlmAgents run concurrently inside a ParallelAgent. Once all finish, a MergerAgent synthesizes their stored results. A SequentialAgent orchestrates the entire flow.

from google.adk.agents import LlmAgent, ParallelAgent, SequentialAgent from google.adk.tools import google_search GEMINI_MODEL = "gemini-2.0-flash" # --- Three researcher sub-agents run in parallel --- researcher_agent_1 = LlmAgent( name="RenewableEnergyResearcher", model=GEMINI_MODEL, instruction="""Research latest advancements in 'renewable energy sources'. Use the Google Search tool. Summarize key findings (1-2 sentences). Output *only* the summary.""", tools=[google_search], output_key="renewable_energy_result" # stored in session state ) researcher_agent_2 = LlmAgent( name="EVResearcher", model=GEMINI_MODEL, instruction="Research latest developments in 'electric vehicle technology'. Summarize (1-2 sentences).", tools=[google_search], output_key="ev_technology_result" ) researcher_agent_3 = LlmAgent( name="CarbonCaptureResearcher", model=GEMINI_MODEL, instruction="Research current state of 'carbon capture methods'. Summarize (1-2 sentences).", tools=[google_search], output_key="carbon_capture_result" ) # --- ParallelAgent runs all researchers concurrently --- parallel_research_agent = ParallelAgent( name="ParallelWebResearchAgent", sub_agents=[researcher_agent_1, researcher_agent_2, researcher_agent_3], ) # --- MergerAgent synthesizes results after parallel agents finish --- merger_agent = LlmAgent( name="SynthesisAgent", model=GEMINI_MODEL, instruction="""Synthesize the research findings into a structured report. Input Summaries: {renewable_energy_result}, {ev_technology_result}, {carbon_capture_result} Ground your report *exclusively* on the provided summaries.""" ) # --- SequentialAgent orchestrates the overall flow --- sequential_pipeline_agent = SequentialAgent( name="ResearchAndSynthesisPipeline", sub_agents=[parallel_research_agent, merger_agent], ) root_agent = sequential_pipeline_agent

Deep Dive: Implementation Layers

Parallelization can be applied at multiple levels of an agentic system. Click each layer to understand the mechanics and trade-offs.

LangChain LCEL: RunnableParallel
Dictionary-based concurrent chain execution
The primary LCEL mechanism for parallelism. Structuring multiple runnable components within a dictionary or list construct causes the LCEL runtime to execute them concurrently. The outputs are collected as a dictionary keyed by the names you assign. RunnablePassthrough can be included to forward the original input alongside the parallel results to a downstream synthesis step.
LangGraph: Parallel Graph Nodes
Multiple nodes initiated from a single state transition
LangGraph's graph structure enables parallelism through topology. Parallel workflows are defined by architecting the graph such that multiple nodes, lacking direct sequential dependencies, can be initiated from a single common node. These parallel pathways execute independently before their results are aggregated at a subsequent convergence point — a "fan-out / fan-in" pattern.
Google ADK: ParallelAgent Primitive
Native multi-agent concurrent execution
Google ADK provides first-class ParallelAgent and SequentialAgent constructs. A ParallelAgent orchestrates the concurrent execution of its sub-agents and completes once all have finished. Sub-agents store their results in shared session state using output_key parameters, which downstream agents then read for synthesis. This is particularly powerful for research, validation, and multi-source data gathering scenarios.
asyncio: Concurrency Foundation
Single-threaded event loop for I/O-bound concurrency
Python's asyncio provides concurrency on a single thread using an event loop that intelligently switches between tasks when one is idle (e.g., waiting for a network request). This creates the effect of multiple tasks progressing at once but the code is still executed by one thread, constrained by Python's GIL. For CPU-bound tasks, use multiprocessing. For I/O-bound tasks (API calls, database queries), asyncio is highly effective.
Error Handling and Partial Failures
Designing resilient parallel workflows
A key challenge in parallel execution is handling partial failures — one branch failing should not necessarily kill the entire workflow. Robust implementations include per-branch try/except blocks, timeout management, fallback values for failed branches, and logging that identifies which sub-task failed. The synthesis step must handle cases where some inputs may be missing or error messages rather than valid results.

At a Glance

WHAT

Many agentic workflows involve multiple sub-tasks that must be completed to achieve a final goal. A purely sequential execution, where each task waits for the previous one to finish, is often inefficient. Latency from external I/O compounds into significant bottlenecks — the total processing time is the sum of all individual durations.

WHY

The Parallelization pattern enables simultaneous execution of independent tasks. By identifying components that do not rely on each other's immediate outputs, a main process can invoke several sub-tasks that run in parallel and wait for all of them to complete. Total wait time equals the slowest branch, not the sum — a dramatic improvement.

Rule of Thumb

Use this pattern when a workflow contains multiple independent operations that can run simultaneously — fetching data from several APIs, processing different chunks of data, or generating multiple pieces of content for later synthesis. If tasks depend on each other's output, they must remain sequential.

Trade-offs

Benefits: dramatically reduced latency for I/O-bound tasks, better resource utilization. Costs: increased design complexity, harder debugging, more complex logging. Race conditions and partial failures require careful handling. Start with the simplest parallelization that solves the problem; add complexity only when needed.

Key Takeaways

The essential insights from the Parallelization pattern.

  • Parallelization is a pattern for executing independent tasks concurrently to improve efficiency — particularly for I/O-bound operations like API calls and database queries.
  • It is especially useful when tasks involve waiting for external resources: the total wait equals the slowest branch, not the sum of all branch times.
  • Adopting a concurrent or parallel architecture introduces substantial complexity and cost, impacting design, debugging, and system logging — weigh this against the latency benefit.
  • Frameworks like LangChain and Google ADK provide built-in support for defining and managing parallel execution, reducing implementation friction.
  • In LangChain LCEL, RunnableParallel is the key construct for running multiple runnables side-by-side and collecting their outputs as a named dictionary.
  • Google ADK can facilitate parallel execution through ParallelAgent, where a coordinator's sub-agents each store results in session state using output_key for downstream synthesis.
  • Parallelization helps reduce overall latency and makes agentic systems more responsive — integrating it with sequential (chaining) and conditional (routing) flows enables sophisticated, high-performance agentic applications.
  • Python's asyncio provides concurrency (not true parallelism) through an event loop — effective for I/O-bound tasks; use multiprocessing for CPU-bound parallelism.
