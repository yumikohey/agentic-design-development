What is Exception Handling and Recovery?

For AI agents to operate reliably in diverse real-world environments, they must be able to manage unforeseen situations, errors, and malfunctions. Just as humans adapt to unexpected obstacles, intelligent agents need robust systems to detect problems, initiate recovery procedures, or at least ensure controlled failure. This essential requirement forms the basis of the Exception Handling and Recovery pattern.

Proactive and Reactive Strategies

This pattern emphasizes both proactive preparation and reactive strategies to ensure continuous operation even when facing challenges. Proactive measures include monitoring tool outputs, checking API responses for error codes, and setting timeouts. Reactive strategies include logging, retries, fallbacks, graceful degradation, and escalation to humans.

Connection to Reflection

This pattern may sometimes be used together with reflection. For example, if an initial attempt fails and raises an exception, a reflective process can analyze the failure and reattempt the task with a refined approach — such as an improved prompt — to resolve the error. This combination of exception handling and reflection creates a highly adaptive agent.

From Fragile to Dependable

Implementing this pattern transforms AI agents from fragile, unreliable systems into robust, dependable components capable of operating resiliently in challenging and highly unpredictable environments. This ensures that agents maintain functionality, minimize downtime, and provide a seamless experience even when faced with unexpected issues.

Why Agents Fail Without Exception Handling
Without structured exception handling, AI agents are brittle — a single unexpected event can cause complete failure. Click each card to understand specific failure modes:

Tool Call Failures

External tools crash or return errors

When a tool fails — due to a bug, unexpected input, or dependency issue — an agent without exception handling will crash or produce garbage output. The entire workflow stops, even if the tool failure was minor and recoverable.

Service Unavailability

APIs return 404, 500, or timeout errors

External services are unreliable. An agent making API calls will inevitably encounter HTTP errors, timeouts, and rate limits. Without handling for these transient failures, the agent fails permanently on what could be a brief outage.

Invalid or Malformed Data

Incoherent outputs from external systems

Tools and APIs can return unexpected formats, incomplete data, or nonsensical responses. An agent that assumes clean, well-formed input will fail unpredictably when real-world data is messier than expected.

Cascading Failures

One error triggers a chain of downstream failures

In a multi-step agent workflow, a single unhandled error early in the pipeline can propagate through subsequent steps, causing the entire system to fail in ways that are difficult to diagnose and recover from.

No Graceful Degradation

Binary outcome: full success or total failure

Without exception handling, the agent provides an all-or-nothing experience. If any component fails, the user gets nothing. A well-designed agent should be able to deliver partial value even when some components are unavailable.

The Real-World Challenge

In production environments, failure is not the exception — it's the norm. Networks go down, APIs change, services rate-limit, databases experience latency spikes, and inputs are never perfectly clean. An agent that can't handle these realities cannot be deployed in any meaningful production setting. Reliability is not a feature; it's a prerequisite.

Three-Phase Exception Handling Framework
The pattern addresses failures through three coordinated phases. Click each phase to explore its mechanisms:
1
Error Detection
Proactively identify operational issues as they arise
Meticulously identify operational issues as they occur. This includes:

• Invalid/malformed tool outputs: validate structure and content
• API error codes: detect 404 (Not Found), 500 (Internal Server Error), 503 (Service Unavailable)
• Unusually long response times: implement timeouts
• Incoherent responses: detect nonsensical output deviating from expected formats
• Proactive monitoring: other agents or specialized monitoring systems can detect anomalies before they escalate.
2
Error Handling
Execute a carefully planned response to detected errors
Once an error is detected, execute a structured response plan:

• Logging: record error details meticulously for later debugging and analysis
• Retries: retry the action, sometimes with slightly adjusted parameters — especially effective for transient errors
• Fallbacks: use alternative strategies or methods to maintain some functionality
• Graceful degradation: maintain partial functionality to provide at least some value
• Notification: alert human operators or other agents for situations requiring intervention
3
Recovery
Restore the agent or system to a stable operational state
Restore stable operation after an error:

• State rollback: reverse recent changes or transactions to undo the effects of the error
• Diagnosis: thoroughly investigate the cause of the error to prevent recurrence
• Self-correction: adjust the agent's plan, logic, or parameters through replanning
• Escalation: in complex or severe cases, delegate the issue to a human operator or higher-level system

The goal is not just to recover, but to learn — each handled exception makes the system more resilient over time.

Why This Transforms Reliability

Implementing this robust exception handling and recovery pattern can transform AI agents from fragile and unreliable systems into robust, dependable components capable of operating effectively and resiliently in challenging and highly unpredictable environments. This ensures the agents maintain functionality, minimize downtime, and provide a seamless and reliable experience even when faced with unexpected issues.

Core Concepts: Handling Strategies
Each error handling strategy serves a different purpose depending on the nature and severity of the failure.

Retries with Backoff

For transient errors (temporary network issues, rate limits), retrying the operation after a brief pause is often sufficient. Exponential backoff — increasing the wait time between retries — prevents overloading an already-stressed service. Always limit the number of retries to avoid infinite loops.

Fallback Mechanisms

When the primary approach fails and retries are exhausted, switch to an alternative. A precise geolocation tool failing can fall back to a city-level lookup. A live stock price API failing can fall back to a cached value. Fallbacks ensure the agent can still deliver partial value even when the optimal path is unavailable.

Graceful Degradation vs. Hard Failure

The goal is to provide some value even when optimal operation is not possible. An agent whose primary tool fails should try a fallback, then gracefully communicate limitations rather than crashing. This "partial success" model is far more user-friendly and operationally resilient than all-or-nothing behavior.

Practical Applications and Use Cases
Exception Handling and Recovery is critical for any agent deployed in a real-world scenario where perfect conditions cannot be guaranteed. Click the tabs to explore:

Customer Service Chatbots

If a chatbot tries to access a customer database and the database is temporarily down, it shouldn't crash. Instead, it should detect the API error, inform the user about the temporary issue, perhaps suggest trying again later, or escalate the query to a human agent.

1
Attempt to query customer database via API
2
Detect API error (e.g., 503 Service Unavailable)
3
Log the error with timestamp and error code
4
Inform user of temporary issue; suggest retry or human escalation

Automated Financial Trading

A trading bot might encounter an "insufficient funds" error or a "market closed" error when executing a trade. It needs to handle these exceptions by logging the error, not repeatedly trying the same invalid trade, and potentially notifying the user or adjusting its strategy. Retrying a failed trade blindly could compound financial losses.

Smart Home Automation

An agent controlling smart lights might fail to turn on a light due to a network issue or device malfunction. It should detect this failure, perhaps retry once, and if still unsuccessful, notify the user that the light could not be turned on and suggest manual intervention — not silently fail or loop indefinitely.

Data Processing Agents

An agent processing a batch of documents might encounter a corrupted file. Rather than halting the entire process, it should skip the corrupted file, log the error, continue processing other files, and report the skipped files at the end. This "skip and report" pattern preserves the value of successfully processing the remaining documents.

Web Scraping Agents

When a web scraping agent encounters a CAPTCHA, a changed website structure, or a server error (404 Not Found, 503 Service Unavailable), it needs to handle these gracefully. This could involve pausing, using a proxy, rotating user agents, or reporting the specific URL that failed rather than crashing the entire scraping job.

Robotics and Manufacturing

A robotic arm performing an assembly task might fail to pick up a component due to misalignment. It needs to detect this failure (via sensor feedback), attempt to readjust, retry the pickup, and if persistent, alert a human operator or switch to a different component. Continuing without detection could cause safety hazards or damage.

Hands-On Code: ADK Sequential Agent with Fallback
This example shows how to build a robust location retrieval system using ADK's SequentialAgent with a primary handler, fallback handler, and response agent.

Primary and fallback handler agents (Google ADK):

from google.adk.agents import Agent, SequentialAgent # Agent 1: Tries the primary tool. Narrow and focused. primary_handler = Agent( name="primary_handler", model="gemini-2.0-flash-exp", instruction=""" Your job is to get precise location information. Use the get_precise_location_info tool with the user's provided address. """, tools=[get_precise_location_info] ) # Agent 2: Acts as the fallback handler, checking state to decide its action. fallback_handler = Agent( name="fallback_handler", model="gemini-2.0-flash-exp", instruction=""" Check if the primary location lookup failed by looking at state["primary_location_failed"]. - If it is True, extract the city from the user's original query and use the get_general_area_info tool. - If it is False, do nothing. """, tools=[get_general_area_info] )

Response agent and SequentialAgent orchestration:

# Agent 3: Presents the final result from the state. response_agent = Agent( name="response_agent", model="gemini-2.0-flash-exp", instruction=""" Review the location information stored in state["location_result"]. Present this information clearly and concisely to the user. If state["location_result"] does not exist or is empty, apologize that you could not retrieve the location. """, tools=[] # This agent only reasons over the final state. ) # The SequentialAgent ensures the handlers run in a guaranteed order. robust_location_agent = SequentialAgent( name="robust_location_agent", sub_agents=[primary_handler, fallback_handler, response_agent] )

How it works: layered exception handling

# This structure provides a layered approach to error handling: # 1. primary_handler tries the precise tool — best-case path # 2. fallback_handler checks state['primary_location_failed']: # - True → uses the general area tool (graceful degradation) # - False → skips (no-op, primary succeeded) # 3. response_agent synthesizes whatever result is in state: # - Found precise result → presents it # - Found general result → presents it with context # - Found nothing → apologizes gracefully (controlled failure) # # The SequentialAgent guarantees execution order, ensuring # the fallback always runs AFTER the primary attempt.
Deep Dive: Advanced Exception Handling Strategies
Beyond the basics, production exception handling requires nuanced strategies. Click each layer to explore:
Exponential Backoff for Retries
Smart retry logic prevents thundering herd problems
Simple immediate retries can overwhelm a struggling service. Exponential backoff adds increasing delays between retries (e.g., 1s, 2s, 4s, 8s). Combined with jitter (random offset), this spreads load and gives the service time to recover. Always cap the maximum number of retries and set a maximum wait time.
Circuit Breaker Pattern
Stop calling services that are clearly down
If a service repeatedly fails, continuing to call it wastes resources and increases latency. The circuit breaker pattern tracks failure rate: when failures exceed a threshold, the circuit "opens" and immediately returns a fallback response without attempting the call. After a cool-down period, the circuit "half-opens" and tests if the service has recovered.
Structured Error Logging
Errors are only useful if they're actionable
Good error logs capture: timestamp, error type and message, stack trace, agent context (what it was trying to do), input that caused the failure, and the recovery action taken. Structured logging (JSON format) enables automated analysis, alerting, and trend detection across production deployments.
Exception Handling with Reflection
Learn from failures, not just handle them
The most sophisticated exception handling combines error recovery with reflection: when an error occurs, the agent doesn't just retry blindly — it analyzes what went wrong and adapts its approach. For LLM-based agents, this might mean modifying the prompt, changing tool parameters, or decomposing the request differently before the next attempt.
Human Escalation Protocols
Know when to involve humans, and how
Not every error can or should be handled autonomously. Clear escalation criteria define when human intervention is required: recurring failures beyond retry limits, high-severity errors with financial or safety implications, ambiguous situations where the agent lacks confidence. Escalation messages should include full context: what happened, what was tried, and what the agent needs from the human.
At a Glance

WHAT

AI agents operating in real-world environments inevitably encounter unforeseen situations, errors, and system malfunctions. These range from tool failures and network issues to invalid data. Without a structured way to manage these problems, agents are fragile, unreliable, and prone to complete failure when faced with unexpected hurdles.

WHY

The Exception Handling and Recovery pattern provides a standardized solution for building robust, resilient agents. It equips them with agentic capability to anticipate, manage, and recover from operational failures through proactive error detection, reactive handling strategies (logging, retries, fallbacks), and recovery protocols (rollback, self-correction, escalation).

RULE OF THUMB

Use this pattern for any AI agent deployed in a dynamic, real-world environment where system failures, tool errors, network issues, or unpredictable inputs are possible and operational reliability is a key requirement. If your agent touches external systems, it needs exception handling.

KEY INSIGHT

The three phases — Error Detection, Error Handling, Recovery — form a complete resilience framework. Detection without handling is just monitoring. Handling without recovery leaves the system in a degraded state. All three phases together create an agent that can truly self-heal and maintain operational integrity under pressure.

Key Takeaways
Essential points from Chapter 12 on Exception Handling and Recovery:
  • Exception Handling and Recovery is essential for building robust and reliable agents that can operate in unpredictable real-world environments.
  • The pattern involves three coordinated phases: detecting errors, handling them gracefully, and implementing recovery strategies.
  • Error detection includes validating tool outputs, checking API error codes (404, 500, 503), implementing timeouts, and monitoring for anomalous responses.
  • Handling strategies include logging for diagnostics, retrying transient failures (with backoff), using fallback mechanisms, and graceful degradation.
  • Recovery focuses on restoring stable operation through state rollback, diagnosis, self-correction by adjusting the plan, or escalation to a human operator.
  • The ADK SequentialAgent pattern with primary and fallback handlers is an effective way to implement layered exception handling in production agents.
  • This pattern ensures agents can operate effectively even in unpredictable real-world environments, maintaining functionality and minimizing downtime.
  • Exception handling may be combined with reflection: when a failure occurs, the agent can analyze it and retry with a refined approach rather than just executing the same failed operation again.
