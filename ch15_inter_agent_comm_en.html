Chapter 15

Inter-Agent Communication (A2A)

Google's Agent2Agent (A2A) protocol is an open HTTP-based standard enabling diverse AI agents — built on different frameworks — to communicate, delegate tasks, and collaborate seamlessly across organizational and technical boundaries.

Pattern Overview

Individual AI agents face limitations when tackling complex, multifaceted problems alone. A2A enables diverse agents — potentially built with different frameworks like LangGraph, CrewAI, or Google ADK — to collaborate effectively through seamless coordination, task delegation, and information exchange.

Key Feature
Interop

Universal Interoperability

A2A enables agents built with any framework (Google ADK, LangGraph, CrewAI, Azure AI Foundry) to communicate through a standardized HTTP protocol. Agents operate as "opaque" systems — the client does not need to understand the remote agent's internal implementation details.

Agent Card Identity

Each agent's digital identity is defined by its Agent Card — a JSON file describing capabilities, skills, endpoint URL, supported interaction modes, and authentication requirements. This enables automatic discovery and capability negotiation between agents.

Flexible Communication Modes

A2A supports synchronous request/response, asynchronous polling, and real-time Server-Sent Events (SSE) streaming — accommodating everything from simple queries to long-running complex tasks with incremental results.

Industry Backing

Supported by Atlassian, Box, LangChain, MongoDB, Salesforce, SAP, ServiceNow. Microsoft integrating into Azure AI Foundry and Copilot Studio. Auth0 and SAP integrating A2A support into their platforms.

Problems A2A Solves

Without a standard protocol, multi-agent systems face critical integration challenges. Click each card to explore.

Framework Isolation

Agents built on different frameworks cannot communicate without custom integrations.

A LangGraph agent cannot natively call a CrewAI agent or a Google ADK agent. Without A2A, integrating specialized agents from different ecosystems requires brittle, one-off custom connectors that become maintenance burdens as frameworks evolve.

No Capability Discovery

Agents have no standard way to discover what other agents can do.

Without a standard capability description format like Agent Card, an orchestrating agent must be hardcoded with knowledge of every remote agent's abilities. This makes dynamic, self-organizing multi-agent systems impossible to build reliably.

Integration Cost & Complexity

Building interoperability from scratch is expensive and time-consuming.

Each custom agent-to-agent integration requires designing a communication protocol, authentication scheme, error handling strategy, and retry logic. This engineering overhead multiplies with each new agent added to the system, hindering complex multi-agent applications.

Task State Management

Long-running distributed tasks lack a standard state management model.

Complex workflows involve tasks that run for seconds, minutes, or hours across multiple agents. Without a standard model for task lifecycle (submitted → working → completed/failed), building reliable multi-agent pipelines with proper error recovery requires reinventing state management each time.

A2A Architecture

A2A provides a layered architecture for agent interoperability. Click each layer to understand how it works.

1
Agent Card (Identity & Discovery)
JSON capability manifest hosted at /.well-known/agent.json
Each A2A server publishes an Agent Card — a JSON file describing its identity, endpoint URL, version, capabilities (streaming, push notifications), skills (with IDs, descriptions, examples), input/output modes, and authentication requirements. This is how agents discover and understand each other without prior coordination.
2
Transport Layer (HTTP + JSON-RPC 2.0)
Standardized communication protocol
All A2A communication occurs over HTTP(S) using the JSON-RPC 2.0 protocol for payloads. Messages contain attributes (key-value metadata like priority or creation time) and one or more parts (the actual content — text, files, or structured JSON). A server-generated contextId groups related tasks to preserve context across multiple interactions.
3
Task Lifecycle Management
Asynchronous work units with state tracking
Tasks are the fundamental units of work in A2A. Each task has a unique ID and moves through states: submitted → working → completed (or failed/canceled). Tasks support parallel processing in complex operations. Tangible outputs from tasks are called artifacts — composed of parts and streamable incrementally as results become available.
4
Security Layer
mTLS, auth declarations, and audit logs
A2A security includes: Mutual TLS (mTLS) for encrypted, authenticated connections preventing unauthorized access and data interception. Authentication requirements explicitly declared in the Agent Card. Credentials passed via HTTP headers (OAuth 2.0 tokens or API keys) to prevent exposure in URLs. Comprehensive audit logs of all inter-agent communications for accountability and troubleshooting.
Core Concepts

A2A is built on five foundational pillars. Understanding each is essential for building A2A-compliant systems.

Core Actors

User: Initiates requests for agent assistance. A2A Client (Client Agent): An application or AI agent acting on the user's behalf to request actions or information from remote agents. A2A Server (Remote Agent): An AI agent providing an HTTP endpoint to process client requests and return results — operates as an "opaque" system.

Agent Card Example (WeatherBot)

{ "name": "WeatherBot", "description": "Provides accurate weather forecasts and historical data.", "url": "http://weather-service.example.com/a2a", "version": "1.0.0", "capabilities": { "streaming": true, "pushNotifications": false, "stateTransitionHistory": true }, "authentication": { "schemes": ["apiKey"] }, "defaultInputModes": ["text"], "defaultOutputModes": ["text"], "skills": [ { "id": "get_current_weather", "name": "Get Current Weather", "description": "Retrieve real-time weather for any location.", "examples": ["What's the weather in Paris?"] }, { "id": "get_forecast", "name": "Get Forecast", "description": "Get 5-day weather predictions.", "examples": ["5-day forecast for New York"] } ] }

Interaction Mechanisms

Synchronous (tasks/send): Client sends request, waits for complete response. For quick operations. Asynchronous Polling: Server acknowledges with "working" + task ID; client polls for completion. For longer tasks. Streaming (tasks/sendSubscribe): Server-Sent Events for real-time incremental results. Push Notifications (Webhooks): Server calls client webhook when task status changes significantly — for very long-running tasks.

Practical Use Cases

A2A is indispensable for building sophisticated AI solutions across diverse domains. Explore the key applications.

Multi-Framework Collaboration

A2A's primary use case: enabling independent AI agents built on different frameworks (ADK, LangChain, CrewAI) to communicate and collaborate. Fundamental for complex multi-agent systems where different agents specialize in different aspects of a problem — a LangGraph research agent works alongside an ADK execution agent.

Automated Workflow Orchestration

In enterprise settings, A2A facilitates complex workflows by enabling agents to delegate and coordinate tasks. An orchestrator agent handles initial data collection, delegates to a specialist analysis agent, then to a report generation agent — all communicating via A2A protocol across different systems.

1
Orchestrator receives workflow trigger (e.g., quarterly report)
2
Data collection agent gathers raw data via A2A
3
Analysis agent processes data, returns structured insights
4
Report agent generates final deliverable from insights

Dynamic Information Retrieval

Agents communicate to retrieve and exchange real-time information. A primary agent requests live market data from a specialized "data fetching agent" which uses external APIs to gather the information and send it back — enabling dynamic, up-to-date responses without embedding every API integration into a single agent.

1
Financial analysis agent needs current market prices
2
Sends A2A request to specialized market data agent
3
Market data agent queries external APIs
4
Returns structured data artifact via A2A response

Cross-Organization Agent Collaboration

A2A enables agents from different organizations to collaborate securely. A supplier's agent can communicate with a customer's procurement agent using standardized A2A, with authentication and authorization handled through the Agent Card's security declarations — without either organization needing to share internal implementation details.

1
Customer procurement agent discovers supplier's catalog agent via Agent Card
2
Authenticates using declared OAuth 2.0 scheme
3
Requests inventory availability as A2A task
4
Supplier agent returns availability artifact; order automatically placed

Specialist Agent Delegation

An orchestrating agent handles user requests but delegates specialized tasks to purpose-built expert agents — a calendar agent for scheduling, a code execution agent for computations, a translation agent for multilingual tasks. Each specialist is independently deployable and discoverable via A2A, enabling modular, scalable architectures.

1
User asks assistant to "schedule a meeting and prepare a French summary"
2
Orchestrator discovers calendar agent and translation agent via A2A
3
Delegates scheduling to calendar agent (Google Calendar integration)
4
Delegates translation; combines results into final response

A2A Task Messages (JSON-RPC 2.0)

// Synchronous Request — tasks/send { "jsonrpc": "2.0", "id": "1", "method": "tasks/send", "params": { "id": "task-001", "sessionId": "session-001", "message": { "role": "user", "parts": [{ "type": "text", "text": "Am I free from 10am to 11am tomorrow?" }] }, "acceptedOutputModes": ["text/plain"] } }

ADK Calendar Agent with A2A Server

import datetime, asyncio, os from google.adk.agents import LlmAgent from google.adk.tools.google_api_tool import CalendarToolset async def create_agent(client_id, client_secret) -> LlmAgent: """Constructs the ADK agent with Calendar tools.""" toolset = CalendarToolset(client_id=client_id, client_secret=client_secret) return LlmAgent( model='gemini-2.0-flash-001', name='calendar_agent', description="An agent that can help manage a user's calendar", instruction=f"""You are an agent that can help manage a user's calendar. Users will request information about their calendar or make changes. Use the provided tools for interacting with the calendar API. If not specified, assume the user wants the 'primary' calendar. When using Calendar API tools, use well-formed RFC3339 timestamps. Today is {datetime.datetime.now()}.""", tools=await toolset.get_tools(), ) def main(host: str, port: int): # Define agent skill and Agent Card skill = AgentSkill( id='check_availability', name='Check Availability', description="Checks a user's availability using Google Calendar", tags=['calendar'], examples=['Am I free from 10am to 11am tomorrow?'], ) agent_card = AgentCard( name='Calendar Agent', description="An agent that can manage a user's calendar", url=f'http://{host}:{port}/', version='1.0.0', defaultInputModes=['text'], defaultOutputModes=['text'], capabilities=AgentCapabilities(streaming=True), skills=[skill], ) adk_agent = asyncio.run(create_agent( client_id=os.getenv('GOOGLE_CLIENT_ID'), client_secret=os.getenv('GOOGLE_CLIENT_SECRET'), )) runner = Runner( app_name=agent_card.name, agent=adk_agent, artifact_service=InMemoryArtifactService(), session_service=InMemorySessionService(), memory_service=InMemoryMemoryService(), ) agent_executor = ADKAgentExecutor(runner, agent_card) request_handler = DefaultRequestHandler( agent_executor=agent_executor, task_store=InMemoryTaskStore() ) a2a_app = A2AStarletteApplication( agent_card=agent_card, http_handler=request_handler) uvicorn.run(Starlette(routes=a2a_app.routes()), host=host, port=port)
Deep Dive: A2A Technical Details

Explore the technical foundations of the A2A protocol. Click each layer to expand.

Agent Discovery Strategies
How clients find A2A servers
Three discovery approaches: (1) Well-Known URI — agents host their Agent Card at /.well-known/agent.json, offering automated accessibility for public or domain-specific use; (2) Curated Registries — centralized catalogs where Agent Cards are published and queried, suited for enterprise environments needing access control; (3) Direct Configuration — Agent Card information is embedded or privately shared, appropriate for closely coupled or private systems where dynamic discovery is not needed. Regardless of method, Agent Card endpoints should be secured via access control or mTLS.
A2A vs. MCP
Complementary protocols with different scopes
A2A and Anthropic's Model Context Protocol (MCP) are complementary. MCP focuses on structuring context for an agent's interaction with external data sources and tools (databases, APIs, file systems). A2A facilitates coordination and communication between agents themselves — task delegation and collaboration. MCP answers "how does an agent use tools?" while A2A answers "how do agents work together?"
Multi-Turn Conversations & Input Required
Interactive task flows with context preservation
A2A supports multi-turn conversations through the input-required task state. When a remote agent needs additional information to complete a task, it can return a task in "input-required" state rather than failing. The contextId groups all messages in a conversation, preserving context across multiple request/response cycles. This enables complex interactive workflows where agents negotiate task requirements before execution.
Streaming & Modality Support
Real-time, multimodal agent communication
A2A is modality-agnostic — beyond text, it supports audio and video data types, enabling rich multimodal AI applications. The streaming capability (tasks/sendSubscribe) establishes a persistent SSE connection, allowing remote agents to continuously push incremental updates as results become available — critical for long-running tasks where users need progressive feedback. The Agent Card explicitly declares whether streaming and push notification capabilities are supported.
Modular Architecture Benefits
Scalability through agent specialization
A2A encourages a modular architecture where specialized agents can operate independently on different ports or services, enabling true system scalability and distribution. Each specialist agent can be developed, deployed, scaled, and updated independently. This separation of concerns mirrors microservice architecture principles — reducing coupling while enabling each agent to be highly optimized for its specific domain. Tools like Trickle AI help visualize and track A2A communications for debugging and optimization.
At a Glance

WHAT IS THE PROBLEM?

Individual AI agents built on different frameworks cannot communicate — they are isolated systems. Without a common protocol, building sophisticated multi-agent systems where specialized agents collaborate is costly, time-consuming, and produces fragile custom integrations.

WHY A2A WORKS

A2A provides an open HTTP-based standard enabling interoperability — agents coordinate, delegate tasks, and share information regardless of their underlying technology. Agent Cards enable automatic discovery and capability negotiation. Flexible interaction mechanisms support diverse use cases from simple queries to long-running complex workflows.

WHEN TO USE IT

Use A2A when orchestrating collaboration between two or more AI agents, especially if built on different frameworks. Ideal for complex, modular applications where specialized agents handle specific workflow parts. Essential when an agent needs to dynamically discover and consume capabilities of other agents.

KEY CAVEATS

Added network latency from HTTP-based communication between agents. Distributed systems complexity — partial failures, timeout handling, and retry logic must be designed carefully. Security overhead of mTLS and authentication for each agent interaction. The protocol is still evolving — implementations may need updates as the standard matures.

Key Takeaways
  • The Google A2A protocol is an open, HTTP-based standard that facilitates communication and collaboration between AI agents built with different frameworks — LangGraph, CrewAI, ADK, Azure AI Foundry, and more.
  • An Agent Card serves as a digital identifier for an agent — a JSON file describing capabilities, skills, endpoint URL, interaction modes, and authentication requirements, enabling automatic discovery.
  • A2A offers both synchronous request/response (tasks/send) and streaming updates (tasks/sendSubscribe via SSE) to accommodate varying communication needs from simple queries to long-running tasks.
  • The protocol supports multi-turn conversations through an input-required task state, allowing agents to request additional information while maintaining context via a shared contextId.
  • A2A encourages modular architecture where specialized agents operate independently, enabling system scalability through agent specialization — mirroring microservice design principles.
  • Security is built-in: mTLS for encrypted connections, authentication requirements declared in the Agent Card, credentials passed via HTTP headers, and comprehensive audit logs for all inter-agent communications.
  • A2A complements MCP (Model Context Protocol) — A2A handles agent-to-agent coordination, while MCP handles agent-to-tool/data-source interaction.
  • Strong industry backing from major technology companies (Atlassian, LangChain, Salesforce, SAP, ServiceNow, Microsoft) demonstrates A2A's growing importance as a foundational multi-agent standard.
