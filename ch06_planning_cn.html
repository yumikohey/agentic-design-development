规划模式

智能行为不仅仅是对当前输入的被动响应，更需要预见性——将复杂任务分解为可管理的步骤，并制定策略以实现目标状态。规划模式赋予智能体这种目标导向的能力。

模式概览

规划模式的核心是让智能体具备将高层目标转化为可执行行动序列的能力。与简单的任务执行智能体不同，规划智能体首先理解初始状态与目标状态，然后自主发现连接二者的最优行动路径。

什么是规划模式？

规划是智能体将复杂目标转化为一系列可执行步骤的能力。就像委托一位专家完成复杂任务时，你定义的是"做什么"（目标及约束），而非"怎么做"。智能体自主理解初始状态，发现达成目标的最优行动序列，并在遇到障碍时灵活调整。

规划的标志性特征：适应性

初始计划只是起点，而非刚性脚本。有能力的智能体能够整合新信息并绕开障碍。例如：如果首选场地不可用，智能体不会简单失败，而是重新评估选项，制定新计划。这种动态适应能力是规划模式的核心价值所在。

规划 vs. 固定工作流

关键决策问题是：解决方案的"如何做"是否需要被发现，还是已经明确已知？当问题已被充分理解且可重复时，将智能体限定在预定固定工作流中更为有效——这能限制不确定性，保证可靠一致的结果。规划适用于高度复杂、需要动态应对的场景。

现实案例：Google DeepResearch

Google Gemini DeepResearch 是规划模式的典型应用。它将用户的查询分解为多步研究计划，动态执行搜索、评估信息缺口，并迭代优化查询策略，最终生成带引用的结构化多页报告——全程自主规划，无需人工干预每个步骤。

问题所在

单纯的反应式智能体在面对复杂多步骤任务时严重受限。点击每张卡片了解具体挑战。

🎯 无法处理多步依赖

复杂任务的各步骤之间存在复杂的依赖关系。

例如入职新员工：必须先创建系统账户，才能分配培训模块；必须先协调不同部门，才能安排具体工作。反应式智能体无法理解并管理这种步骤间的依赖逻辑，导致执行顺序混乱或关键步骤遗漏。

🧱 遇到障碍直接失败

当预期路径不可行时，简单智能体无法自适应调整。

如果首选的酒店已满、API 调用失败或数据源不可用，没有规划能力的智能体只会报错或返回不完整结果。缺乏"遇到障碍重新规划"的能力，使其无法应对真实世界的动态不确定性。

📊 无法综合复杂信息

生成复杂输出需要结构化的多阶段信息处理流程。

生成一份研究报告需要：信息搜集阶段、数据摘要阶段、内容结构化阶段和迭代优化阶段。简单的单次响应智能体无法管理这种多阶段流程，导致输出质量低下或内容不完整。

🔄 策略缺失，结果不可靠

没有策略框架的智能体会产生不一致的低质量结果。

面对"分析市场竞争态势"这类复杂请求，没有规划能力的智能体要么无从下手，要么随机执行部分操作，最终交付不完整的分析。缺乏从高层目标到具体行动的映射能力，导致输出质量无法保证。

🔍 信息缺口无法识别

简单智能体不知道自己不知道什么，无法主动补全信息缺口。

Google DeepResearch 的核心能力之一是在研究过程中主动识别知识缺口，并动态生成新的搜索查询来填补这些缺口。没有规划能力的智能体无法进行这种元认知，导致研究不充分、结论片面。

解决方案：规划模式

规划模式通过让智能体先制定计划再执行，赋予其处理复杂任务的能力。点击各节点了解规划流程的每个阶段。

1
理解目标与约束
状态评估阶段
智能体首先理解初始状态（现有资源、已知信息、当前条件）和目标状态（期望的最终结果）。同时识别约束条件——预算、时间限制、可用工具等。这是规划的基础：清楚地知道"从哪里来"和"要到哪里去"。例如：组织团队外出活动——初始状态是预算、人数、日期偏好；目标状态是成功预订的活动安排。
2
生成行动计划
计划制定阶段
基于对目标和约束的理解，智能体生成一个具体的多步骤行动计划。这个计划将高层目标分解为一系列有序的、可执行的子任务，并考虑步骤间的依赖关系。LLM 特别擅长这个阶段：基于海量训练数据，它们能生成合理有效的行动序列。计划通常呈现为有序列表或树状结构，如 DeepResearch 生成的研究计划。
3
执行计划步骤
计划执行阶段
智能体按计划顺序执行各步骤，调用必要的工具或与各类系统交互。在执行过程中，智能体持续监控执行状态，收集新信息，并评估每步的结果是否符合预期。执行不是机械地遵循脚本，而是在结构化框架内保持对新信息的响应能力。
4
动态适应与重新规划
自适应调整阶段
当遇到障碍（首选资源不可用、步骤失败、发现新信息）时，智能体不会简单失败，而是重新评估情况，调整计划，寻找替代路径。这是规划模式相较于固定工作流最关键的优势——对不可预见情况的适应能力。DeepResearch 在搜索过程中识别信息缺口并动态调整搜索策略，就是这种能力的体现。
5
综合输出最终结果
结果交付阶段
完成所有计划步骤后，智能体将各阶段的执行结果综合为连贯的最终输出。这可能是一份结构化报告、一系列完成的操作，或一个综合性的答案。综合阶段确保最终输出不只是碎片化的步骤结果之和，而是有机整合的完整解决方案。

核心概念

掌握规划模式所需的关键技术概念与术语。

目标导向行为（Goal-Directed Behavior）

规划模式使智能体从被动反应转变为主动的目标导向行为。智能体不只是响应当前输入，而是始终朝着明确定义的目标状态推进。这要求智能体能够维护"距离目标还有多远"的内部状态表示，并据此选择下一步行动。

任务分解（Task Decomposition）

将高层目标拆解为可管理的子任务序列是规划的核心操作。有效的分解需要识别步骤间的依赖关系（哪些步骤必须先完成），以及可以并行的步骤。LLM 的优势在于能够基于训练数据中积累的大量问题解决模式，生成合理的分解方案。

# CrewAI 中的任务分解示例 task = Task( description=( f"1. 为主题 '{topic}' 创建大纲\n" f"2. 按大纲撰写约200字的摘要" ), expected_output="包含计划和摘要的最终报告", agent=planner_writer_agent )

迭代细化（Iterative Refinement）

规划并非一次性静态生成，而是迭代细化的过程。随着执行推进，智能体获取新信息，据此评估和修正计划。Google DeepResearch 在每轮搜索后识别信息缺口，生成新查询，这种搜索-评估-优化循环就是迭代细化的典型体现。

灵活性与可预测性的权衡

动态规划提供了处理未知情况的灵活性，但也引入了不可预测性。当问题的解决路径已知且可重复时，固定工作流更可靠——它限制自主性以保证一致性。关键判断：任务的"如何做"是否需要被实时发现？答案是则使用规划，否则使用固定工作流。

应用场景

规划模式在需要多步骤推理和动态适应的复杂场景中发挥关键作用。选择场景深入了解。

深度研究与信息综合

Google DeepResearch 是规划模式在信息综合领域的代表性应用。

1
计划生成：将用户查询分解为多步研究计划，呈现给用户审阅和修改，实现协作式研究方向塑造
2
迭代搜索：动态执行搜索，基于已获取的信息主动识别知识缺口，调整后续搜索策略
3
信息评估：对收集到的数据进行相关性和知识缺口评估，进行后续搜索以弥补缺口
4
综合报告：将经过验证的信息整合为带引用的结构化多页报告，支持音频概述、图表等交互功能

程序性任务自动化

将复杂业务流程（如入职新员工）分解为有序的可执行步骤序列。

1
流程分析：智能体分解业务流程，识别步骤依赖关系（如：先创建账户，才能分配权限）
2
有序执行：按依赖关系有序调用各系统 API，管理跨部门协调
3
异常处理：遇到步骤失败时，重新规划替代路径，确保整体目标达成

机器人与自主导航

规划在机器人学中是状态空间遍历的基础——从初始状态到目标状态的最优路径发现。

1
状态定义：定义初始位置、目标位置及环境约束（障碍物、交通规则）
2
路径规划：生成满足约束的最优行动序列，优化时间或能耗等指标
3
动态重规划：遇到新障碍时实时调整路径，保持向目标推进

多步骤客户支持

复杂客户问题的诊断与解决往往需要系统性的多步骤规划。

1
问题诊断：制定系统性诊断计划，逐步排查潜在原因
2
解决方案执行：按计划实施解决方案，必要时调用工具或升级处理
3
结果验证：验证解决效果，若未解决则执行升级处理流程

竞争态势分析

DeepResearch 风格的规划特别适合需要系统性多源数据收集的竞争分析。

1
分析框架制定：生成覆盖市场趋势、竞品规格、用户情感、营销策略的多维分析计划
2
系统性数据收集：自动化执行各维度数据收集，替代人工跟踪多个竞争对手的繁琐工作
3
战略性综合：将数据整合为支持战略决策的综合分析报告

学术文献综述

在学术探索中，规划型智能体能够系统性地进行大规模文献综述。

1
研究框架规划：制定涵盖关键文献识别、概念发展追踪、研究前沿映射的综述计划
2
文献系统检索：跨多个数据库系统性检索，处理数量远超人工能力的文献来源
3
知识综合：将文献发现综合为结构化综述，减少选择性偏误，揭示不易察觉的关键信息

代码实战

以下展示两种规划模式实现：CrewAI 框架的规划型智能体，以及 OpenAI Deep Research API 的调用示例。

CrewAI — 规划写作智能体

使用 CrewAI 创建一个先制定计划再撰写摘要的智能体，演示显式规划步骤如何提升输出质量。

import os from dotenv import load_dotenv from crewai import Agent, Task, Crew, Process from langchain_openai import ChatOpenAI load_dotenv() llm = ChatOpenAI(model="gpt-4-turbo") # 定义具有规划能力的智能体 planner_writer_agent = Agent( role='文章规划师与撰写者', goal='针对指定主题先制定计划，再撰写简洁有力的摘要', backstory=( '您是一位专业的技术写作者和内容策略师。' '您的优势在于先制定清晰可操作的计划，' '确保最终摘要既信息丰富又易于理解。' ), verbose=True, allow_delegation=False, llm=llm ) # 定义包含显式规划步骤的任务 topic = "强化学习在人工智能中的重要性" high_level_task = Task( description=( f"1. 为主题 '{topic}' 创建要点大纲计划\n" f"2. 按照计划撰写约200字的摘要" ), expected_output=( "包含两个独立部分的最终报告：\n\n" "### 计划\n" "- 摘要主要观点的要点列表\n\n" "### 摘要\n" "- 结构清晰、内容完整的主题摘要" ), agent=planner_writer_agent, ) crew = Crew( agents=[planner_writer_agent], tasks=[high_level_task], process=Process.sequential, ) result = crew.kickoff() print(result)

OpenAI Deep Research API

OpenAI 的深度研究 API 内置了完整的规划-搜索-综合流水线，可编程访问自主研究能力。

from openai import OpenAI client = OpenAI(api_key="YOUR_OPENAI_API_KEY") system_message = """您是一位专业研究员，负责撰写结构化的数据驱动报告。 请聚焦于数据丰富的洞察，使用可靠来源，并包含行内引用。""" user_query = "研究司美格鲁肽对全球医疗体系的经济影响" # 调用 Deep Research API response = client.responses.create( model="o3-deep-research-2025-06-26", input=[ { "role": "developer", "content": [{"type": "input_text", "text": system_message}] }, { "role": "user", "content": [{"type": "input_text", "text": user_query}] } ], reasoning={"summary": "auto"}, tools=[{"type": "web_search_preview"}] ) # 获取最终报告 final_report = response.output[-1].content[0].text print(final_report) # 获取引用注释 annotations = response.output[-1].content[0].annotations for i, citation in enumerate(annotations): cited_text = final_report[citation.start_index:citation.end_index] print(f"引用 {i+1}: {citation.title} - {citation.url}")

深入解析

规划模式在实际应用中涉及多个深层技术考量。点击各层了解详细分析。

Google DeepResearch 架构
规划模式的工业级实现
Google Gemini DeepResearch 将用户查询分解为多点研究计划，呈现给用户审阅修改（协作塑造研究方向），获批后启动迭代搜索分析循环。系统动态制定并优化搜索查询，主动识别知识缺口并填补，处理数百个来源后生成带引用的结构化报告。关键设计是异步处理——调查过程可能涉及分析数百个来源，系统对单点故障具有弹性，用户可断开连接并在完成时收到通知。
OpenAI Deep Research API 特性
透明度与可扩展性
与 ChatGPT 抽象化的流程不同，OpenAI Deep Research API 暴露所有中间步骤：智能体的推理过程、执行的具体搜索查询，以及运行的任何代码。这允许详细调试、分析和深度理解最终答案的构建过程。API 支持 Model Context Protocol（MCP），允许开发者将智能体连接到私有知识库和内部数据源，将公开网络研究与专有信息融合。
LLM 的规划能力来源
为何 LLM 擅长规划？
LLM 特别适合规划任务，因为它们在训练数据中接触了海量的问题解决模式、项目计划、流程文档和方法论指南。这使它们能够基于任务描述生成合理的行动计划，而无需手动编码每种场景的处理逻辑。通过明确提示要求规划步骤（如 CrewAI 任务描述中的"先创建计划，再撰写"），开发者可以有意识地激活这种规划行为，显著提升复杂任务的输出质量。
规划深度与任务复杂性
从简单规划到复杂迭代规划
规划的深度应与任务复杂性匹配。CrewAI 示例展示了简单的两步规划（制定大纲→写作），适合中等复杂任务。DeepResearch 则代表高级迭代规划：计划在执行过程中动态演化，每轮搜索结果都会影响后续计划。对于最简单的任务（单步操作），规划开销可能超过其带来的收益。判断标准：任务是否有多个相互依赖的步骤，且每步结果会影响后续决策？若是，则规划是有价值的。
引用与透明度
可验证性对规划输出的重要性
高质量的规划型研究系统必须提供可验证的引用。DeepResearch 明确返回所搜索和参考的完整来源列表，OpenAI API 通过注释（annotations）提供行内引用，包含被引文本的精确位置（start_index、end_index）、标题和 URL。这种透明度对于商业决策、学术研究或政策建议等高风险应用场景至关重要——用户需要能够验证 AI 生成结论的来源依据。

一览概要

快速掌握规划模式的核心要素，以及何时、为何、如何使用它。

问题（WHAT）

复杂问题往往无法通过单次操作解决，需要预见性来实现期望结果。没有结构化方法，智能体难以将高层目标分解为可执行步骤，无法处理多步骤依赖关系，面对障碍时也无法有效重规划，导致复杂任务的结果不完整或错误。

方案（WHY）

规划模式通过让智能体首先制定连贯计划来应对目标，将高层目标分解为有序的可操作步骤或子目标。LLM 尤其擅长此类任务：基于海量训练数据生成合理计划，管理复杂工作流，协调各类工具，并按逻辑顺序处理依赖关系。

适用时机（WHEN）

当用户请求过于复杂、无法通过单次操作或工具调用处理时使用。适用于自动化多步流程（如生成详细研究报告、入职新员工、执行竞争分析），以及任何需要一系列相互依赖操作才能达成最终综合结果的场景。

经验法则（HOW）

关键问题：解决方案的"如何做"是否需要被发现，还是已经明确已知？若需被发现（高度复杂动态任务），使用规划。若已知（可重复可预测流程），使用固定工作流——它更可靠、更可预测。规划牺牲部分可预测性以换取灵活性。

核心要点

本章关于规划模式最重要的收获。

  • 规划模式是将智能体从简单反应式响应者提升为战略性目标导向执行者的基础性组件，使其能够处理需要多步骤推理的复杂任务。
  • 规划的核心是任务分解：将高层目标分解为有序的、可执行的子任务序列，同时识别步骤间的依赖关系。
  • LLM 特别擅长规划，因为它们在训练数据中接触了海量的问题解决模式、项目计划和方法论指南，能够生成合理的行动序列。
  • 规划模式的标志性优势是适应性：当遇到障碍时，智能体重新评估情况并调整计划，而非直接失败——这是固定工作流无法提供的能力。
  • 关键决策标准是：解决方案的"如何做"是否需要被实时发现？若是，选择规划；若已明确，选择固定工作流（更可靠、更可预测）。
  • Google DeepResearch 代表规划模式的高级应用：迭代计划生成、动态搜索策略调整、知识缺口识别，以及带引用的结构化报告综合。
  • OpenAI Deep Research API 将这种能力程序化，提供透明的中间步骤（推理、搜索查询、代码执行）和可验证的行内引用，支持高风险应用场景。
  • 规划扩展了智能体的行动边界——从简单 CrewAI 双步规划（制定大纲→撰写）到复杂的 DeepResearch 式多轮迭代研究，均体现了这一模式的核心价值。

深入理解 AI Agent 的核心：规划（Planning）设计模式的第一性原理

引言（具体痛点场景）

想象一下这样一个场景：你对你的 AI 助手说："帮我组织一次团队的异地团建（offsite）。" 如果这是一个单纯的反应式（Reactive）AI 模型，它可能会立刻开始根据其训练数据预测下一个 Token，最终吐出一份长篇大论的"团建建议清单"或者是几个酒店的推荐链接。但这并不是你真正想要的。

在真实的业务环境中，复杂的请求往往无法通过单一的动作或工具调用来解决。如果没有一种结构化的方法，Agentic（代理化）系统在面对涉及多个步骤和复杂依赖关系的多方面请求时，就会显得捉襟见肘。它无法将高层次的目标分解为一系列可管理的、可执行的微小任务。其结果就是，系统无法有效地制定战略，在面对错综复杂的目标时，往往只能给出不完整或完全错误的结果。

智能行为不仅仅是对即时输入的被动反应，它还需要远见（Foresight）。这就是为什么我们需要引入规划（Planning）设计模式。今天，我们将剥开高级框架的表象，从第一性原理出发，深入剖析为什么 AI 系统需要学会"三思而后行"，它是如何工作的，以及如何在代码中实现它。

问题陈述

在 AI 的语境下，我们可以将规划 Agent 视为一个你向其委派复杂目标的"专家"。当你要求它"组织团队异地团建"时，你实际上只是在定义"是什么"（What）——即最终的目标（成功预订团建）以及相关的约束条件（预算、参与人数、期望日期）。

然而，你并没有告诉它"怎么做"（How）。

核心问题在于：系统必须自主地规划出通向该目标的路径。在收到请求之前，这个计划是不存在的；它是针对该请求动态生成的。AI 需要一种机制，能够将人类高层次的意图转化为由离散的、可执行的步骤组成的结构化工作流，这就是我们在构建复杂 AI 系统时面临的核心工程挑战。

第一性原理推导：为什么要规划？

让我们回到最基础的计算理论。什么是"规划"？

从第一性原理来看，规划是系统从"初始状态（Initial State）"向"目标状态（Goal State）"移动时，制定行动序列的能力。这本质上是一个状态空间遍历（State-space Traversal）的过程。

想象一个在物理世界中导航的机器人，或者一个在虚拟信息网络中穿梭的虚拟实体。为了从 A 点到达 B 点，它必须生成一条路径或一系列动作。在这个过程中，它必须优化诸如时间或能量消耗等指标，同时还要遵守环境约束（例如避开障碍物或遵守交通规则）。

对于大语言模型（LLM）而言，如果没有规划，它就像一个蒙着眼睛在迷宫里狂奔的人，每一步都在依靠直觉（预测下一个词的概率）做出局部最优的选择，但很容易陷入死胡同。规划模式（Planning Pattern）提供了一个标准化的解决方案：它强制 Agentic 系统在采取任何实际行动之前，首先创建一个连贯的计划。它将高维、模糊的意图"降维"为一条清晰的逻辑执行链。由于 LLM 吸收了海量的训练数据，它们非常擅长根据任务描述生成合理且有效的逐步计划。

核心概念详解

要彻底掌握规划模式，我们需要深刻理解以下几个核心概念。在这里，我们不仅要问"是什么"，更要问"为什么"。

1. 任务拆解与逻辑编排 (Decomposition & Orchestration)

是什么：规划涉及将高层次目标分解为一系列更小、可操作的步骤或子目标。例如，在一个程序化的任务自动化场景中（如新员工入职），目标会被分解为：创建系统账户、分配培训模块、与不同部门协调等有向的子任务序列。

为什么：因为复杂的系统调用和工具使用存在严格的依赖关系。你不能在员工账户创建之前就为其分配培训模块。Agent 必须生成一个计划，以合乎逻辑的顺序执行这些步骤，并调用必要的工具或与其他系统交互来管理这些依赖。

2. 动态适应性 (Adaptability)

是什么：一个初始计划仅仅是一个起点，而不是一个僵化的剧本（Rigid script）。

为什么：真实世界充满了不可预测的变量。如果 Agent 首选的场地不可用，或者选定的餐饮服务商已被订满，一个具备规划能力的 Agent 绝不能简单地抛出错误并崩溃。它必须适应。它会记录下这个新的约束条件，重新评估其选项，并制定一个新的计划（例如建议备选场地或日期）。这种在执行过程中不断根据新信息调整航向的能力，才是规划 Agent 真正的威力所在。

3. 目标导向行为 (Goal-Oriented Behavior)

是什么：系统不再仅仅是"回答问题"，而是"解决问题"。

为什么：在结构化信息合成（例如生成复杂的研究报告）或多步客户支持（诊断、实施解决方案、升级）等场景中，单次对话是不够的。规划模式允许 Agent 超越简单的反应式动作，提供一个逻辑框架来解决需要一系列相互依赖的操作才能完成的复杂问题。

关键机制分析：Google DeepResearch 是如何思考的？

为了理解规划模式在顶尖系统中的运作机制，让我们深度剖析 Google Gemini DeepResearch 系统。这不是一个简单的"一问一答"模型，而是一个为了自主信息检索和综合而设计的 Agent 系统。

1. 意图解构与人机协作规划：这个系统并不是瞬间完成所有的动作。它是一个受管理的、长时间运行的过程。为什么这么设计？ 因为研究任务的容错率很低。它首先将用户的提示（Prompt）解构为一个包含多个要点的研究计划（Research Plan）。最关键的是，这个计划会被呈现给用户进行审查和修改。这种机制允许在真正执行极其昂贵的搜索任务之前，进行研究轨迹的"协作塑造"。

2. 动态迭代搜索循环 (Iterative Search-and-Analysis Loop)：一旦计划被批准，Agentic 流水线就会启动。它不仅仅是执行一堆预定义的搜索；相反，Agent 会根据收集到的信息动态地制定和改进其查询。为什么？ 因为未知的知识领域往往是随着探索而展开的。Agent 必须主动识别知识空白（Knowledge Gaps）、确证数据点（Corroborating data points）并解决差异。

3. 异步管理与容错架构 (Asynchronous Management)：DeepResearch 的一个关键架构组件是异步管理整个过程的能力。为什么？ 因为一项深入的研究可能涉及分析数百个来源。如果采用同步阻塞的方式，任何一个网络超时都可能导致整个任务崩溃。异步设计确保了系统对单点故障的弹性，并允许用户在任务进行时离开，在完成时收到通知。

4. 批判性综合与透明引用 (Critical Evaluation & Citations)：收集数据只是第一步，综合阶段才是规划模式的终点。模型不仅是对结果进行字符串拼接，而是识别主要主题，将其组织成连贯的叙事和逻辑部分。同时，它显式地返回查阅过的完整来源列表，并作为引用（Citations）呈现。为什么？ 在企业和学术应用中，信任比智能更重要。提供透明度和对原始信息的直接访问，使得系统生成的结构化多页报告不仅详尽，而且可验证。通过自动化这种迭代的搜索-过滤循环（这是人工研究的核心瓶颈），系统可以在相同时间内处理比人类研究员大得多的信息量，从而减少选择偏差（Selection bias），带来更稳健的理解。

框架实战代码示例

理论已经足够，让我们来看看代码是如何实现这一模式的。

示例 1：使用 Crew AI 构建顺序规划 Agent

import os
from dotenv import load_dotenv
from crewai import Agent, Task, Crew, Process
from langchain_openai import ChatOpenAI

load_dotenv()

llm = ChatOpenAI(model="gpt-4-turbo")

planner_writer_agent = Agent(
    role='Article Planner and Writer',
    goal='Plan and then write a concise, engaging summary on a specified topic.',
    backstory=(
        'You are an expert technical writer and content strategist. '
        'Your strength lies in creating a clear, actionable plan before writing, '
        'ensuring the final summary is both informative and easy to digest.'
    ),
    verbose=True,
    allow_delegation=False,
    llm=llm
)

topic = "The importance of Reinforcement Learning in AI"
high_level_task = Task(
    description=(
        f"1. Create a bullet-point plan for a summary on the topic: '{topic}'.\n"
        f"2. Write the summary based on your plan, keeping it around 200 words."
    ),
    expected_output=(
        "A final report containing two distinct sections:\n\n"
        "### Plan\n"
        "- A bulleted list outlining the main points of the summary.\n\n"
        "### Summary\n"
        "- A concise and well-structured summary of the topic."
    ),
    agent=planner_writer_agent,
)

crew = Crew(
    agents=[planner_writer_agent],
    tasks=[high_level_task],
    process=Process.sequential,
)

result = crew.kickoff()
print(result)

示例 2：揭秘 OpenAI Deep Research API 的内部思考

from openai import OpenAI

client = OpenAI(api_key="YOUR_OPENAI_API_KEY")

system_message = """You are a professional researcher preparing a structured, data-driven report."""
user_query = "Research the economic impact of semaglutide on global healthcare systems."

response = client.responses.create(
    model="o3-deep-research-2025-06-26",
    input=[
        {"role": "developer", "content": [{"type": "input_text", "text": system_message}]},
        {"role": "user", "content": [{"type": "input_text", "text": user_query}]}
    ],
    reasoning={"summary": "auto"},
    tools=[{"type": "web_search_preview"}]
)

# 揭秘：检查中间步骤
print("--- INTERMEDIATE STEPS ---")

try:
    reasoning_step = next(item for item in response.output if item.type == "reasoning")
    print("\n[Found a Reasoning Step]")
    for summary_part in reasoning_step.summary:
        print(f"  - {summary_part.text}")
except StopIteration:
    print("\nNo reasoning steps found.")

try:
    search_step = next(item for item in response.output if item.type == "web_search_call")
    print("\n[Found a Web Search Call]")
    print(f"  Query Executed: '{search_step.action['query']}'")
    print(f"  Status: {search_step.status}")
except StopIteration:
    print("\nNo web search steps found.")

决策指南与最佳实践

引入规划能力会增加系统的延迟和计算成本，因此我们需要基于第一性原理来做架构决策。决策的核心法则只有一个：用户的请求是否过于复杂，以至于无法通过单个操作或工具来处理？

或者换一种问法：解决该问题的"方法（How）"是需要被探索发现的，还是已经完全已知并固化的？

应该使用规划模式的场景：

  • 多步流程自动化：如撰写详细的研究报告、执行竞争对手分析（汇总市场趋势、竞品规格、公众情绪）。
  • 复杂的环境遍历：无论是物理机器人的导航，还是虚拟软件环境中的多 API 依赖调用。

最佳实践：在企业级应用中，利用如模型上下文协议（MCP, Model Context Protocol）来扩展 Agent，使其可以连接到内部私有知识库，将公共网络的深入研究与企业专有数据相融合。

常见陷阱

在学习了强大的规划能力后，工程师最容易犯的错误就是"过度工程（Over-engineering）"。我们必须清醒地认识到：动态规划是一种特定的工具，而不是万能的灵丹妙药（Universal solution）。

这里存在一个根本的权衡（Trade-off）：灵活性（Flexibility）与可预测性（Predictability）之间的矛盾。

陷阱：在不需要思考的地方让 Agent 思考。

为什么这是错的：当一个问题的解决方案已经是被充分理解且高度可重复的（Well-understood and repeatable）时，强迫 Agent 进行动态规划反而引入了巨大的系统熵值。在这些场景下，将 Agent 约束在一个预先确定的、固定的工作流（Fixed workflow）中才是更有效的做法。这限制了 Agent 的自主权，从而减少了不确定性和不可预测行为的风险，保证了可靠、一致的输出结果。永远不要为了使用 Agent 而使用 Agent；对于一个固定格式的每日天气抓取脚本，传统的 Cron Job 加上简单的 API 调用远比一个复杂的 Planning Agent 优秀得多。

总结

在构建当代人工智能系统时，规划（Planning）模式扮演着基石般的角色。它成功地将 Agent 系统从简单的被动响应者（Reactive responders）升维到了具备战略眼光的、目标导向的执行者（Strategic, goal-oriented executors）。

借助现代大语言模型强大的逻辑演绎能力，系统能够自主地将复杂的高层目标拆解为连贯、可操作的步骤。正如我们从 CrewAI 中简单的强制顺序计划生成，一路看到 Google DeepResearch 和 OpenAI Deep Research API 中极度复杂的动态迭代研究、自我修正、透明化溯源与代码执行。

归根结底，规划不仅是一种代码模式，它构成了人类意图与自动化执行复杂问题之间的核心桥梁。通过将问题求解过程结构化，规划模式赋予了 Agent 统筹错综复杂的工作流并最终交付综合性卓越成果的能力。当你下次再对系统说出那个模糊的"What"时，一个设计精良的 Planning Agent，已经在它的"大脑"深处为你推演了千万条通向目标的"How"。
