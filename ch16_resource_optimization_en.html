What is Resource-Aware Optimization?

Resource-Aware Optimization enables intelligent agents to dynamically monitor and manage computational, temporal, and financial resources during operation. This differs from simple planning, which primarily focuses on action sequencing. Resource-Aware Optimization requires agents to make decisions regarding action execution to achieve goals within specified resource budgets or to optimize efficiency.

Choosing the Right Model for the Right Task

This involves choosing between more accurate but expensive models and faster, lower-cost ones, or deciding whether to allocate additional compute for a more refined response versus returning a quicker, less detailed answer. For example, if a financial analyst needs a preliminary report immediately, the agent uses a faster, affordable model. If high accuracy is required for a critical investment decision, the agent allocates more resources to a powerful, slower, but more precise model.

Fallback Mechanisms for Graceful Degradation

A key strategy is the fallback mechanism — a safeguard when a preferred model is unavailable due to being overloaded or throttled. To ensure graceful degradation, the system automatically switches to a default or more affordable model, maintaining service continuity instead of failing completely. This makes systems resilient to external API constraints.

Why This Matters

LLM-based applications can be expensive and slow, and using the best model for every task is often inefficient. This pattern creates a fundamental trade-off management system between the quality of output and the resources required to produce it, enabling scalable, economically viable AI systems in production.

The Problem: Static Resource Usage
Without dynamic resource management, agentic systems suffer from several critical failure modes. Click each card to learn more:

Budget Overruns

Always using the most powerful model exhausts API budgets

When every query — regardless of complexity — is routed to the most expensive model, costs scale linearly with volume. A simple factual lookup costs as much as a multi-step reasoning task, making production deployment unsustainable.

Latency for Simple Queries

Powerful models introduce unnecessary delay

In real-time systems, routing a simple "What is the capital of France?" to a 70B-parameter model incurs hundreds of milliseconds of unnecessary latency. Users experience slow responses even for trivial queries.

Service Interruptions

No fallback when primary model is unavailable

If the primary model provider hits rate limits, experiences downtime, or filters the request, a system without fallback logic simply fails. This creates unacceptable reliability gaps for production applications.

Edge Device Constraints

Heavy models drain battery and exceed memory

Agents deployed on IoT devices, mobile phones, or edge hardware face strict compute and power budgets. Running large models on-device is infeasible; without optimization, the agent simply cannot operate in these environments.

Inefficient Multiagent Load

Agents ignore each other's current load

In multi-agent systems, tasks may be assigned to agents that are already at capacity, while idle agents go unused. Without adaptive task allocation based on current computational load, the system is inefficient.

Data Transfer Waste

Downloading full datasets when summaries suffice

Agents that always retrieve full datasets when summarized data would answer the question waste bandwidth and storage. This is particularly costly in cloud environments where data egress is metered.

The Solution: Intelligent Resource Routing
Resource-Aware Optimization builds a multi-agent architecture where a Router Agent classifies queries and routes them to the appropriate model. Click each step to see details:
1
Query Classification
Router Agent assesses complexity
How: The Router Agent examines the incoming query — analyzing word count, semantic complexity, whether it requires factual recall vs. multi-step reasoning, or whether it needs current internet data.

Simple approach: Word count threshold (e.g., <20 words → Flash, ≥20 words → Pro)
Advanced approach: Use a fast LLM to classify the query into categories: simple, reasoning, or internet_search

Output: A routing decision with the target model identifier
2
Model Dispatch
Route to cost-appropriate agent
Simple queries → Gemini Flash / GPT-4o-mini: Fast, cheap, sufficient for factual recall

Reasoning queries → Gemini Pro / o4-mini: Slower, more expensive, handles multi-step logic

Internet search queries → GPT-4o + Google Search API: Retrieves current information before answering

The dispatch step also handles the Critique Agent loop — evaluating response quality and potentially re-routing for improvement.
3
Fallback & Graceful Degradation
Maintain service under constraints
Sequential fallback: If the primary model is rate-limited or unavailable, the system automatically tries the next model in a priority list.

Example with OpenRouter: {"models": ["anthropic/claude-3.5-sonnet", "gryphe/mythomax-l2-13b"]}

The system returns the response from whichever model succeeds first, logging which model was ultimately used. This ensures continuity without manual intervention.
Core Concepts
Six foundational concepts enable resource-aware optimization. Click each layer to expand:
Dynamic Model Switching
Select models based on task complexity and available resources
When faced with simple queries, a lightweight, cost-effective LLM is deployed. Complex, multifaceted problems use more sophisticated and resource-intensive models. The switching logic can be rule-based (word count) or ML-driven (a classifier LLM that analyzes query nuances).
Adaptive Tool Use & Selection
Choose tools based on API cost, latency, and execution time
Agents intelligently choose from a suite of tools, selecting the most appropriate and efficient one for each specific sub-task. Factors include API usage costs, latency requirements, and execution time. This dynamic selection enhances overall system efficiency by optimizing external API and service usage.
Contextual Pruning & Summarization
Minimize prompt token count to reduce inference costs
Strategically minimizing the prompt token count by intelligently summarizing and selectively retaining only the most relevant information from the interaction history. This prevents unnecessary computational overhead and keeps costs down as conversations grow longer.
Proactive Resource Prediction
Forecast future workloads to prevent bottlenecks
Anticipating resource demands by forecasting future workloads and system requirements. This allows for proactive allocation and management of resources, ensuring system responsiveness and preventing bottlenecks before they occur.
Learned Resource Allocation Policies
Improve allocation strategies over time via feedback
A learning mechanism enabling agents to adapt and optimize their resource allocation strategies over time based on feedback and performance metrics. The Critique Agent's consistent identification of suboptimal routing (e.g., routing complex queries to Flash) feeds back to improve the router's logic through fine-tuning.
Graceful Degradation & Fallback
Continue functioning even under severe resource constraints
Intelligent agent systems can continue to function, albeit at reduced capacity, even when resource constraints are severe. The system gracefully degrades performance and falls back to alternative strategies to maintain operation and provide essential functionality — never failing completely.
Practical Applications
Resource-Aware Optimization applies across many real-world domains. Click tabs to explore:

Cost-Optimized LLM Usage

An agent deciding whether to use a large, expensive LLM for complex tasks or a smaller, more affordable one for simpler queries, based on a budget constraint.

1
Classify incoming query as simple, reasoning, or search-required
2
Route simple queries to GPT-4o-mini or Gemini Flash (10x cheaper)
3
Reserve GPT-4 or Gemini Pro for complex reasoning tasks
4
Track cumulative cost and enforce budget limits per session

Latency-Sensitive Real-Time Operations

In real-time systems, an agent chooses a faster but potentially less comprehensive reasoning path to ensure a timely response.

1
Measure query urgency based on SLA requirements (e.g., <500ms)
2
If latency budget is tight, skip the Critique Agent loop
3
Use streaming API responses to reduce perceived latency
4
Cache frequent answers to eliminate model calls entirely

Energy Efficiency on Edge Devices

For agents deployed on edge devices or with limited power, optimizing processing to conserve battery life is essential.

1
Monitor battery level and thermal state of the device
2
Switch to local, quantized small models (e.g., Gemma 2B) when battery <20%
3
Defer non-urgent processing to when the device is charging
4
Compress context windows to reduce compute requirements

Hierarchical Travel Planner

High-level planning uses a powerful model; repetitive tool calls use a cheaper model.

1
Gemini Pro (Planner Agent): Understand complex multi-city itinerary request, break into steps, reason about constraints
2
Gemini Flash (Tool Agents): Look up flight prices, hotel availability, restaurant reviews — simple web queries
3
Gemini Pro (Synthesizer): Combine results into a coherent final travel plan

Result: 60–70% cost reduction with no loss in plan quality for the complex reasoning portions.

Adaptive Multi-Agent Task Allocation

In multi-agent systems, agents self-assign tasks based on their current computational load or available time.

1
Each agent reports its current load (queue depth, CPU usage)
2
Orchestrator routes new tasks to the least-loaded capable agent
3
Agents with lower-spec models handle simpler subtasks; specialists handle complex ones
4
System scales horizontally by adding agents during peak load
Hands-On Code
The following examples show resource-aware routing implemented with Google ADK and OpenAI.

Two Agents: Pro vs Flash (Google ADK)

Define two agents with identical setups but different models and cost profiles:

# Conceptual Python structure (Google ADK) from google.adk.agents import Agent # High-capability agent for complex queries gemini_pro_agent = Agent( name="GeminiProAgent", model="gemini-2.5-pro", description="A highly capable agent for complex queries.", instruction="You are an expert assistant for complex problem-solving." ) # Cost-effective agent for simple queries gemini_flash_agent = Agent( name="GeminiFlashAgent", model="gemini-2.5-flash", description="A fast and efficient agent for simple queries.", instruction="You are a quick assistant for straightforward questions." )

Router Agent: Word-Count Heuristic

A QueryRouterAgent routes based on query length as a simple complexity proxy:

class QueryRouterAgent(BaseAgent): name: str = "QueryRouter" description: str = "Routes queries to appropriate LLM based on complexity." async def _run_async_impl(self, context: InvocationContext): user_query = context.current_message.text query_length = len(user_query.split()) if query_length < 20: # Simple query threshold response = await gemini_flash_agent.run_async( context.current_message ) yield Event(author=self.name, content=f"Flash Agent: {response}") else: # Complex query → Pro model response = await gemini_pro_agent.run_async( context.current_message ) yield Event(author=self.name, content=f"Pro Agent: {response}")

OpenAI: 3-Category Router with Internet Search

Classifies each query into simple / reasoning / internet_search, then routes accordingly:

def classify_prompt(prompt: str) -> dict: system_msg = { "role": "system", "content": ( "Classify user prompts into ONE of three categories:\n" "- simple: direct factual questions\n" "- reasoning: logic or multi-step inference\n" "- internet_search: requires current data\n\n" 'Respond ONLY with JSON: { "classification": "simple" }' ) } response = client.chat.completions.create( model="gpt-4o", messages=[system_msg, {"role": "user", "content": prompt}] ) return json.loads(response.choices[0].message.content) def generate_response(prompt: str, classification: str, search_results=None) -> tuple: if classification == "simple": model = "gpt-4o-mini" # Cheapest elif classification == "reasoning": model = "o4-mini" # Reasoning model else: # internet_search model = "gpt-4o" # Needs context understanding return response, model

Dynamic Model Switching: The strategic selection of LLMs based on task complexity and available resources. Lightweight models handle simple queries; sophisticated models handle complex, multifaceted problems. The key is making this switch automatically based on measurable signals.

The Critique Agent

The Critique Agent evaluates responses from language models, providing feedback for multiple purposes: self-correction (identifying errors), performance monitoring (tracking accuracy/relevance metrics), and indirect budget management (identifying suboptimal routing choices, such as directing simple queries to a Pro model or complex queries to a Flash model, which leads to poor results). This informs adjustments that improve resource allocation and cost savings.

OpenRouter's Two Routing Modes

  • Automated Model Selection ("model": "openrouter/auto"): Routes to an optimized model chosen from a curated set based on prompt content. The model identifier is returned in the response metadata.
  • Sequential Model Fallback ("models": [...]): Specifies a hierarchical list. If the primary model fails (unavailable, rate-limited, content-filtered), the next model is tried automatically until one succeeds.

Energy-Efficient Deployment

Specifically tailored for environments with stringent resource constraints, this approach aims to minimize the energy footprint of intelligent agent systems, extending operational time and reducing overall running costs. For IoT and mobile deployments, this is not optional — it determines whether the agent can run at all.

At a Glance

WHAT

Resource-Aware Optimization manages consumption of computational, temporal, and financial resources in intelligent systems. LLM-based applications can be expensive and slow — selecting the best model for every task is often inefficient.

WHY

An agentic system that intelligently monitors and allocates resources based on the task at hand. A Router Agent classifies query complexity; a Critique Agent refines routing logic over time through feedback.

WHEN TO USE

Operating under strict financial budgets for API calls; latency-sensitive applications; agents on resource-constrained hardware (edge, mobile); complex multi-step workflows with varying resource requirements.

RULE OF THUMB

Use this pattern when the cost difference between models is 5x or greater, when latency SLAs vary by query type, or when operating in environments with hard resource ceilings (budget caps, battery limits, rate limits).

Architecture Summary

The pattern typically employs: (1) a Router Agent to classify incoming request complexity, (2) specialized Answering Agents at different capability/cost tiers, (3) a Critique Agent to evaluate quality and refine routing logic, and (4) a Fallback chain to ensure service continuity. This dynamic, multi-agent approach ensures the system operates efficiently, balancing response quality with cost-effectiveness.

Key Takeaways
  1. Resource-Aware Optimization is essential for production AI systems — intelligent agents can manage computational, temporal, and financial resources dynamically. Decisions about model usage and execution paths are made based on real-time constraints and objectives.
  2. Multi-Agent Architecture for Scalability: Google's ADK provides a multi-agent framework enabling modular design. Different agents (answering, routing, critique) handle specific tasks, making the system easy to scale and maintain.
  3. Dynamic, LLM-Driven Routing: A Router Agent directs queries to language models (Gemini Flash for simple, Gemini Pro for complex) based on query complexity and budget. This optimizes both cost and performance simultaneously.
  4. Critique Agent Functionality: A dedicated Critique Agent provides feedback for self-correction, performance monitoring, and refining routing logic, enhancing overall system effectiveness over time.
  5. Optimization Through Feedback and Flexibility: Evaluation capabilities for critique and model integration flexibility contribute to adaptive and self-improving system behavior.
  6. Graceful Degradation via Fallback Chains: Tools like OpenRouter enable sequential fallback across models, ensuring the system never fails completely when a primary model is unavailable, rate-limited, or filters content.
