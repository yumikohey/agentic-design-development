模型上下文协议（MCP）

一个开放标准，让大语言模型以统一、可互操作的方式连接外部工具、数据源和服务——从定制集成到通用适配器的跨越。

什么是模型上下文协议？

模型上下文协议（MCP）是一个开放标准，旨在标准化大语言模型（LLM）与外部应用程序、数据源和工具之间的通信方式。可以将其想象为一个通用适配器——让任何 LLM 无需为每个系统单独定制集成，就能接入任意外部系统、数据库或工具。

客户端–服务端架构

MCP 采用客户端–服务端架构运行。它定义了数据（资源）、交互模板（提示词）和可执行函数（工具）如何由 MCP 服务端暴露，并由 MCP 客户端（即 LLM 宿主应用或 AI 代理）消费。这种标准化方式大幅降低了将 LLM 集成到各种运行环境的复杂度。

MCP 的局限性

MCP 是"智能接口"的契约，其效果高度依赖于底层 API 的设计质量。如果开发者仅将已有的旧系统 API 原封不动地包装进 MCP，效果可能大打折扣。例如，若接口只支持逐条获取工单详情，代理在汇总高优先级工单时将极为缓慢。真正有效的 MCP 集成需要底层 API 同时提供过滤、排序等确定性功能。

数据格式的重要性

MCP 只负责建立连接，并不保证数据对代理友好。例如，若一个文档存储的 MCP 服务端以 PDF 格式返回文件，而消费代理无法解析 PDF 内容，则该服务端基本毫无用处。正确的做法是先创建一个返回 Markdown 等文本格式的 API，让代理能够真正读取和处理数据。

LLM 集成的碎片化困境
在 MCP 出现之前，每次将 LLM 与外部系统集成都是一次性的定制工程，带来了诸多问题。点击卡片查看详情：

重复的定制集成

每个 LLM 与工具的组合都需要独立开发

每家 LLM 提供商（OpenAI、Gemini、Claude 等）都有各自的工具调用格式，与每个外部系统的集成代码无法复用，工程成本极高。

可扩展性受限

新增工具需要修改每个 LLM 的集成代码

当系统需要接入新工具时，必须为每个在用的 LLM 分别修改集成代码，维护成本随工具数量呈指数级增长。

工具无法动态发现

LLM 只能使用预先硬编码的工具集

传统方式下，LLM 在每次对话时被明确告知可用工具的列表，无法在运行时动态发现新能力，灵活性大受限制。

遗留系统难以复用

旧有服务无法轻易纳入 AI 工作流

企业大量遗留系统和内部工具难以集成进现代 AI 代理，因为每次集成都需要大量定制开发，成本过高。

安全与权限管理混乱

每个集成都有各自的认证机制

没有统一标准时，每个工具集成都需要实现独立的认证和授权机制，安全审计困难，漏洞风险增加。

核心矛盾

LLM 要真正作为代理发挥作用，就必须超越文本生成——它需要访问实时数据、执行外部操作、与各类系统交互。但在没有统一标准的情况下，构建这种能力的复杂度让大多数应用望而却步。

MCP：通用连接标准
MCP 通过建立统一的客户端–服务端协议，解决了碎片化集成问题。点击每个步骤查看交互流程详情：
1
能力发现（Discovery）
MCP 客户端查询服务端
MCP 客户端（代表 LLM）向 MCP 服务端发起查询，询问其提供哪些能力。服务端返回一份清单，列出所有可用工具（如 send_email）、数据资源（如 customer_database）和提示词模板。

优势：这种"按需发现"机制让代理能够在不重新部署的情况下适配新能力，极大提升了系统的灵活性和可扩展性。
2
请求构造（Request Formulation）
LLM 决定使用某个工具
LLM 判断需要使用已发现的某个工具（例如发送邮件），随即构造一个符合 MCP 标准的请求，指定工具名称（send_email）和所需参数（收件人、主题、正文）。

标准化：请求格式完全由 MCP 规范定义，与具体的 LLM 提供商无关，实现真正的互操作性。
3
服务端执行（Server Execution）
MCP 服务端处理请求
MCP 服务端接收到请求后，完成身份验证、参数校验，然后执行具体操作（如调用邮件 API 的 send() 函数）。执行完成后，将标准化响应返回给客户端。

结果：响应指明操作是否成功，并包含相关输出（如邮件的确认 ID）。客户端将结果传回 LLM，更新上下文，推进任务执行。

联邦化模型的价值

通过采用联邦化模型，MCP 使各类遗留服务和独立工具只需包装一个 MCP 兼容接口，便能纳入现代 AI 生态。这些服务无需改造内部逻辑，即可被任意符合 MCP 标准的 LLM 代理发现和调用，极大降低了集成成本，提升了资产复用价值。

MCP 与工具函数调用的对比
MCP 和传统工具函数调用都是让 LLM 与外部能力交互的机制，但二者在设计理念和适用场景上存在重要差异。

工具函数调用（Tool Function Calling）

工具函数调用是 LLM 向特定预定义工具发出的直接请求。LLM 根据对用户意图的理解，格式化一个请求并由应用代码执行，结果返回给 LLM。这种方式往往是专有的，不同 LLM 提供商的实现各不相同。

MCP（模型上下文协议）

MCP 是一个标准化接口，让 LLM 能够发现、通信并使用外部能力。它是一个开放协议，目标是建立一个生态系统：任何符合标准的工具都能被任意符合标准的 LLM 使用，实现跨系统的互操作性和复用性。

MCP 的四个核心组件：

{ "组件": { "LLM": "核心智能体，处理用户请求、制定计划、决定何时访问外部信息", "MCP客户端": "LLM 的封装层，将 LLM 意图转化为符合 MCP 标准的请求，负责发现和连接服务端", "MCP服务端": "外部世界的网关，向授权客户端暴露工具、资源和提示词", "第三方服务": "MCP 服务端管理的实际外部工具或数据源（如数据库、邮件 API、天气服务）" } }

类比理解

工具函数调用就像给 AI 配备一套专用工具（特定的扳手和螺丝刀），适合任务固定的工作台。MCP 则像创建一套通用标准化电源插座系统——它本身不提供工具，但让任何厂商生产的合规工具都能即插即用，打造一个动态扩展的工作台生态。

实际应用与用例
MCP 大幅拓展了 AI/LLM 的能力边界，使其更加通用和强大。点击标签探索九大应用场景：

数据库集成

MCP 允许 LLM 和代理无缝访问结构化数据库中的数据。例如，通过 MCP Toolbox for Databases，代理可以查询 Google BigQuery 数据集以获取实时信息、生成报告或更新记录，所有操作均由自然语言指令驱动。

1
代理接收用户的自然语言查询请求
2
通过 MCP 客户端发现数据库服务端的可用工具
3
构造标准化查询请求并发送至 MCP 服务端
4
服务端执行数据库查询并返回结构化结果
5
代理将结果整合为用户可读的报告或答案

生成式媒体编排

MCP 使代理能够集成先进的生成式媒体服务。通过 MCP Tools for Genmedia Services，代理可以编排涉及 Google Imagen（图像生成）、Veo（视频创作）、Chirp 3 HD（语音合成）或 Lyria（音乐创作）的复杂工作流，在 AI 应用中实现动态内容创作。

外部 API 交互

MCP 为 LLM 调用任意外部 API 并获取响应提供了标准化通道。代理可以获取实时天气数据、查询股票价格、发送邮件，或与 CRM 系统交互，将其能力从核心语言模型大幅扩展至外部世界。

基于推理的信息提取

借助 LLM 强大的推理能力，MCP 支持高效的、依赖查询上下文的信息提取，超越传统搜索和检索系统。相比传统搜索工具返回整个文档，代理能够分析文本并精确提取直接回答用户复杂问题的具体条款、数据或陈述。

自定义工具开发

开发者可以构建自定义工具并通过 MCP 服务端（如使用 FastMCP）暴露出来。这使得专用内部函数或专有系统能够以标准化、易消费的格式对 LLM 和其他代理开放，无需直接修改 LLM 本身。

复杂工作流编排

通过组合各种 MCP 暴露的工具和数据源，代理可以编排高度复杂的多步骤工作流。例如，代理可以从数据库检索客户数据、生成个性化营销图片、起草定制邮件，然后通过不同 MCP 服务一键发送——全程无需人工干预。

IoT 设备控制

MCP 可以促进 LLM 与物联网（IoT）设备的交互。代理可以使用 MCP 向智能家居设备、工业传感器或机器人发送指令，实现对物理系统的自然语言控制和自动化管理。

代码实战：ADK 与 FastMCP
以下示例展示如何使用 Google ADK 连接本地 MCP 服务端，以及如何用 FastMCP 构建自定义 MCP 服务端。

示例一：ADK 代理使用 MCPToolset 连接文件系统服务端

import os from google.adk.agents import LlmAgent from google.adk.tools.mcp_tool.mcp_toolset import MCPToolset, StdioServerParameters # 创建目标文件夹的绝对路径 TARGET_FOLDER_PATH = os.path.join( os.path.dirname(os.path.abspath(__file__)), "mcp_managed_files" ) os.makedirs(TARGET_FOLDER_PATH, exist_ok=True) root_agent = LlmAgent( model='gemini-2.0-flash', name='filesystem_assistant_agent', instruction=( '帮助用户管理他们的文件。你可以列出、读取和写入文件。' f'你当前操作的目录是：{TARGET_FOLDER_PATH}' ), tools=[ MCPToolset( connection_params=StdioServerParameters( command='npx', args=[ "-y", "@modelcontextprotocol/server-filesystem", TARGET_FOLDER_PATH, ], ), # 可选：过滤暴露的工具，仅允许读取 # tool_filter=['list_directory', 'read_file'] ) ], )

示例二：用 FastMCP 创建自定义 MCP 服务端

from fastmcp import FastMCP, Client # 初始化 FastMCP 服务端 mcp_server = FastMCP() # 用装饰器注册工具，文档字符串即为 LLM 看到的工具描述 @mcp_server.tool def greet(name: str) -> str: """ 生成个性化问候语。 参数： name: 被问候者的姓名。 返回： 一条问候字符串。 """ return f"你好，{name}！很高兴认识你。" # 启动服务端，监听 localhost:8000 if __name__ == "__main__": mcp_server.run( transport="http", host="127.0.0.1", port=8000 )

示例三：ADK 代理作为 MCP 客户端消费 FastMCP 服务端

import os from google.adk.agents import LlmAgent from google.adk.tools.mcp_tool.mcp_toolset import MCPToolset, HttpServerParameters FASTMCP_SERVER_URL = "http://localhost:8000" root_agent = LlmAgent( model='gemini-2.0-flash', name='fastmcp_greeter_agent', instruction='你是一个友善的助手，能够通过"greet"工具来问候用户。', tools=[ MCPToolset( connection_params=HttpServerParameters( url=FASTMCP_SERVER_URL, ), # 仅暴露 greet 工具 tool_filter=['greet'] ) ], )
MCP 进阶注意事项
MCP 是强大的框架，但全面评估需要考虑几个关键维度。点击各层级查看详细分析：
工具 vs 资源 vs 提示词
理解三种核心组件的角色
资源（Resource）：静态数据，如 PDF 文件或数据库记录。
工具（Tool）：可执行函数，执行某个动作，如发送邮件或查询 API。
提示词（Prompt）：模板，指导 LLM 如何与资源或工具交互，确保交互结构化且高效。

三者各司其职，共同构成完整的 MCP 服务端能力矩阵。
动态能力发现
MCP 的核心优势之一
MCP 客户端可以动态查询服务端，了解其提供的工具和资源。这种"即时发现"机制对于需要适配新能力而无需重新部署的代理来说极具价值。代理可以在运行时感知新增的工具，动态扩展自身能力边界。
安全与访问控制
通过任何协议暴露工具都需要严格的安全措施
MCP 实现必须包含身份认证和授权机制，以控制哪些客户端可以访问哪些服务端，以及允许执行哪些具体操作。缺乏安全保障的 MCP 服务端可能成为严重的安全漏洞。
本地 vs 远程服务端
部署位置影响性能与安全
本地服务端：与代理运行在同一台机器上，使用 JSON-RPC over STDIO 通信，适合处理敏感数据，速度快、延迟低。

远程服务端：部署在独立服务器上，通过 Streamable HTTP 和 Server-Sent Events (SSE) 通信，支持跨组织共享工具，适合规模化场景。
错误处理策略
MCP 必须定义清晰的错误通信机制
协议必须明确定义各类错误（工具执行失败、服务端不可用、请求无效等）如何反馈给 LLM，确保 LLM 能够理解失败原因并可能尝试替代方案。完善的错误处理是 MCP 系统稳健性的重要保障。
一览概要

是什么（WHAT）

LLM 要有效地作为代理运作，必须超越简单的文本生成。它们需要与外部环境交互，访问实时数据并使用外部软件。没有标准化通信方式，每次 LLM 与外部工具的集成都是定制的、复杂的、不可复用的工程。

为什么（WHY）

MCP 提供了标准化解决方案，作为 LLM 与外部系统之间的通用接口。它建立了开放协议，定义如何发现和使用外部能力。客户端–服务端模型让服务端暴露工具、数据资源和提示词，LLM 应用作为客户端动态发现和交互。

经验法则（RULE）

在构建需要与多样化、持续演进的外部工具集交互的复杂、可扩展或企业级代理系统时使用 MCP。当不同 LLM 和工具之间的互操作性是优先级，且代理需要动态发现新能力而无需重新部署时，MCP 是理想选择。

FastMCP 的价值

FastMCP 是一个高级 Python 框架，大幅简化了 MCP 服务端的开发。它通过简单装饰器快速定义工具、资源和提示词，并自动从 Python 函数签名、类型提示和文档字符串生成 AI 模型接口规范，最大限度减少手动配置和人为错误。

核心要点
本章关于模型上下文协议的关键收获：
  • MCP 是一个开放标准，促进 LLM 与外部应用程序、数据源和工具之间的标准化通信。
  • 它采用客户端–服务端架构，定义资源、提示词和工具的暴露与消费方式。
  • MCP 与工具函数调用不同：MCP 是开放的、可互操作的协议，而工具函数调用通常是专有的、供应商特定的。
  • MCP 客户端可以动态查询服务端以发现可用能力，无需预先硬编码工具列表。
  • FastMCP 大幅简化了 MCP 服务端开发，特别适合用 Python 实现的工具。
  • Google ADK 同时支持使用现有 MCP 服务端（通过 MCPToolset）和将 ADK 工具暴露为 MCP 服务端。
  • MCP 服务端的效果依赖于底层 API 的质量——简单包装旧有 API 不足以让代理高效工作。
  • 安全性至关重要：MCP 实现必须包含身份认证和授权，防止未经授权的访问。
