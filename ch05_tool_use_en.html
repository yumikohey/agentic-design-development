Tool Use (Function Calling)

The pattern that breaks LLMs out of their training data — enabling agents to call external APIs, databases, code interpreters, and other services to act in the real world.

What is the Tool Use Pattern?

The Tool Use pattern, often implemented through a mechanism called Function Calling, enables an agent to interact with external APIs, databases, services, or even execute code. It allows the LLM at the core of the agent to decide when and how to use a specific external function based on the user's request or the current state of the task. This is what transforms a language model from a text generator into an agent capable of sensing, reasoning, and acting in the digital world.

Beyond "Function Calling"

While "function calling" aptly describes invoking predefined code functions, the broader concept of "tool calling" acknowledges that an agent's capabilities can extend far beyond simple function execution. A "tool" can be a traditional function, a complex API endpoint, a database request, or even an instruction directed at another specialized agent — enabling primary agents to delegate tasks to "analyst agents" or query external knowledge bases.

Why It's Fundamental

This pattern is fundamental because it breaks the limitations of the LLM's training data cutoff and allows it to access up-to-date information, perform calculations it can't do internally, interact with user-specific data, or trigger real-world actions. Without tools, an agent is limited to knowledge frozen at training time.

Framework Support

LangChain, LangGraph, and Google Agent Developer Kit (ADK) provide robust support for defining tools and integrating them into agent workflows, leveraging the native function calling capabilities of modern LLMs like those in the Gemini or OpenAI series. The @tool decorator, create_tool_calling_agent, and AgentExecutor patterns make tool integration straightforward.

Problems Tool Use Solves
LLMs are powerful text generators, but they are fundamentally disconnected from the outside world. Click each card to explore the specific limitations this pattern addresses:

Static Training Data

LLM knowledge is frozen at the training cutoff

LLMs cannot know about events that happened after their training cutoff date. Without tools, agents cannot answer questions about current stock prices, today's weather, recent news, or any other real-time information.

No Access to Private Data

Agents cannot query user-specific or company data

An LLM has no knowledge of a user's account balance, order history, calendar, or internal company databases. Tools enable agents to query these private systems and retrieve personalized, accurate information on demand.

Unreliable Arithmetic

LLMs cannot perform precise calculations

Language models are notorious for arithmetic errors, especially with large numbers, complex formulas, or multi-step calculations. Tools like code interpreters or calculator functions provide deterministic, exact results that LLMs cannot guarantee.

Cannot Trigger Real-World Actions

Text generation alone cannot send emails or control devices

An agent that can only generate text cannot actually send an email, create a calendar event, place an order, or control a smart home device. Tools provide the bridge between the LLM's reasoning and real-world effects.

Cannot Execute Code

LLMs can write code but cannot run it

LLMs can generate Python snippets but cannot execute them to verify correctness, process data, or produce computed outputs. Code interpreter tools provide a sandboxed execution environment, enabling agents to write, run, and analyze code results.

How Tool Use Works
The Tool Use pattern follows a structured sequence. The LLM decides when tools are needed and generates structured function call requests that the framework executes. Click each step:
1
Tool Definition
Developer describes tools to the LLM
External functions or capabilities are defined and described to the LLM. The description includes the function's purpose, its name, and the parameters it accepts (with types and descriptions). This metadata allows the LLM to understand what each tool does and when to use it.

In LangChain: @langchain_tool decorator on a Python function
In ADK: Built-in tools like google_search, or custom LlmAgent tools
2
LLM Decision
The LLM decides whether to call a tool
The LLM receives the user's request and the available tool definitions. Based on its understanding of the request and the tools, it decides if calling one or more tools is necessary to fulfill the request. It may also decide to respond directly from its training knowledge if no tool is needed.

Example: "What's the weather in London?" → LLM determines it needs the weather tool
3
Function Call Generation
LLM outputs a structured call request
If the LLM decides to use a tool, it generates a structured output (often a JSON object) that specifies the name of the tool to call and the arguments (parameters) to pass to it, extracted from the user's request. This is not executed by the LLM directly — it's a request to the orchestration layer.

Example output: {"tool": "get_weather", "args": {"location": "London"}}
4
Tool Execution & Result
Framework runs the tool, returns result to LLM
The agentic framework (AgentExecutor, ADK Runner) intercepts the structured output, identifies the requested tool, and executes the actual external function with the provided arguments. The output or result from the tool execution is then returned to the agent. The LLM uses this result as context to formulate its final response or decide on the next step.

Result flow: Tool result → LLM context → Final natural language response
Core Concepts

Tool Definition Anatomy

A well-defined tool has three critical components: (1) a name that the LLM uses to refer to it, (2) a clear description explaining what the tool does and when to use it (this is the most important part — the LLM reads this to decide when to call the tool), and (3) a parameter schema describing the input parameters with their types and descriptions.

Agents as Tools (Tool Chaining)

A powerful concept is using specialized agents as tools for a primary orchestrator agent. A "primary agent" can delegate a complex data analysis task to a dedicated "analyst agent" or query an external knowledge base through its API. This allows building sophisticated multi-agent systems where the tool-use pattern enables rich inter-agent communication.

Vertex AI Extensions vs. Function Calling

Google's Vertex AI Extensions are structured API wrappers that enable a model to connect with external APIs for real-time data processing and action execution. The key difference: Vertex AI automatically executes extensions, whereas function calls require manual execution by the user or client. Extensions offer enterprise-grade security, data privacy, and performance guarantees.

Code Examples
Tool Use implementation across LangChain (search tool agent) and CrewAI (stock price agent).

LangChain: Tool-Calling Agent with Search Tool

from langchain_google_genai import ChatGoogleGenerativeAI from langchain_core.tools import tool as langchain_tool from langchain.agents import create_tool_calling_agent, AgentExecutor from langchain_core.prompts import ChatPromptTemplate llm = ChatGoogleGenerativeAI(model="gemini-2.0-flash", temperature=0) # --- Define a Tool --- @langchain_tool def search_information(query: str) -> str: """Provides factual information on a given topic. Use this to find answers to questions like 'capital of France' or 'weather in London?'.""" simulated_results = { "weather in london": "Currently cloudy, 15°C.", "capital of france": "The capital of France is Paris.", } return simulated_results.get(query.lower(), f"No specific info found for '{query}'.") # --- Create Tool-Calling Agent --- agent_prompt = ChatPromptTemplate.from_messages([ ("system", "You are a helpful assistant."), ("human", "{input}"), ("placeholder", "{agent_scratchpad}"), ]) agent = create_tool_calling_agent(llm, [search_information], agent_prompt) agent_executor = AgentExecutor(agent=agent, tools=[search_information], verbose=True) # --- Run the agent --- result = agent_executor.invoke({"input": "What's the weather like in London?"}) print(result["output"])

CrewAI: Stock Price Lookup Tool

from crewai import Agent, Task, Crew from crewai.tools import tool # --- Define Tool: returns clean data or raises error --- @tool("Stock Price Lookup Tool") def get_stock_price(ticker: str) -> float: """Fetches the latest simulated stock price for a given ticker. Returns price as float. Raises ValueError if ticker not found.""" simulated_prices = {"AAPL": 178.15, "GOOGL": 1750.30, "MSFT": 425.50} price = simulated_prices.get(ticker.upper()) if price is not None: return price raise ValueError(f"Simulated price for '{ticker.upper()}' not found.") # --- Define Agent with the tool --- financial_analyst = Agent( role='Senior Financial Analyst', goal='Analyze stock data using provided tools and report key prices.', backstory='Experienced analyst adept at using data sources for stock info.', tools=[get_stock_price], allow_delegation=False, ) # --- Define Task and run Crew --- task = Task( description="What is the current simulated stock price for Apple (AAPL)?", expected_output="A sentence stating the simulated stock price for AAPL.", agent=financial_analyst, ) crew = Crew(agents=[financial_analyst], tasks=[task], verbose=True) result = crew.kickoff() print(result)
Deep Dive: Design Considerations
Writing Effective Tool Descriptions
The description is what the LLM reads to decide when to call the tool
The tool description is the most critical part of tool definition. The LLM reads these descriptions — not the code — to decide when and how to call a tool. Effective descriptions clearly state: what the tool does, when to use it (including example queries), what inputs it expects, and what it returns. Ambiguous or vague descriptions lead to missed tool calls or incorrect parameter extraction. Think of the description as documentation written for the LLM.
Error Handling in Tools
Tools should return clean data or raise informative errors
As shown in the CrewAI example, the best practice is for tools to return clean data (floats, dicts, strings) for success cases and raise specific Python exceptions (ValueError, TypeError) for failure cases rather than returning error strings. This enables the agent to distinguish between successful results and errors, and to decide how to react — retry, use a fallback, or report the failure to the user.
Multi-Step Tool Calling
Agents can call multiple tools in sequence to answer complex questions
Agents are not limited to a single tool call per query. The LLM can decide to call multiple tools in sequence: for example, first calling a stock price API to get the current price, then calling a calculator tool to compute profit/loss, and finally formatting the combined result as a natural language response. The AgentExecutor manages this loop, repeatedly invoking the LLM and executing tools until the LLM generates a final answer.
ADK Built-In Tools
Google ADK provides natively integrated tools like google_search and code execution
Google ADK includes a library of natively integrated tools: google_search (direct interface to Google Search), built_in_code_execution (sandboxed Python interpreter for mathematical and data processing tasks), and VSearchAgent (Vertex AI Search for enterprise data). These built-in tools integrate seamlessly with ADK agents and eliminate the need for custom implementation of common functionality.
Security Considerations
Tools that execute actions require careful authorization controls
Tools that trigger real-world actions (sending emails, executing payments, controlling devices) require careful security controls. Best practices include: validate all inputs before passing to external systems, implement rate limiting to prevent abuse, require explicit user confirmation for destructive or irreversible actions, use API keys stored in environment variables (never hardcoded), and consider sandboxing for code execution tools to prevent unauthorized system access.
At a Glance

WHAT

LLMs are powerful but disconnected from the outside world. Their knowledge is static, they can't perform actions or retrieve real-time information. Without a bridge to external systems, their utility for solving real-world problems is severely constrained.

WHY

Tool Use provides a standardized solution: describing available external functions ("tools") to the LLM so it can request them when needed. An orchestration layer executes these requests and feeds results back, giving the LLM the ability to act on up-to-date, external information.

WHEN TO USE

Whenever an agent needs to break out of the LLM's internal knowledge and interact with the outside world: real-time data (weather, stock prices), private information (user's database), precise calculations, executing code, or triggering actions (emails, device control).

KEY INSIGHT

Thinking in terms of "tool calling" rather than "function calling" better captures the full potential: tools can be APIs, databases, code interpreters, or even other specialized agents — enabling a primary agent to orchestrate across a diverse ecosystem of digital resources.

Key Takeaways
  • Tool Use (Function Calling) allows agents to interact with external systems and access dynamic information, breaking the fundamental limitation of static LLM training data.
  • The LLM decides when to use a tool and generates structured function call requests; the orchestration framework (AgentExecutor, ADK Runner) executes the actual tool and returns results.
  • Tools must be defined with clear, descriptive metadata — especially the description — as the LLM reads this to understand when and how to call each tool.
  • Thinking in terms of "tool calling" rather than "function calling" is more powerful: tools can be APIs, databases, code interpreters, or even specialized sub-agents.
  • LangChain simplifies tool definition using the @tool decorator and provides create_tool_calling_agent and AgentExecutor for building tool-using agents.
  • Google ADK includes natively integrated built-in tools: google_search for web search, built_in_code_execution for code running, and VSearchAgent for enterprise data access.
  • Tools that trigger real-world actions require careful security controls: input validation, rate limiting, API key management, and user confirmation for destructive operations.
  • Tool Use is the cornerstone pattern for building powerful, interactive, and externally aware agents that can sense, reason, and act in the real world.
