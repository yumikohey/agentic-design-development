Chapter 08

Memory Management

Effective memory management is crucial for intelligent agents — providing both short-term contextual memory within the LLM window and long-term persistent memory across sessions, enabling agents to learn, personalize, and maintain coherent conversations over time.

Pattern Overview

Agents require different types of memory — much like humans — to operate efficiently. Memory refers to an agent's ability to retain and utilize information from past interactions, observations, and learning experiences, enabling informed decisions, conversational context, and improvement over time.

Short-Term Memory (Contextual)

Similar to working memory, this holds information currently being processed or recently accessed. For LLM-based agents, short-term memory lives within the context window — recent messages, replies, tool results, and reflections. It is ephemeral: lost when the session concludes.

Long-Term Memory (Persistent)

Acts as a repository for information agents need to retain across interactions, tasks, or extended periods. Data is stored externally — in databases, knowledge graphs, or vector databases — and retrieved via semantic search when needed.

Session and State (ADK)

In Google ADK, a Session tracks an individual chat thread with its event history and temporary State data. The SessionService manages lifecycle: creation, recording activity, and termination. State acts as the session's scratchpad for dynamic, turn-specific data.

MemoryService (ADK)

The MemoryService provides a searchable long-term knowledge repository sourced from multiple past sessions or external data. It uses add_session_to_memory to ingest sessions and search_memory to retrieve relevant past knowledge for the current context.

Problems Addressed

Without effective memory management, agents are fundamentally stateless — unable to provide coherent or personalized experiences. Click to expand each problem.

Stateless Agents

Without memory, agents cannot maintain context between interactions, treating every exchange as brand new.

A stateless agent cannot remember that a user just said their name, completed step 1 of a workflow, or expressed a preference. This makes complex, multi-turn tasks impossible and creates a frustrating user experience.

Context Window Limits

The LLM context window has finite capacity — long conversations eventually overflow and lose critical earlier information.

Even with large context windows, processing every token on every call is expensive. Critical earlier information gets truncated, causing the agent to "forget" key constraints, user preferences, or prior decisions mid-conversation.

No Cross-Session Learning

Agents cannot retain knowledge from one session to apply in future sessions without persistent memory.

Every new conversation starts from scratch. The agent cannot recall that a user prefers Python over JavaScript, previously struggled with topic X, or completed onboarding last week. This prevents meaningful personalization.

No Personalization

Without user-specific persistent memory, every interaction is generic and fails to adapt to individual needs.

Generic responses frustrate users who expect the system to "know" them. Personalization requires remembering user preferences, past behaviors, and context from previous sessions — all requiring persistent memory.

Expensive Re-Processing

Without summarization or selective retrieval, agents must re-process all prior context on every turn — costly and slow.

Passing the entire conversation history on every API call increases latency and token costs dramatically. Efficient memory management uses summarization and vector search to inject only the most relevant context.

The Dual-Memory Solution

The standardized solution implements a dual-component memory system distinguishing between short-term and long-term storage.

1
Short-Term Context Window
Recent interaction data within the LLM's context
Recent messages, agent replies, tool call results, and reflections are maintained within the context window. This enables conversational coherence within a single session. Summarization techniques can compress older segments to preserve space for new information.
2
Long-Term External Storage
Persistent knowledge in vector stores or databases
Information that must persist beyond a single session is stored in external databases — typically vector stores for semantic search. On subsequent sessions, the agent queries this store to retrieve relevant past knowledge and integrates it into the current context window.
3
Session State Management
Dynamic key-value scratchpad for the active conversation
Session state stores dynamic data points relevant to the active conversation: user preferences, task progress, incremental data collection, and conditional flags. Keys can be scoped: session-specific (no prefix), user-wide (user:), app-wide (app:), or turn-only (temp:).
4
Memory Retrieval and Injection
Semantic search brings relevant past context into the present
When starting a new session, the agent queries long-term memory using semantic similarity search. The most relevant past knowledge is retrieved and injected into the current context window — bridging multiple interactions into a coherent, continuous experience.
Core Concepts

The technical building blocks of agent memory systems in Google ADK and LangChain.

ADK Session Object

A Session encapsulates a unique conversation thread: identifiers (id, app_name, user_id), a chronological event log, the session state dictionary, and a last_update_time timestamp. Managed by the SessionService — not manipulated directly.

State Key Prefixes

State keys use prefixes to define data scope and persistence. Keys without prefixes are session-specific. user: associates data with a user across all sessions. app: designates data shared among all users. temp: is valid only for the current processing turn and is never persisted.

Long-Term Memory Types (LangChain)

Three types analogous to human memory: Semantic (remembering facts — user profiles, domain knowledge); Episodic (remembering experiences — past task sequences, few-shot examples); Procedural (remembering rules — agent instructions, system prompts that can self-update via reflection).

Practical Use Cases

Memory is essential for agents that must do more than answer a single question.

Chatbots and Conversational AI

Conversational flow relies on short-term memory to track the ongoing dialogue. Long-term memory enables chatbots to recall user preferences, past issues, and prior discussions — offering personalized, continuous interactions across sessions without the user having to repeat themselves.

1
Short-term memory maintains the current conversation thread
2
At session start, relevant long-term memories are retrieved
3
Agent personalizes responses based on recalled preferences
4
Session is saved to long-term memory on conclusion

Task-Oriented Agents

Agents managing multi-step tasks need short-term memory to track previous steps, current progress, and overall goals — maintained in session state. Long-term memory is crucial for accessing specific user-related data not in the immediate context: past configurations, previous decisions, domain expertise built up over time.

1
Session state tracks task progress and intermediate results
2
Long-term memory provides historical context and user preferences
3
Task completion updates both state and long-term knowledge

Personalized Experiences

Agents offering tailored interactions utilize long-term memory to store and retrieve user preferences, past behaviors, and personal information. This allows agents to adapt responses and suggestions to each individual — transforming generic assistants into truly personal ones.

1
User profile stored in long-term memory with user: prefix keys
2
Retrieved at session start and injected into context
3
Preferences updated incrementally as user behavior evolves

Retrieval-Augmented Generation (RAG)

Agents designed for knowledge-intensive question answering access a knowledge base as their long-term memory. The agent retrieves relevant documents or data using semantic similarity search and integrates retrieved context to produce accurate, grounded responses.

1
Query is embedded and used for semantic similarity search
2
Top-K relevant documents retrieved from vector store
3
Retrieved context injected into prompt for grounded generation

Learning and Self-Improvement

Agents can refine performance by storing successful strategies, mistakes, and new information in long-term memory. Reinforcement learning agents store learned strategies this way. The Reflection technique prompts agents to update their own procedural memory (system prompt) based on past performance.

1
Outcome of each interaction is evaluated
2
Successful and failed strategies stored in long-term memory
3
Agent reflects and updates its own instructions based on history

Autonomous Systems

Robots and self-driving vehicles require memory for maps, routes, object locations, and learned behaviors. Short-term memory handles the immediate surroundings and current navigation state; long-term memory provides general environmental knowledge and learned safe behaviors.

1
Sensor data populates short-term situational awareness
2
Long-term memory provides maps and learned environment models
3
New experiences update the persistent environmental knowledge base
Code Examples

Practical memory implementations using Google ADK and LangChain.

ADK: Session State via output_key

from google.adk.agents import LlmAgent from google.adk.sessions import InMemorySessionService, Session from google.adk.runners import Runner from google.genai.types import Content, Part # Agent saves its final text response to session state automatically greeting_agent = LlmAgent( name="Greeter", model="gemini-2.0-flash", instruction="Generate a short, friendly greeting.", output_key="last_greeting" # saved to session.state ) app_name, user_id, session_id = "state_app", "user1", "session1" session_service = InMemorySessionService() runner = Runner( agent=greeting_agent, app_name=app_name, session_service=session_service ) session = session_service.create_session( app_name=app_name, user_id=user_id, session_id=session_id ) # Run the agent user_message = Content(parts=[Part(text="Hello")]) for event in runner.run( user_id=user_id, session_id=session_id, new_message=user_message ): if event.is_final_response(): print("Agent responded.") # Check state after run — state correctly updated updated_session = session_service.get_session(app_name, user_id, session_id) print(f"State after run: {updated_session.state}")

LangChain: ConversationBufferMemory with LLMChain

from langchain_openai import OpenAI from langchain.chains import LLMChain from langchain.prompts import PromptTemplate from langchain.memory import ConversationBufferMemory # 1. Define LLM and prompt with {history} placeholder llm = OpenAI(temperature=0) template = """You are a helpful travel agent. Previous conversation: {history} New question: {question} Response:""" prompt = PromptTemplate.from_template(template) # 2. Configure memory — memory_key matches {history} in prompt memory = ConversationBufferMemory(memory_key="history") # 3. Build the chain with memory conversation = LLMChain(llm=llm, prompt=prompt, memory=memory) # 4. Multi-turn conversation — history accumulates automatically r1 = conversation.predict(question="I want to book a flight.") r2 = conversation.predict(question="My name is Sam, by the way.") r3 = conversation.predict(question="What was my name again?") print(r3) # Agent recalls "Sam" from memory

Memory Architecture Deep Dive

From ephemeral context to persistent knowledge — the layers of agent memory. Click each layer to explore.

Context Window (Working Memory)
Ephemeral — lost when session ends
The LLM's context window is the agent's working memory. It holds the current conversation, tool outputs, and agent reflections. With large context models, this can be substantial — but it is always ephemeral. Processing it repeatedly is expensive, making summarization strategies essential for long conversations.
Session State (Scratchpad)
Turn-scoped key-value store within a session
Session state is a dictionary of key-value pairs that tracks dynamic data relevant to the active conversation: task progress, conditional flags, intermediate results. It persists across turns within a session but is scoped to that session. Key prefixes (user:, app:, temp:) extend the scope. Always update via append_event — not direct mutation.
Semantic Memory (Facts)
User profiles, preferences, domain knowledge
Semantic memory retains specific facts and concepts — user preferences, domain knowledge, entity information. Stored as continuously updated user profiles (JSON documents) or collections of factual documents in a vector store. Retrieved by semantic similarity to inject relevant facts into the context window.
Episodic Memory (Experiences)
Past interaction sequences and task trajectories
Episodic memory recalls past events or actions — sequences of interactions that accomplished a task, or failed attempts to avoid. In practice, implemented through few-shot example prompting where an agent learns from past successful interaction sequences. The LangGraph BaseStore enables cross-session episodic memory storage.
Procedural Memory (Rules & Instructions)
System prompt, behavioral rules — can self-update
Procedural memory is the agent's core instructions and behavioral rules — often contained in the system prompt. Agents can modify their own prompts to adapt and improve. The Reflection technique prompts an agent with its current instructions and recent interactions, then asks it to generate refined instructions that are stored back to the BaseStore.
At a Glance

WHAT

Without a memory mechanism, agents are stateless — unable to maintain conversational context, learn from experience, or personalize responses. This limits them to simple, one-shot interactions.

WHY

A dual-component memory system distinguishes short-term (context window) from long-term (external vector stores). ADK provides Session + State for short-term and MemoryService for long-term. Together they enable coherent, personalized, and learning agents.

WHEN TO USE

Use whenever an agent must do more than answer a single question: multi-turn conversations, multi-step tasks, personalization, learning from feedback, or any scenario where context must persist across turns or sessions.

RULE OF THUMB

Implement memory management when the agent is expected to learn or adapt based on past successes, failures, or newly acquired information — or when user satisfaction depends on remembering who they are.

Key Takeaways
  • Memory enables agents to maintain history, learn from experience, personalize interactions, and manage complex time-dependent tasks.
  • Short-term memory (the context window) is ephemeral, limited by LLM capacity, and lost when the session ends — but is immediately accessible.
  • Long-term memory persists across sessions using external databases, especially vector stores accessed via semantic search.
  • In Google ADK, the three core memory abstractions are Session (chat thread), State (session scratchpad), and Memory (searchable cross-session repository).
  • State key prefixes define scope: no prefix = session, user: = user-wide, app: = application-wide, temp: = current turn only.
  • Direct mutation of session.state is discouraged — always update via the append_event mechanism to ensure proper persistence and event tracking.
  • Long-term memory has three human-analogous types: Semantic (facts), Episodic (experiences), and Procedural (rules/instructions that can self-update).
  • The Vertex AI Memory Bank and VertexAiRagMemoryService provide production-grade, scalable persistent memory with semantic search capabilities.
