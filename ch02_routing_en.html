Routing

Introduce conditional logic into your agent's workflow — dynamically evaluate criteria to direct inputs to the most appropriate specialized handler, tool, or sub-process.

What is the Routing Pattern?

While sequential processing via prompt chaining is foundational for deterministic, linear workflows, its applicability is limited in scenarios requiring adaptive responses. Real-world agentic systems must often arbitrate between multiple potential actions based on contingent factors — the state of the environment, user input, or the outcome of a preceding operation. This capacity for dynamic decision-making is achieved through routing.

Conditional Logic in Agents

Routing introduces conditional logic into an agent's operational framework, enabling a shift from a fixed execution path to a model where the agent dynamically evaluates specific criteria to select from a set of possible subsequent actions. This allows for more flexible and context-aware system behavior — transforming a static executor of pre-defined sequences into a dynamic system.

A Customer Service Example

An agent designed for customer inquiries, when equipped with a routing function, can first classify an incoming query to determine the user's intent. Based on this classification, it can direct the query to: a specialized agent for direct question-answering, a database retrieval tool for account information, or an escalation procedure for complex issues — rather than defaulting to a single, predetermined response pathway.

Framework Support

LangChain, LangGraph, and Google's Agent Development Kit (ADK) provide explicit constructs for defining and managing routing logic. LangGraph's state-based graph architecture is particularly well-suited for complex routing where decisions are contingent on the accumulated state of the entire system. Google's ADK provides foundational components for structuring an agent's capabilities and interaction models.

Problems That Routing Solves
Without routing, agents are limited to fixed, linear execution paths. Click each card to see the specific problem addressed:

One-Size-Fits-All Processing

Every request processed the same way regardless of type

Without routing, all inputs are funneled through the same pipeline regardless of their nature. A "check order status" request and a "technical support" request receive identical treatment, producing suboptimal results for both.

Inability to Specialize

No way to delegate to specialist handlers

Complex systems benefit from specialist sub-agents: a booking agent for travel, an info agent for FAQs, a technical agent for debugging. Without routing, a single generalist agent must handle all domains — degrading quality across the board.

Rigid Sequential Execution

Cannot adapt flow based on runtime conditions

Prompt chains are linear. If the outcome of one step should change which next step runs (e.g., if sentiment is negative, escalate; otherwise, resolve automatically), pure chaining cannot accommodate this without routing logic.

Tool Selection Failure

Calls the wrong tool for the job

A coding assistant that uses a web search tool for a "debug this code" request and a code analysis tool for "what's the capital of France?" is wasting resources and producing poor results. Routing maps intent to the correct tool.

Escalation Blindness

No mechanism to identify and escalate complex cases

Some requests require human intervention or a more capable model. Without routing, there's no mechanism to detect "this is beyond my capability" and redirect to a human operator or escalation pathway, leading to poor resolutions.

The Routing Architecture
The core of routing is a mechanism that evaluates the input and directs the flow. Click each node to explore the architecture:
1
Input Reception
Receive and normalize the incoming request
The system receives the raw input — a user query, an API payload, a document, or a state change event. This input is the trigger for the routing decision. The input may be preprocessed or normalized before being analyzed.

Input: Raw user request, document, or event
Output: Normalized input ready for classification
2
Router / Classifier
Evaluate criteria and determine the destination
The router is the decision-making core. It analyzes the input and outputs a routing decision. This can be implemented as:

• LLM-based: Prompt the LLM to classify and output a category label
• Embedding-based: Convert input to a vector and compare to route embeddings
• Rule-based: If-else logic on keywords or structured data
• ML Classifier: A fine-tuned discriminative model

Input: Normalized request
Output: Routing decision (e.g., "booker", "info", "escalate")
3
Dispatch to Handler
Direct the request to the appropriate specialized sub-process
Based on the routing decision, the system dispatches the original input to the appropriate handler. Handlers can be: specialized sub-agents, tool chains, database retrieval pipelines, or escalation procedures. Each handler is optimized for its specific domain.

Input: Routing decision + original request
Output: Response from the specialized handler
4
Response Consolidation
Return the handler's result to the user
The handler's output is returned to the user or to the next stage in a larger workflow. In some architectures, a post-processing step standardizes the format of responses from different handlers before delivery.

Input: Handler output
Output: Final response delivered to user or downstream process
Routing Mechanisms
The core component of the Routing pattern is the mechanism that performs evaluation and directs the flow. There are four primary approaches, each with different trade-offs.

LLM-Based Routing

The language model itself is prompted to analyze the input and output a specific identifier or instruction indicating the next step or destination. For example, a prompt might ask the LLM to "Analyze the following user query and output only the category: 'Order Status', 'Product Info', 'Technical Support', or 'Other'." The agentic system reads this output and directs the workflow accordingly.

Embedding-Based Routing

The input query is converted into a vector embedding. This embedding is then compared to embeddings representing different routes or capabilities. The query is routed to the destination whose embedding is most similar. This is useful for semantic routing, where the decision is based on the meaning of the input rather than just keywords.

Rule-Based Routing

Predefined rules or logic (e.g., if-else statements, switch cases) based on keywords, patterns, or structured data extracted from the input. This can be faster and more deterministic than LLM-based routing, but is less flexible for handling nuanced or novel inputs. Best used when categories are well-defined and stable.

ML Model-Based Routing

Employs a discriminative model (such as a classifier) that has been specifically trained on a small corpus of labeled data to perform the routing task. Unlike LLM-based routing, the routing logic is encoded within the fine-tuned model's learned weights — the decision-making component is not a generative model executing a prompt at inference time. This provides high speed and consistency for high-volume routing scenarios.

LangChain's RunnableBranch is a key primitive for implementing routing:

delegation_branch = RunnableBranch( (lambda x: x['decision'].strip() == 'booker', branches["booker"]), (lambda x: x['decision'].strip() == 'info', branches["info"]), branches["unclear"] # default fallback branch )

Hands-On Code
Two implementations are shown: one with LangChain's RunnableBranch, and one with Google's ADK demonstrating coordinator-based routing.

LangChain: RunnableBranch Router

This code constructs a "coordinator" that routes user requests to different simulated sub-agent handlers based on the request's intent (booking, information, or unclear). An LLM classifies the intent and a RunnableBranch dispatches accordingly.

pip install langchain langgraph google-cloud-aiplatform langchain-google-genai google-adk pydantic
from langchain_google_genai import ChatGoogleGenerativeAI from langchain_core.prompts import ChatPromptTemplate from langchain_core.output_parsers import StrOutputParser from langchain_core.runnables import RunnablePassthrough, RunnableBranch # --- Initialize LLM --- llm = ChatGoogleGenerativeAI(model="gemini-2.5-flash", temperature=0) # --- Specialist Handler Functions --- def booking_handler(request: str) -> str: return f"Booking Agent handled: '{request}'. Simulated booking action." def info_handler(request: str) -> str: return f"Info Agent handled: '{request}'. Simulated info retrieval." def unclear_handler(request: str) -> str: return f"Could not delegate: '{request}'. Please clarify." # --- Coordinator Router Chain --- coordinator_router_prompt = ChatPromptTemplate.from_messages([ ("system", """Analyze the user's request and determine which specialist handler should process it. - If related to booking flights or hotels, output 'booker'. - For all other general information questions, output 'info'. - If the request is unclear, output 'unclear'. ONLY output one word: 'booker', 'info', or 'unclear'."""), ("user", "{request}") ]) coordinator_router_chain = coordinator_router_prompt | llm | StrOutputParser() # --- Define Branches (RunnableBranch) --- branches = { "booker": RunnablePassthrough.assign( output=lambda x: booking_handler(x['request']['request'])), "info": RunnablePassthrough.assign( output=lambda x: info_handler(x['request']['request'])), "unclear": RunnablePassthrough.assign( output=lambda x: unclear_handler(x['request']['request'])), } delegation_branch = RunnableBranch( (lambda x: x['decision'].strip() == 'booker', branches["booker"]), (lambda x: x['decision'].strip() == 'info', branches["info"]), branches["unclear"] # Default fallback ) # --- Combine into coordinator agent --- coordinator_agent = { "decision": coordinator_router_chain, "request": RunnablePassthrough() } | delegation_branch | (lambda x: x['output']) # --- Run --- result = coordinator_agent.invoke({"request": "Book me a flight to London."}) print(result)

Google ADK: Coordinator with Sub-Agents

In the ADK paradigm, routing is implemented by defining a discrete set of "tools" or sub-agents that represent the agent's functions. A coordinator agent holds sub-agents and uses its LLM to match user intent to the correct specialist via Auto-Flow delegation.

from google.adk.agents import Agent from google.adk.tools import FunctionTool # --- Tool functions for specialist agents --- def booking_handler(request: str) -> str: """Handles booking requests for flights and hotels.""" return f"Booking action for '{request}' has been simulated." def info_handler(request: str) -> str: """Handles general information requests.""" return f"Information request for '{request}'. Simulated retrieval." # --- Create ADK Agents --- booking_agent = Agent( name="Booker", model="gemini-2.0-flash", description="A specialist agent for flight and hotel booking requests.", tools=[FunctionTool(booking_handler)] ) info_agent = Agent( name="Info", model="gemini-2.0-flash", description="A specialist agent for general information requests.", tools=[FunctionTool(info_handler)] ) # --- Coordinator with explicit routing instructions --- coordinator = Agent( name="Coordinator", model="gemini-2.0-flash", instruction="""You are the main coordinator. Analyze incoming user requests and delegate them to the appropriate specialist agent. Do NOT answer the user directly. - For booking flights or hotels, delegate to the 'Booker' agent. - For all other information questions, delegate to the 'Info' agent.""", description="A coordinator that routes user requests to specialist agents.", sub_agents=[booking_agent, info_agent] # Enables LLM-driven Auto-Flow )
Deep Dive: Routing Mechanisms Compared
Each routing mechanism has distinct characteristics that make it suitable for different scenarios. Click each layer to explore the trade-offs:
LLM-Based Routing
Flexible, semantic understanding of intent
Prompts the LLM to output a routing decision. High flexibility — handles novel, nuanced inputs well and understands semantic meaning. The downside: adds latency (one LLM call per routing decision), can be non-deterministic, and costs tokens. Best for: open-ended queries with many possible intents, when nuance matters more than speed.
Embedding-Based Routing
Semantic similarity matching via vector space
Converts the input query to a vector embedding and compares it to pre-computed embeddings for each route. Routes to the most semantically similar destination. Faster than LLM routing (no generation step), robust to paraphrasing and synonyms. Requires maintaining an embedding index. Best for: semantic routing with a defined set of categories, when speed and semantic flexibility are both important.
Rule-Based Routing
Deterministic, fast, fully predictable
Uses predefined if-else logic, keyword matching, or pattern recognition. Completely deterministic and very fast — no model inference required. Fully auditable and debuggable. The limitation: brittle for novel inputs, requires manual maintenance as categories evolve, and fails on paraphrasing. Best for: well-defined, stable categories where speed and predictability are paramount.
ML Model-Based Routing
Trained discriminative classifier, optimized for routing
A small, fine-tuned classifier trained specifically for the routing task. The routing logic is encoded in the model's weights — not executed as a generative prompt. Very fast (far faster than LLM routing), highly consistent, and optimized for the specific routing task. LLMs may be used offline to generate synthetic training data. Best for: high-volume routing pipelines where latency and consistency are critical.
LangGraph State-Based Routing
Graph topology with conditional state transitions
LangGraph's graph architecture defines nodes and edges where transitions between nodes are governed by conditional functions or model-based evaluations. Routing decisions can be contingent on the entire accumulated state of the system — not just the immediate input. This is ideal for complex multi-step workflows where routing depends on what has already happened in the conversation or workflow.
At a Glance

THE PROBLEM

Sequential prompt chains have fixed execution paths. Real-world systems need to respond differently to different inputs — routing different request types to different specialized handlers, tools, or escalation paths.

THE SOLUTION

Routing introduces conditional logic: a router evaluates each input and directs it to the most appropriate specialized handler. This transforms agents from static executors into dynamic, context-aware systems.

THE MECHANISMS

Four approaches: LLM-based (flexible, semantic), Embedding-based (semantic similarity), Rule-based (deterministic, fast), and ML Classifier (trained, optimized). Choice depends on trade-offs between speed, flexibility, and consistency.

WHEN TO USE

Use when: different input types need different processing paths; you have specialized sub-agents or tools; you need to escalate complex cases; or the system must adapt its behavior based on input classification or system state.

Key Takeaways
  • Routing introduces conditional logic into an agent's operational framework, enabling dynamic selection of execution paths rather than fixed sequential processing.
  • The core component of any routing implementation is a mechanism that evaluates input criteria and directs the flow to the appropriate handler, sub-agent, or tool.
  • There are four primary routing mechanisms: LLM-based (flexible, semantic), Embedding-based (similarity-driven), Rule-based (deterministic, fast), and ML Classifier-based (trained, optimized for high volume).
  • Routing can be applied at multiple points in an agent's lifecycle: at the outset to classify a primary task, at intermediate points within a processing chain, or during a subroutine to select the most appropriate tool.
  • LangChain's RunnableBranch is the key primitive for implementing routing in LCEL-based workflows. LangGraph's graph structure enables routing based on accumulated system state.
  • Google ADK implements routing through coordinator agents with sub-agents — the presence of sub-agents enables LLM-driven Auto-Flow delegation.
  • Routing transforms agents from static executors of pre-defined sequences into dynamic systems capable of making intelligent decisions about the most effective method for accomplishing a task under changing conditions.
  • Routing is often combined with prompt chaining: the router decides which chain to execute, and prompt chaining handles the sequential steps within each branch.

Summary

Routing provides the capacity for logical arbitration that is essential for creating functionally diverse and context-aware systems. It facilitates the development of more adaptive execution flows that can respond dynamically and appropriately to a wider range of inputs and state changes — the critical capability that separates simple chatbots from truly intelligent agentic systems.
