工具使用（函数调用）

突破大语言模型训练数据限制的关键模式——让智能体能够调用外部 API、数据库、代码解释器等服务，在真实世界中采取行动。

什么是工具使用模式？

工具使用模式（Tool Use Pattern），通常通过函数调用（Function Calling）机制实现，它使智能体能够与外部 API、数据库、服务交互，甚至执行代码。它让智能体核心的大语言模型能够根据用户请求或当前任务状态，决定何时以及如何使用特定的外部函数。这正是将语言模型从文本生成器转变为能够在数字世界中感知、推理和行动的智能体的关键所在。

超越"函数调用"的概念

虽然"函数调用"准确描述了调用预定义代码函数的过程，但更宽泛的"工具调用"概念承认，智能体的能力可以远超简单的函数执行。"工具"可以是传统函数、复杂的 API 端点、数据库请求，甚至是向另一个专门智能体发出的指令——使主智能体能够将任务委托给"分析智能体"或通过 API 查询外部知识库。

为何这是基础性模式

这一模式之所以是基础性的，是因为它突破了大语言模型训练数据截止日期的限制，使其能够访问最新信息、执行内部无法完成的计算、与用户特定数据交互，或触发真实世界的行动。没有工具，智能体只能局限于训练时冻结的知识。

框架支持

LangChain、LangGraph 和 Google Agent Developer Kit（ADK）为定义工具并将其集成到智能体工作流中提供了强大支持，充分利用了 Gemini 或 OpenAI 系列等现代大语言模型的原生函数调用能力。@tool 装饰器、create_tool_calling_agent 和 AgentExecutor 模式使工具集成变得简单直接。

工具使用模式解决的问题
大语言模型是强大的文本生成器，但从根本上与外部世界隔绝。点击卡片探索这一模式所应对的具体局限性：

静态训练数据

大语言模型的知识在训练截止时被冻结

大语言模型无法了解训练截止日期之后发生的事情。没有工具，智能体无法回答关于当前股票价格、今日天气、近期新闻或任何其他实时信息的问题。

无法访问私有数据

智能体无法查询用户特定或公司数据

大语言模型对用户的账户余额、订单历史、日历或公司内部数据库一无所知。工具使智能体能够查询这些私有系统，按需检索个性化、准确的信息。

算术不可靠

大语言模型无法进行精确计算

语言模型在算术上出错是众所周知的，尤其是大数字、复杂公式或多步骤计算。代码解释器或计算器函数等工具提供了大语言模型无法保证的确定性、精确结果。

无法触发真实世界行动

仅靠文本生成无法发送邮件或控制设备

只能生成文本的智能体无法真正发送电子邮件、创建日历事件、下单或控制智能家居设备。工具在大语言模型的推理与真实世界效果之间提供了桥梁。

无法执行代码

大语言模型能写代码但无法运行

大语言模型可以生成 Python 代码片段，但无法执行它们来验证正确性、处理数据或产生计算输出。代码解释器工具提供沙盒执行环境，使智能体能够编写、运行并分析代码结果。

工具使用的工作原理
工具使用模式遵循结构化的顺序。大语言模型决定何时需要工具并生成结构化的函数调用请求，框架负责执行。点击每个步骤：
1
工具定义
开发者向大语言模型描述工具
外部函数或能力被定义并描述给大语言模型。描述包括函数的用途、名称以及它接受的参数（含类型和描述）。这些元数据使大语言模型能够理解每个工具的功能以及何时使用它。

在 LangChain 中：在 Python 函数上使用 @langchain_tool 装饰器
在 ADK 中：内置工具如 google_search，或自定义 LlmAgent 工具
2
大语言模型决策
大语言模型决定是否调用工具
大语言模型接收用户请求和可用工具定义。根据对请求和工具的理解，它决定是否需要调用一个或多个工具来满足请求。如果不需要工具，它也可以直接从训练知识中回答。

示例："伦敦今天天气怎么样？"→ 大语言模型判断需要天气工具
3
函数调用生成
大语言模型输出结构化调用请求
如果大语言模型决定使用工具，它会生成一个结构化输出（通常是 JSON 对象），指定要调用的工具名称以及从用户请求中提取的参数。这不是由大语言模型直接执行的——它是对编排层的请求。

输出示例：{"tool": "get_weather", "args": {"location": "伦敦"}}
4
工具执行与结果返回
框架运行工具，将结果返回给大语言模型
智能体框架（AgentExecutor、ADK Runner）拦截结构化输出，识别请求的工具，并使用提供的参数执行实际的外部函数。工具执行的输出或结果随后被返回给智能体。大语言模型使用该结果作为上下文，制定最终响应或决定下一步行动。

结果流：工具结果 → 大语言模型上下文 → 最终自然语言响应
核心概念

工具定义的剖析

一个定义良好的工具有三个关键组成部分：(1) 大语言模型用来引用它的名称，(2) 清晰的描述，解释工具的功能以及何时使用它（这是最重要的部分——大语言模型读取此内容来决定何时调用工具），以及 (3) 描述输入参数及其类型和描述的参数模式。

将智能体作为工具（工具链）

一个强大的概念是将专门智能体用作主编排者智能体的工具。"主智能体"可以将复杂的数据分析任务委托给专门的"分析智能体"，或通过 API 查询外部知识库。这使得构建复杂的多智能体系统成为可能，其中工具使用模式实现了丰富的智能体间通信。

Vertex AI 扩展 vs. 函数调用

Google 的 Vertex AI 扩展是结构化 API 包装器，使模型能够与外部 API 连接以进行实时数据处理和行动执行。关键区别在于：Vertex AI 自动执行扩展，而函数调用需要用户或客户端手动执行。扩展提供企业级安全性、数据隐私和性能保证。

实际应用场景
在几乎任何智能体需要超越文本生成来执行操作或检索特定动态信息的场景中，工具使用模式都适用。选择一个应用场景：

从外部来源检索信息

访问大语言模型训练数据中不存在的实时数据或信息。

1
应用场景：天气查询智能体。
2
工具：接受位置参数并返回当前天气状况的天气 API。
3
智能体流程：用户询问"伦敦天气怎么样？"→ 大语言模型识别需要天气工具 → 以"伦敦"调用工具 → 工具返回数据 → 大语言模型格式化为友好响应。

与数据库和 API 交互

对结构化数据执行查询、更新或其他操作。

1
应用场景：电商智能体。
2
工具：检查产品库存、获取订单状态或处理付款的 API 调用。
3
智能体流程：用户询问"X 产品有货吗？"→ 大语言模型调用库存 API → 工具返回库存数量 → 大语言模型告知用户库存状态。

执行计算和数据分析

使用外部计算器、数据分析库或统计工具获得精确结果。

1
应用场景：金融分析智能体。
2
工具：计算器函数、股票市场数据 API、电子表格工具。
3
智能体流程："苹果股价是多少？如果我以150美元买了100股，利润是多少？"→ 调用股票 API → 调用计算器 → 格式化回复。

发送通信

发送电子邮件、消息，或调用外部通信服务 API。

1

2
工具：邮件发送 API。

智能体流程："给张三发邮件告知明天的会议"→ 大语言模型提取收件人、主题、正文 → 调用邮件工具 → 确认发送。

执行代码

智能体流程：用户提供代码问"这段代码输出什么？"→ 大语言模型使用代码解释器运行它 → 分析输出 → 提供解释。

与智能家居设备、IoT 平台或其他互联系统交互。

1
应用场景：智能家居控制智能体。

工具：控制智能灯、温控器或家电的 API。
3
智能体流程："关掉客厅的灯"→ 大语言模型调用智能家居工具，附带命令和目标设备 → 确认操作完成。

代码实战
工具使用在 LangChain（搜索工具智能体）和 CrewAI（股票价格智能体）中的实现示例。

LangChain：带搜索工具的函数调用智能体

from langchain_google_genai import ChatGoogleGenerativeAI from langchain_core.tools import tool as langchain_tool from langchain.agents import create_tool_calling_agent, AgentExecutor from langchain_core.prompts import ChatPromptTemplate llm = ChatGoogleGenerativeAI(model="gemini-2.0-flash", temperature=0) # --- 定义工具 --- @langchain_tool def search_information(query: str) -> str: """提供关于特定主题的事实信息。 使用此工具查找以下类型问题的答案： '法国的首都' 或 '伦敦的天气？'。""" simulated_results = { "伦敦的天气": "目前多云，15°C。", "法国的首都": "法国的首都是巴黎。", } return simulated_results.get(query.lower(), f"未找到关于'{query}'的具体信息。") # --- 创建工具调用智能体 --- agent_prompt = ChatPromptTemplate.from_messages([ ("system", "你是一个有帮助的助手。"), ("human", "{input}"), ("placeholder", "{agent_scratchpad}"), ]) agent = create_tool_calling_agent(llm, [search_information], agent_prompt) agent_executor = AgentExecutor(agent=agent, tools=[search_information], verbose=True) # --- 运行智能体 --- result = agent_executor.invoke({"input": "伦敦天气怎么样？"}) print(result["output"])

CrewAI：股票价格查询工具

from crewai import Agent, Task, Crew from crewai.tools import tool # --- 定义工具：返回干净数据或抛出错误 --- @tool("股票价格查询工具") def get_stock_price(ticker: str) -> float: """获取给定股票代码的最新模拟股票价格。 以浮点数形式返回价格。如果未找到代码则抛出 ValueError。""" simulated_prices = {"AAPL": 178.15, "GOOGL": 1750.30, "MSFT": 425.50} price = simulated_prices.get(ticker.upper()) if price is not None: return price raise ValueError(f"未找到'{ticker.upper()}'的模拟价格。") # --- 定义配备工具的智能体 --- financial_analyst = Agent( role='高级金融分析师', goal='使用提供的工具分析股票数据并报告关键价格。', backstory='擅长使用数据源获取股票信息的经验丰富的分析师。', tools=[get_stock_price], allow_delegation=False, ) # --- 定义任务并运行团队 --- task = Task( description="苹果公司（AAPL）目前的模拟股票价格是多少？", expected_output="一句话说明 AAPL 的模拟股票价格。", agent=financial_analyst, ) crew = Crew(agents=[financial_analyst], tasks=[task], verbose=True) result = crew.kickoff() print(result)
深入解析：设计考量
编写有效的工具描述
描述是大语言模型决定何时调用工具所读取的内容
工具描述是工具定义中最关键的部分。大语言模型读取的是这些描述——而非代码——来决定何时以及如何调用工具。有效的描述应清晰说明：工具的功能、何时使用它（包括示例查询）、期望的输入以及返回的内容。模糊或含混的描述会导致遗漏工具调用或错误的参数提取。将描述视为专门为大语言模型编写的文档。
工具中的错误处理
工具应返回干净数据或抛出有信息量的错误
如 CrewAI 示例所示，最佳实践是工具在成功时返回干净数据（浮点数、字典、字符串），在失败时抛出特定 Python 异常（ValueError、TypeError），而不是返回错误字符串。这使智能体能够区分成功结果和错误，并决定如何反应——重试、使用备用方案，或向用户报告失败。
多步工具调用
智能体可以按顺序调用多个工具来回答复杂问题
智能体不限于每次查询只进行一次工具调用。大语言模型可以决定按顺序调用多个工具：例如，先调用股票价格 API 获取当前价格，再调用计算器工具计算盈亏，最后将组合结果格式化为自然语言响应。AgentExecutor 管理此循环，反复调用大语言模型并执行工具，直到大语言模型生成最终答案。
ADK 内置工具
Google ADK 提供原生集成的工具如 google_search 和代码执行
Google ADK 包含原生集成工具库：google_search（直接接口到 Google 搜索引擎）、built_in_code_execution（用于数学和数据处理任务的沙盒 Python 解释器）以及 VSearchAgent（用于企业数据的 Vertex AI 搜索）。这些内置工具与 ADK 智能体无缝集成，无需为常见功能自定义实现。
安全考量
触发真实世界行动的工具需要谨慎的授权控制
触发真实世界行动的工具（发送电子邮件、执行付款、控制设备）需要严格的安全控制。最佳实践包括：在传递给外部系统之前验证所有输入，实施速率限制以防止滥用，对破坏性或不可逆的操作要求明确的用户确认，使用存储在环境变量中的 API 密钥（绝不硬编码），以及对代码执行工具进行沙盒处理以防止未经授权的系统访问。
一览概要

问题所在

大语言模型功能强大，但从根本上与外部世界隔绝。它们的知识是静态的，无法执行行动或检索实时信息。没有与外部系统的桥梁，它们解决现实问题的能力受到严重限制。

解决方案

工具使用提供了标准化解决方案：向大语言模型描述可用的外部函数（"工具"），使其能够在需要时请求调用。编排层执行这些请求并将结果反馈回来，赋予大语言模型基于最新外部信息采取行动的能力。

使用时机

每当智能体需要突破大语言模型内部知识并与外部世界交互时：实时数据（天气、股票价格）、私有信息（用户数据库）、精确计算、执行代码，或触发行动（发送邮件、控制设备）。

核心洞见

以"工具调用"而非"函数调用"的思维来看待这一模式，更能体现其全部潜力：工具可以是 API、数据库、代码解释器，甚至是其他专门智能体——使主智能体能够在多元化的数字资源生态系统中进行编排。

核心要点
  • 工具使用（函数调用）允许智能体与外部系统交互并访问动态信息，突破了静态大语言模型训练数据的根本限制。
  • 大语言模型决定何时使用工具并生成结构化函数调用请求；编排框架（AgentExecutor、ADK Runner）执行实际工具并返回结果。
  • 工具必须使用清晰、描述性的元数据定义——尤其是描述部分——因为大语言模型读取此内容来了解何时以及如何调用每个工具。
  • 以"工具调用"而非"函数调用"的思维来看待这一模式更为强大：工具可以是 API、数据库、代码解释器，甚至是专门的子智能体。
  • LangChain 使用 @tool 装饰器简化工具定义，并提供 create_tool_calling_agent 和 AgentExecutor 来构建使用工具的智能体。
  • Google ADK 包含原生集成的内置工具：google_search 用于网络搜索，built_in_code_execution 用于运行代码，VSearchAgent 用于企业数据访问。
  • 触发真实世界行动的工具需要严格的安全控制：输入验证、速率限制、API 密钥管理，以及对破坏性操作要求用户确认。
  • 工具使用是构建强大、交互式、能感知外部世界的智能体——使其能够在真实世界中感知、推理和行动——的基石模式。

为大模型装上与世界互动的双手：深入理解"工具使用"设计模式

引言（具体痛点场景）

想象一下，你拥有一个绝顶聪明的朋友，他阅读了人类历史上所有公开的文本，精通量子物理、莎士比亚和高级编程。但是，他被锁在一个完全封闭的密室里，没有网络连接，也没有任何传感器。如果你问他"今天的伦敦天气如何？"或者"帮我把这封邮件发给老板"，他只能无奈地告诉你："根据我被锁进密室前（训练数据截止日期）的知识，伦敦通常多雨，但我无法告诉你现在的确切天气，我也无法连接你的邮箱服务器。"

这就是今天大多数基础大语言模型（LLM）面临的真实困境。它们是强大的文本生成引擎，但在默认状态下，它们与外部世界是完全断开的，知识是静态的，且缺乏执行具体动作的能力。为了解决这个痛点，让AI从"会说话的百科全书"真正进化为"能干活的数字员工"，我们需要引入Agent架构中最核心的设计模式：工具使用（Tool Use），或称函数调用（Function Calling）。

问题陈述

从根本上说，大语言模型的局限性在于其"闭环"特性。它们的输出仅仅基于训练数据中编码的静态权重。这种固有的局限性阻碍了它们完成需要以下条件的应用场景：

  1. 实时信息：获取当下的天气、最新的股票价格或突发新闻。
  2. 私有/动态数据：查询企业内部数据库、检查特定用户的订单状态。
  3. 确定性计算：进行复杂的数学运算或统计分析（大模型是概率模型，容易在精确计算上产生幻觉）。
  4. 对世界产生影响：发送邮件、控制智能家居设备、在云端执行代码。

如果没有一座连接大模型和外部系统的桥梁，LLM解决现实世界复杂问题的实用性将受到极大的限制。

第一性原理推导：为什么要"调用工具"？

让我们采用第一性原理来思考这个问题：大模型的最底层机制是什么？是Next-Token Prediction（下一个词元预测）。

大模型无法"点击"按钮，也无法直接"发送"网络请求，它唯一能做的事情就是生成文本。那么，我们如何用纯粹的文本输出来触发真实的物理或数字动作呢？

答案是：结构化协议（Structured Protocol）。

我们需要一种机制，让大模型的文本输出不再是对用户的直接回答，而是变成给外部系统的机器可读指令。我们将外部世界的API、数据库查询和系统命令封装成一个个的"函数"（Functions），并用自然语言向大模型解释这些函数的用途和参数。当大模型通过推理发现自己需要某个信息时，它不会去瞎猜（幻觉），而是预测出一串结构化的文本（通常是JSON格式），这串文本精确地描述了它想调用哪个函数以及传入什么参数。

这种机制巧妙地将大模型的概率性推理能力与外部系统的确定性执行能力解耦并连接了起来。这就是工具使用的本质：大模型负责"想"，外部工具负责"做"，然后再把"做"的结果返回给大模型继续"想"。

核心概念详解：Tool Use 还是 Function Calling？

在业界，这两个词经常混用，但理解它们的微妙区别有助于建立更深层的直觉：

  • Function Calling（函数调用）：这是一个更偏向底层技术的词汇。它描述了具体的技术实现机制——大模型生成一个包含函数名和参数的结构化输出，然后由宿主环境（Agent框架）去执行这个代码函数。
  • Tool Use（工具使用）：这是一个更宏大的架构视角。虽然底层是通过调用函数实现的，但我们应该把视野打开。一个"工具"不一定只是几行Python代码，它可以是一个复杂的API端点，一个针对私有知识库的数据库检索（RAG），甚至是向另一个专门的子Agent发送的任务指令。这种视角让我们能够构建更复杂的生态系统，其中主Agent扮演交响乐团指挥的角色，调度各种数字化资源和智能实体。

关键机制分析：工具调用的六步生命周期

工具使用模式并非一蹴而就，它包含一个精密的、循环的信息流反馈机制。了解这六个步骤是掌握该模式的核心：

  1. 工具定义（Tool Definition）：为什么需要这一步？因为大模型不具备预知能力。你必须向大模型清晰地描述你提供了哪些"手"。这包括工具的名称、用途说明、所需参数及其数据类型。这是最关键的一步，工具描述的质量直接决定了模型能否正确决定何时使用它。
  2. 大模型决策（LLM Decision）：当用户提出请求时，大模型会分析请求并查看可用的工具列表。在这里，大模型进行了一次推理：回答这个问题需要外部信息吗？哪个工具能提供这个信息？
  3. 生成结构化调用（Function Call Generation）：如果大模型决定使用工具，它会暂停生成面向用户的自然语言回复，转而生成一个JSON对象。这个JSON精准地从用户请求中提取出工具所需的参数（例如：从"伦敦天气如何"中提取出地点="London"）。
  4. 工具执行（Tool Execution）：注意，大模型本身不执行代码！拦截这个JSON对象并真正去执行API请求的是Agentic框架（如LangChain或Google ADK）的编排层。
  5. 观察/结果返回（Observation/Result）：API调用或代码执行的结果（可能是成功的数据，也可能是错误报错）会被捕获，并作为"观察结果"反馈给大模型。
  6. 大模型处理（LLM Processing）：大模型接收到工具的输出结果，将其纳入上下文，然后进行第二次推理：这个结果足够回答用户的问题吗？如果足够，就将其转化为人类友好的自然语言回复给用户；如果不够，它可能会决定调用另一个工具（多步调用）。

框架实战代码示例：LangChain基础的工具绑定

在LangChain中，工具调用是一个两阶段的过程：先定义工具，然后将工具绑定到具备工具调用能力的大模型上。

from langchain_core.tools import tool as langchain_tool
from langchain.agents import create_tool_calling_agent, AgentExecutor
from langchain_core.prompts import ChatPromptTemplate

# 1. 为什么用装饰器？方便将现有的Python逻辑直接转化为大模型可理解的工具
@langchain_tool
def search_information(query: str) -> str:
    """Provides factual information on a given topic."""
    # 这里的Docstring会被抽取为工具描述发送给LLM！
    # ... 模拟搜索逻辑 ...
    simulated_results = {
        "weather in london": "It's currently 15°C and cloudy in London.",
        "capital of france": "The capital of France is Paris.",
    }
    return simulated_results.get(query.lower(), f"No specific information found for '{query}'.")

tools = [search_information]

# 2. 为什么需要AgentExecutor？
# 大模型只输出意图(JSON)，AgentExecutor充当了步骤4和5：真正运行函数并将结果返回给LLM
agent_prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant."),
    ("human", "{input}"),
    ("placeholder", "{agent_scratchpad}"),
])

agent = create_tool_calling_agent(llm, tools, agent_prompt)
agent_executor = AgentExecutor(agent=agent, verbose=True, tools=tools)

这段代码的核心在于解释了职责的划分：大模型（llm）只负责生成调用请求，而AgentExecutor才是管理执行环境和循环调用的运行时引擎。

决策指南与最佳实践：何时以及如何使用？

黄金法则（Rule of Thumb）：只要Agent需要突破LLM内部静态知识的壁垒，去与外部世界进行任何形式的互动（获取实时数据、操作私有数据、执行动作、进行精确计算），就必须采用工具使用模式。

构建工具模式的最佳实践：

  1. 把 Docstring 写成 Prompt：工具的描述、参数名称和类型提示不仅仅是为了代码自文档化，它们最终会变成大模型的System Prompt的一部分。描述越精确（说明工具能做什么、不能做什么、参数格式的具体要求），大模型调用得越准。
  2. 工具应该职责单一：不要写一个"处理一切"的超级函数。应该像微服务设计一样，提供职责单一、粒度清晰的工具集（例如：get_weather 和 get_stock 应该是两个工具，而不是一个 get_info 工具）。
  3. 返回原始数据结构：工具的返回值应尽量是纯净的 JSON 或基础数据类型，把"解读和美化数据"的工作交给 LLM 去完成。

6个黄金应用场景：

  • 信息检索：访问实时数据或LLM训练数据中不存在的信息，例如天气Agent使用天气API。
  • 数据库与API交互：对结构化数据执行查询、更新或其他操作，例如电商Agent检查产品库存。
  • 计算与数据分析：使用外部计算器、数据分析库或统计工具，确保计算精确性。
  • 发送通信：发送电子邮件、消息或调用外部通信服务。
  • 执行代码：在安全沙箱环境中运行代码片段，执行特定任务。
  • 控制其他系统或设备：与智能家居设备、IoT平台或其他连接系统交互。

常见陷阱

  1. 客户端执行与自动扩展的区别：标准的函数调用（Function Calling）需要你在客户端手动编写框架代码去截获JSON并执行函数。Google Cloud 的 Vertex Extensions 提供了一种更高级的机制：扩展程序会自动、安全地在云端代为执行 API 调用，而无需客户端干预。搞混这两种执行模式会导致架构设计上的误判。
  2. 死循环与幻觉调用：如果工具的报错信息不清晰，大模型可能会陷入"尝试调用工具 -> 失败 -> 用相同的错误参数再次调用"的死循环。必须在 AgentExecutor 中设置最大重试次数，并在工具层面返回具有指导意义的错误信息。推荐抛出标准Python异常，而不是返回格式化的错误字符串，这样Agent能够利用自身的推理能力去决定下一步动作。

总结

工具使用（Tool Use / Function Calling）模式代表了大型语言模型发展史上的一个重要分水岭：它标志着AI从单纯的"思考者"转变为能够切实改造世界的"行动者"。

通过建立一种大模型能够理解的结构化交互协议，并结合强大的编排框架（如 LangChain、CrewAI、Google ADK），我们将 LLM 无与伦比的自然语言推理能力与传统软件系统精确、实时的执行能力完美融合在一起。理解并精通这种模式，是我们在今天设计下一代真正具备解决复杂现实问题能力的 Agent 系统的基石。
