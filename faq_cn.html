什么是"智能体设计模式"？+
智能体设计模式是针对构建智能自主系统（智能体）时遇到的常见问题的可重用高层次解决方案。这些模式为设计智能体行为提供了结构化框架，就像软件设计模式为传统编程所做的那样。它们帮助开发者构建更健壮、更可预测、更有效的 AI 智能体。
本书的主要目标是什么？+
本书旨在提供一个实践性、动手型的设计和构建智能体系统入门指南。它超越了理论讨论，提供了具体的架构蓝图，供开发者创建能够以可靠方式执行复杂、目标导向行为的智能体。
目标读者是谁？+
本书面向正在使用大语言模型（LLM）和其他 AI 组件构建应用的 AI 开发者、软件工程师和系统架构师。适合那些希望从简单的提示-响应交互迈向创建复杂自主智能体的人。
涵盖了哪些关键智能体模式？+
本书涵盖 21 种模式，包括：反思（自我批评）、规划（目标分解）、工具使用（外部 API）、多智能体协作（专家团队）、人机协同（人工监督）、记忆管理、知识检索（RAG）、路由、并行化、异常处理、防护栏/安全模式等。
常见误区
以下是开发者初次接触智能体系统时最常见的误解。点击每张卡片查看澄清说明。

"智能体只是聊天机器人"

错误——智能体主动规划并采取行动

聊天机器人只是响应提示。智能体感知环境、做出决策、使用工具、维护状态，并采取多步骤行动来实现目标——远超单轮问答。

"自主性越高越好"

错误——自主性需要相应的安全保障

更高的自主性要求更强的安全基础设施。没有防护栏和人机协同模式，高度自主的智能体可能采取有害或不可逆的真实世界行动。人机协同模式对高风险任务尤为关键。

"提示链 = 完整智能体"

错误——它只是众多基础模式之一

提示链是最简单的智能体模式——一个线性管道。生产级智能体还需要工具使用、记忆、反思和错误处理等叠加层。仅靠链式结构无法很好地处理动态、有状态或多轮场景。

"反思只是重新提示"

错误——它需要结构化的批评

仅仅要求模型"检查你的工作"通常只会产生改写而非真正改进。真正的反思需要针对特定标准进行结构化批评——最好由单独的评论家智能体来完成——从而实现真正的质量提升，而不仅仅是风格上的修改。

持续的人工干预会降低效率吗？+
可能会，这就是为什么关键在于找到正确的平衡。人机协同（HITL）应在关键检查点实施——计划审批、工具使用确认、歧义消解、最终输出审查——而不是每一个单独的动作都需要。目标是建立协作伙伴关系，智能体处理大量工作，人类提供战略指导。
模式专项问答
关于书中关键模式的详细问答。点击任意问题展开。
反思模式
反思模式如何减少幻觉？+
通过提示智能体审查其工作——尤其是将陈述与已知来源进行比较或检查推理步骤——反思模式迫使智能体更加扎根于所提供的上下文，减少生成无依据信息的可能性。智能体被明确要求验证其声明，而不是将第一个输出视为最终结果。
反思的两种主要类型是什么？+
"检查你的工作"反思——基本形式，直接要求智能体审查并修复之前的输出。适合发现简单错误。"内部评论家"反思——更高级的形式，使用单独的"评论家"智能体或专用提示，根据特定标准评估"工作者"智能体的输出，实现更严格和有针对性的改进。
规划模式
为什么规划对复杂任务是必要的？+
LLM 可能难以处理需要多个步骤或依赖关系的任务。没有计划，智能体可能会失去对整体目标的追踪，遗漏关键步骤，或无法将一步的输出作为下一步的输入。计划提供了清晰的路线图，确保原始请求的所有要求都以合理的顺序得到满足。
智能体如何在执行计划时处理错误？+
健壮的规划模式允许动态调整。如果某个步骤失败或情况发生变化，智能体可以被提示从当前状态"重新规划"。它分析错误，修改剩余步骤，或添加新步骤来克服障碍。这由异常处理与恢复模式（第 12 章）来补充。
多智能体协作
什么是编排器智能体？+
常见的多智能体架构涉及编排器智能体（有时称为"管理者"或"指挥者"）。编排器理解整体目标，将其分解，并将子任务委派给适当的专业智能体。然后它收集专家的结果并综合成最终输出。这是第 7 章探讨的主要架构模式。
智能体之间如何相互通信？+
通信通常由编排器管理——将一个智能体的输出作为上下文传递给下一个。共享的"草稿板"或消息总线（智能体在此发布结果）是另一种常见方法。智能体间通信（A2A）和模型上下文协议（MCP）模式为这种通信提供了标准化方法。
概念问答
关于智能体系统的底层概念、术语和心智模型的问题。
"工具"和"技能"有什么区别？+
工具通常指智能体可以调用的外部资源（如天气 API、计算器）。技能是一种更综合的能力，将工具使用与内部推理结合起来执行特定功能（如"预订航班"的技能可能包括使用日历和航空公司 API 以及规划逻辑）。技能是由工具构建的高阶能力。
什么是 ReAct（推理与行动）框架？+
ReAct 是一个将推理和行动集成在一起的流行框架。智能体遵循以下循环：思考（推理应该做什么）、行动（决定使用哪个工具及其输入）、观察（查看工具结果）。这个循环持续直到收集了足够的信息。它是工具使用和规划模式的基础。
什么是"智能体轨迹"？+
智能体轨迹是智能体执行任务时所有步骤的完整日志——包括它的所有思考、行动（工具调用）和观察。分析轨迹是调试和理解智能体行为的关键部分。它是评估与监控模式（第 19 章）中使用的主要工件。
什么是"提示泄露"？+
提示泄露是指系统提示的某些部分（工具定义、内部指令）无意中出现在智能体对用户的最终响应中。这会暴露实现细节并使用户感到困惑。防止方法包括将推理和最终答案生成使用独立提示，以及明确指示不重复系统提示内容。
四个智能体复杂度级别是什么？+
0 级（核心推理引擎）：没有工具、记忆或环境交互的 LLM。1 级（连接型问题解决者）：使用工具和 RAG 获取实时信息。2 级（战略型问题解决者）：工具使用 + 规划 + 上下文工程 + 自我改进。3 级（协作型多智能体）：多个专业智能体被编排以处理复杂工作流。
实践问答
关于在真实系统中应用这些模式的问题。选择一个应用领域。
应该从哪个模式开始？+
对于任何多步骤任务，从提示链（第 1 章）开始——它是最简单的基础模式。当智能体需要实时信息或外部行动时添加工具使用（第 5 章）。当上下文必须在多轮中持久化时添加记忆管理（第 8 章）。当输出质量不足时添加反思（第 4 章）。只有在单智能体方法达到上限时才引入多智能体。
本书使用哪些框架？+
本书使用三个主要框架：LangChain（及 LangGraph）用于灵活的链式和基于图的智能体工作流；CrewAI 用于具有角色和任务的多智能体编排；Google 智能体开发工具包（ADK）用于与 Google AI 基础设施集成的生产级智能体。每个代表应用模式的不同"画布"。
智能体如何决定使用哪个工具？+
智能体被提供可用工具列表，包括每个工具的描述和所需参数。当面临无法仅凭内部知识处理的请求时，智能体的推理能力允许它选择最合适的工具。路由模式（第 2 章）可以在此选择过程之上添加显式的条件逻辑。
实现工具使用的主要挑战是什么？+
错误处理：工具可能失败、返回意外数据或超时——智能体必须识别这些情况并决定是否重试、使用其他工具或寻求帮助。安全性：授予工具访问权限有严重影响；对敏感操作的保障、权限和人工审批至关重要。提示工程：智能体必须被提示生成正确格式的工具调用。
什么时候应该使用多智能体而不是单智能体？+
在以下情况使用多智能体：(1) 任务需要在单个系统提示中无法捕获的真正不同专业知识；(2) 工作流对单个智能体来说太复杂，无法在不失去上下文的情况下跟踪；(3) 独立子任务的并行执行可以显著加速；(4) 质量控制受益于独立的评论家视角。先从单智能体开始，只有在必要时才增加复杂度。
多智能体系统有哪些优势？+
模块化与专业化：每个智能体可以针对其特定任务进行微调（研究员、写作者、评论家），从而实现更高质量。降低复杂性：将复杂工作流分解为专业角色使系统更易于设计、调试和维护。模拟头脑风暴：不同智能体提供不同视角，产生更具创意和鲁棒性的解决方案，类似于人类团队的工作方式。
评估智能体为什么比传统软件更难？+
传统软件具有确定性输出（相同输入 → 相同输出）。使用 LLM 的智能体是非确定性的，其性能通常是主观的。评估它们需要评估输出的质量和相关性，而不仅仅是它是否在技术上"正确"。这需要新的评估方法：基于结果的评估、基于过程的评估和人工评估共同发挥作用。
如何为非确定性系统创建可靠测试？+
虽然你无法保证输出的确切措辞，但你可以编写检查关键元素的测试：最终响应是否包含特定的必要信息？智能体是否成功调用了带有正确参数的特定工具？它是否完成了所需步骤？通常在专用测试环境中使用模拟工具来完成。LLM 作为评判者（第 19 章）提供了自动评分方法。
实现问答
关于构建智能体的技术问题，包括关键代码模式。
提示智能体与简单的 ChatGPT 提示有何不同？+
提示智能体涉及创建一个详细的"系统提示"或章程，作为其操作指令。这超越了单个用户查询——它定义了智能体的角色、可用工具、应遵循的模式（如 ReAct 或规划）、约束和性格。系统提示实际上就是智能体的"编程"。
一个好的系统提示包含哪些内容？+
一个强大的系统提示通常包括：角色与目标（智能体是谁及其主要目的）；工具定义（可用工具、描述和使用格式）；约束与规则（智能体不应做什么）；流程指令（遵循哪些模式——"首先制定计划，然后逐步执行"）；示例轨迹（几个成功的思考-行动-观察循环示例可显著提高可靠性）。

最简系统提示结构

以下结构展示了生产级智能体系统提示的关键组成部分。

# 最简智能体系统提示结构（中文版） SYSTEM_PROMPT = """ 你是一个研究助理智能体。你的目标是通过从 多个来源收集信息来回答用户问题。 ## 可用工具 - search(query: str) -> str: 在网络上搜索信息 - fetch_doc(url: str) -> str: 从 URL 检索内容 - summarize(text: str) -> str: 对长文档进行总结 ## 流程 1. 分析用户的请求 2. 创建简短计划（编号步骤） 3. 使用适当的工具执行每个步骤 4. 将结果综合成最终答案 ## 约束 - 始终引用你的来源 - 不捏造统计数据或引用 - 如果工具失败，尝试替代方法 - 如果目标不明确，请求澄清 """
进阶问答
面向构建大规模生产级智能体系统的开发者的问题。
智能体系统未来的发展趋势是什么？+
本书指出以下方向：更自主的智能体，需要更少的人工干预并能自行学习适应；高度专业化的智能体，作为开放生态系统中的服务（"智能体经济"）；更好的工具和平台，提供更复杂的框架用于构建、测试和部署健壮的多智能体系统；以及新安全模式，解决对齐、鲁棒性和对抗性场景问题。
什么是模型上下文协议（MCP）？+
MCP 是由 Anthropic 开发的新兴标准，规范了智能体和工具之间如何交换信息。它不需要每个集成都定制代码，而是定义了工具发现、调用和响应的标准化协议。这是"智能体生态系统"愿景的基础——使智能体能够发现和使用未明确编程其使用的工具。
防护栏/安全模式章节如何处理安全性？+
第 18 章涵盖：输入验证/净化防止注入攻击；输出过滤/后处理屏蔽有害内容；行为约束通过明确的系统提示规则；回调监控每个智能体行动；关注点分离限制智能体权限仅限于所需内容；检查点和回滚处理可恢复的失败；以及外部审核 API作为最终安全层。
什么是"上下文工程"，为什么它很重要？+
上下文工程是在智能体工作流的每个步骤中战略性地选择、打包和管理最相关信息的过程。由于 LLM 的上下文窗口有限，输入不相关的信息会浪费容量并降低性能。上下文工程确保每个智能体步骤获得简短、聚焦而强大的上下文——最大化准确性和效率。

判断题测验
通过这些判断题测试你对智能体设计模式的理解。
关键澄清
构建智能体系统时最重要的澄清要点。
  • 智能体不仅仅是聊天机器人。智能体感知、规划、使用工具、维护状态，并朝目标采取多步骤行动。这种差异是架构层面的，不仅仅是模型能力的问题。
  • 反思需要结构化批评，而不仅仅是重新提示。简单地要求"检查你的工作"很少能产生有意义的改进。评论家智能体或结构化评估标准对于真正的质量提升是必要的。
  • 规划与规划执行是独立的关注点。智能体首先创建计划（规划模式），然后逐步执行（提示链/工具使用）。在失败时重新规划的能力使其具有鲁棒性。
  • 人机协同不是失败的表现。它是在关键决策点保证安全和质量的深思熟虑的设计选择。目标是协作智能，而不仅仅是自主智能。
  • 从简单开始，每次添加一种模式。提示链 → 添加工具使用 → 添加记忆 → 添加反思 → 添加多智能体。每次添加都应解决更简单系统中观察到的特定局限性。
  • 非确定性需要新的测试策略。测试关键结果和成功的工具调用，而不是精确的输出字符串。LLM 作为评判者和基于结果的评估是你的主要工具。
