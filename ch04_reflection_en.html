Reflection

A pattern where agents evaluate their own outputs and iteratively refine them — introducing a feedback loop of generation, critique, and improvement to produce higher-quality results.

What is the Reflection Pattern?

The Reflection pattern involves an agent evaluating its own work, output, or internal state and using that evaluation to improve its performance or refine its response. It is a form of self-correction or self-improvement, allowing the agent to iteratively refine its output or adjust its approach based on feedback, internal critique, or comparison against desired criteria.

The Feedback Loop

Unlike a simple sequential chain, Reflection introduces a feedback loop. The agent doesn't just produce an output; it then examines that output, identifies potential issues or areas for improvement, and uses those insights to generate a better version. Reflection can occasionally be facilitated by a separate agent whose specific role is to analyze the output of an initial agent.

Producer-Critic Model

A highly effective implementation separates the process into two distinct logical roles: a Producer and a Critic (the "Generator-Critic" or "Producer-Reviewer" model). Using two specialized agents with distinct system prompts often yields more robust and unbiased results, preventing the cognitive bias of an agent reviewing its own work.

Relationship to Other Patterns

The effectiveness of Reflection is enhanced when the LLM has memory of the conversation (Chapter 8). It also intersects with goal-setting and monitoring (Chapter 11), where Reflection acts as the corrective engine, using monitored feedback to analyze deviations and adjust strategy, transforming a passive executor into a purposeful, adaptive system.

Problems Reflection Solves
Even with sophisticated agentic workflows, an agent's initial output or plan might not be optimal, accurate, or complete. Click each card to explore the specific failures this pattern addresses:

Suboptimal First Outputs

Initial responses often lack polish or completeness

Agents producing an initial output or plan frequently generate results that are functional but not optimal. Without a review cycle, these first-draft outputs become the final answer, even when significant improvements are possible with additional passes.

Factual Inaccuracies

LLMs can generate plausible but incorrect information

Language models sometimes produce factually incorrect statements with high confidence. A separate critic agent, acting as a meticulous fact-checker, can verify claims against the original task requirements and flag inaccuracies for correction.

Code Bugs and Inefficiencies

Generated code may contain errors or style issues

When generating code, LLMs may produce implementations that compile but have bugs, miss edge cases, or violate style requirements. A senior engineer critic persona can catch these issues — bugs, missing error handling, inefficiencies — and guide refinement.

Cognitive Bias in Self-Review

A single agent reviewing its own work misses its own blind spots

When the same LLM call both generates and reviews output, it is prone to confirming its own assumptions. A separate critic agent with a distinct system prompt approaches the output with a fresh perspective, dedicated entirely to finding errors and areas for improvement.

No Mechanism for Improvement

Basic workflows lack built-in self-correction

Standard agentic patterns (chaining, routing, parallelization) do not include any mechanism for the agent to recognize and fix its own errors. The initial response becomes the final one regardless of quality. Reflection addresses this structural gap.

The Reflection Loop
The Reflection pattern introduces a structured cycle of generation, evaluation, and refinement. Click each step to explore what happens:
1
Execution (Generate)
Role: Producer Agent
The agent (Producer) performs the task and generates an initial output. Its sole focus is on generating the content — writing code, drafting text, creating a plan. It takes the initial prompt and produces the first version of the output.

Input: Task prompt + requirements
Output: First draft / initial attempt
2
Evaluation / Critique
Role: Critic Agent
The Critic agent (often a separate LLM call with a different system prompt) analyzes the result from the previous step. This evaluation checks for factual accuracy, coherence, style, completeness, and adherence to instructions. The Critic is designed to find flaws and provide structured feedback — or signal that the output is already perfect.

Input: Original task + first draft
Output: Structured critique / "CODE_IS_PERFECT"
3
Reflection / Refinement
Role: Producer Agent (with critique context)
Based on the critique, the Producer agent determines how to improve. This involves generating a refined output, adjusting parameters, or even modifying the overall plan. The conversation history containing the task, previous output, and critique provides full context for targeted improvement.

Input: Original task + previous output + critique
Output: Refined version
4
Iteration (Loop or Stop)
Role: Orchestrator
The refined output can be re-evaluated and the process repeats until a satisfactory result is achieved or a stopping condition is met (e.g., critique signals "perfect", max iterations reached). This iterative cycle progressively enhances quality, leading to more accurate, coherent, and reliable outcomes.

Stopping conditions: Critic approves OR max iterations reached
Output: Final, high-quality result
Core Concepts

The Two-Agent Architecture

The most powerful implementation of Reflection uses two distinct agents: the Producer (focuses entirely on generating content) and the Critic (sole purpose is to evaluate the Producer's output using a specialized persona like "senior software engineer" or "meticulous fact-checker"). This separation prevents cognitive bias and enables specialized, structured feedback.

Conversation History as Context

The reflection loop maintains a conversation history containing the original task, generated outputs, and critiques. This history provides full context for each subsequent step, enabling the agent to learn from past critiques and avoid repeating errors. Without memory, each reflection is isolated; with memory, it becomes cumulative.

Stopping Conditions

A key design decision is when to stop iterating. Common approaches include: (1) a sentinel phrase from the critic (e.g., "CODE_IS_PERFECT"), (2) a maximum number of iterations to control cost, or (3) a quality threshold score. Balancing thoroughness against latency and cost is essential for practical deployment.

Practical Applications
Reflection is valuable in scenarios where output quality, accuracy, or adherence to complex constraints is critical. Select a use case:

Creative Writing and Content Generation

Refining generated text, stories, poems, or marketing copy to achieve polish and effectiveness.

1
Use Case: An agent writing a blog post.
2
Reflection: Generate a draft, critique it for flow, tone, and clarity, then rewrite based on the critique. Repeat until the post meets quality standards.
3
Benefit: Produces more polished and effective content that would be difficult to achieve in a single pass.

Code Generation and Debugging

Writing code, identifying errors, and fixing them through iterative critique.

1
Use Case: An agent writing a Python function.
2
Reflection: Write initial code, run tests or static analysis, identify errors or inefficiencies via a senior engineer critic, then modify the code based on findings.
3
Benefit: Generates more robust, functional code with proper edge case handling and documentation.

Complex Problem Solving

Evaluating intermediate steps or proposed solutions in multi-step reasoning tasks.

1
Use Case: An agent solving a logic puzzle or math problem.
2
Reflection: Propose a step, evaluate if it leads closer to the solution or introduces contradictions, backtrack or choose a different step if needed.
3
Benefit: Improves the agent's ability to navigate complex problem spaces that require backtracking.

Summarization and Information Synthesis

Refining summaries for accuracy, completeness, and conciseness.

1
Use Case: An agent summarizing a long document.
2
Reflection: Generate an initial summary, compare it against key points in the original document, refine to include missing information or improve accuracy.
3
Benefit: Creates more accurate and comprehensive summaries with fewer omissions.

Planning and Strategy

Evaluating a proposed plan and identifying potential flaws or improvements before execution.

1
Use Case: An agent planning a series of actions to achieve a goal.
2
Reflection: Generate a plan, simulate its execution or evaluate its feasibility against constraints, revise the plan based on the evaluation.
3
Benefit: Develops more effective and realistic plans with fewer blind spots.

Conversational Agents

Reviewing previous turns to maintain context, correct misunderstandings, or improve response quality.

1
Use Case: A customer support chatbot.
2
Reflection: After generating a user response, review the conversation history and the last message to ensure coherence and that the user's latest input is accurately addressed.
3
Benefit: Leads to more natural and effective conversations with fewer misunderstandings.
Code Examples
The following examples demonstrate the Reflection pattern using LangChain (iterative loop) and Google ADK (Generator-Critic pipeline).

LangChain: Iterative Reflection Loop

from langchain_openai import ChatOpenAI from langchain_core.messages import SystemMessage, HumanMessage from dotenv import load_dotenv load_dotenv() llm = ChatOpenAI(model="gpt-4o", temperature=0.1) def run_reflection_loop(): task_prompt = """Write a Python function named `calculate_factorial`. It must: accept integer n, compute n!, include a docstring, handle n=0, raise ValueError for negative n.""" max_iterations = 3 message_history = [HumanMessage(content=task_prompt)] current_code = "" for i in range(max_iterations): # --- GENERATE / REFINE STAGE --- if i == 0: response = llm.invoke(message_history) else: message_history.append( HumanMessage(content="Please refine the code using the critiques.")) response = llm.invoke(message_history) current_code = response.content message_history.append(response) # --- REFLECT STAGE (Critic persona) --- reflector_prompt = [ SystemMessage(content="""You are a senior software engineer. Critically evaluate the code against the original task. Look for bugs, style issues, missing edge cases. If perfect, respond with 'CODE_IS_PERFECT'. Otherwise, provide a bulleted critique."""), HumanMessage(content=f"Task:\n{task_prompt}\n\nCode:\n{current_code}") ] critique = llm.invoke(reflector_prompt).content # --- STOPPING CONDITION --- if "CODE_IS_PERFECT" in critique: break message_history.append( HumanMessage(content=f"Critique:\n{critique}")) return current_code

Google ADK: Generator-Critic Pipeline

from google.adk.agents import SequentialAgent, LlmAgent # The first agent generates the initial draft generator = LlmAgent( name="DraftWriter", instruction="Write a short, informative paragraph about the user's subject.", output_key="draft_text" # output saved to state ) # The second agent critiques the draft from the first agent reviewer = LlmAgent( name="FactChecker", instruction="""You are a meticulous fact-checker. Read the text in state key 'draft_text'. Verify factual accuracy of all claims. Output a dict: {'status': 'ACCURATE'|'INACCURATE', 'reasoning': '...'}""", output_key="review_output" ) # SequentialAgent ensures generator runs before reviewer review_pipeline = SequentialAgent( name="WriteAndReview_Pipeline", sub_agents=[generator, reviewer] )
Deep Dive: Design Dimensions
Click each layer to explore key design considerations for implementing the Reflection pattern effectively:
Single-Agent vs. Dual-Agent Reflection
Architectural choice that impacts objectivity and quality
A single agent can perform self-reflection by reviewing its own output. However, this introduces cognitive bias — the model tends to agree with its previous output. Using two separate LLM calls with distinct system prompts (Producer + Critic) yields more robust and unbiased results. The Critic approaches the output with a fresh perspective, dedicated entirely to finding errors and areas for improvement. For high-stakes tasks, always prefer the dual-agent approach.
Cost and Latency Trade-offs
Each reflection cycle requires additional LLM calls
The iterative process significantly enhances output quality but comes at the cost of increased latency and computational expense. Every refinement loop requires at least one new LLM call, making it suboptimal for time-sensitive applications. The pattern is also memory-intensive; with each iteration, the conversational history expands. Design your stopping conditions carefully to balance quality against cost and speed.
State Management for Iteration
How to maintain context across multiple reflection cycles
True iterative reflection requires stateful workflows. In LangChain, this can be achieved through graph-based frameworks like LangGraph that support state management and conditional transitions. In simpler implementations, maintaining a conversation history list (task → output → critique → refined output) provides the necessary context. Google ADK's LoopAgent is also available for iterative loops with state management.
Critic Persona Design
The system prompt for the Critic agent is critical
The quality of reflection depends heavily on how you instruct the Critic. Effective Critic prompts give a specialized role ("You are a senior software engineer"), define specific evaluation criteria (factual accuracy, code quality, stylistic requirements, completeness), instruct it to produce structured output (bulleted critique), and provide a stopping signal ("CODE_IS_PERFECT" when no issues are found). Domain-specific critics produce far better results than generic reviewers.
Integration with Memory and Goals
Reflection is amplified by contextual awareness
The effectiveness of Reflection is significantly enhanced when the LLM keeps a memory of the conversation. This conversational history provides crucial context for the evaluation phase, allowing the agent to assess its output against previous interactions, user feedback, and evolving goals. It enables the agent to learn from past critiques and avoid repeating errors. When combined with goal monitoring, Reflection acts as the corrective engine, analyzing deviations and adjusting strategy.
At a Glance

WHAT

An agent's initial output is often suboptimal. Basic workflows lack a built-in process for the agent to recognize and fix its own errors. Reflection solves this by having the agent evaluate its own work, preventing the initial response from being the final one regardless of quality.

WHY

Reflection offers self-correction and refinement via a feedback loop where a "producer" generates output and a "critic" evaluates it against predefined criteria. This iterative process progressively enhances quality, leading to more accurate, coherent, and reliable outcomes.

WHEN TO USE

Use Reflection when quality, accuracy, and detail of the final output are more important than speed and cost. Particularly effective for polished long-form content, writing and debugging code, and creating detailed plans. Use a separate critic for high objectivity needs.

TRADE-OFFS

Benefits come at the cost of increased latency and expense. Each refinement loop requires new LLM calls. The pattern is also memory-intensive as history expands. Full iterative reflection requires stateful workflows like LangGraph; single-step reflection can use simpler LCEL chains.

Knowledge Check
Key Takeaways
  • The Reflection pattern introduces a feedback loop of execution, evaluation/critique, and refinement — moving agents beyond single-pass execution toward iterative self-improvement.
  • The primary advantage is the ability to iteratively self-correct and refine outputs, leading to significantly higher quality, accuracy, and adherence to complex instructions.
  • The Producer-Critic model (separate agents with distinct system prompts) is a powerful implementation that prevents cognitive bias and enables more specialized, objective feedback.
  • A well-designed Critic agent must have a specialized role, clear evaluation criteria, structured output format, and a stopping signal (like "CODE_IS_PERFECT") to terminate the loop.
  • These benefits come at the cost of increased latency and computational expense — every refinement loop requires at least one additional LLM call. Balance quality vs. cost carefully.
  • Full iterative reflection requires stateful workflows (LangGraph, ADK LoopAgent); a single reflection step can be implemented with simpler LCEL chains or sequential ADK pipelines.
  • Google ADK facilitates reflection through sequential workflows where one agent's output is critiqued by another, enabling subsequent refinement steps.
  • Reflection is most valuable when quality, accuracy, and completeness matter more than speed — for tasks like generating polished content, writing and debugging code, and creating detailed plans.
