并行化模式

在智能体工作流中，许多复杂任务包含可同时执行的独立子任务。并行化模式通过并发运行这些独立组件，显著降低总体延迟，提升系统响应速度。

模式概览

并行化是一种通过同时执行独立任务来优化智能体性能的关键模式。不同于逐步等待每个步骤完成的顺序方式，并行执行允许多个独立任务同时运行，大幅缩短整体执行时间。

什么是并行化？

并行化涉及并发执行多个组件，如 LLM 调用、工具使用，甚至整个子智能体。核心思路是识别工作流中不依赖其他部分输出的步骤，并同时执行它们，而非一个接一个地运行。

并行 vs. 顺序执行

顺序方式：搜索来源A → 摘要A → 搜索来源B → 摘要B → 综合答案（5步串行）

并行方式：同时搜索A和B → 同时摘要A和B → 综合答案（3步，大幅提速）

框架支持

LangChain（通过 LCEL 和 RunnableParallel）、LangGraph（图拓扑中的并行分支）、Google ADK（原生 ParallelAgent 原语）等现代智能体框架均内置了并行执行支持。

关键价值

对于涉及外部服务（如 API 或数据库）的任务，并行化尤为有效。在等待网络响应期间，系统可同时发起多个请求，将总延迟从各任务时间之和缩减为最慢单任务的时间。

问题所在

顺序执行在处理包含多个独立子任务的复杂工作流时面临严重的效率瓶颈。点击每张卡片了解具体挑战。

⏱ 累积延迟

顺序执行时总耗时是所有步骤时间之和。

若每次 API 调用需要 2 秒，顺序执行 5 个独立调用就需要 10 秒。而并行执行只需约 2 秒——节省了 80% 的等待时间。这对实时交互体验至关重要。

🔒 资源闲置浪费

等待一个任务完成时，计算资源处于空闲状态。

顺序工作流在等待外部 I/O（网络请求、数据库查询）时，CPU 和内存资源完全闲置。并行化让这些资源在等待期间处理其他独立任务，最大化利用率。

📉 响应性差

用户需要等待所有顺序步骤依次完成。

在研究类任务中，顺序方式意味着用户在最终结果出现前需要等待每一步——即使这些步骤完全独立。这严重影响用户体验，尤其在商业场景中难以接受。

🧱 扩展性受限

增加新数据源或分析维度会线性增加总时间。

顺序架构下，每新增一个数据源就会线性增加总处理时间。而并行架构中，只要新任务独立于其他任务，增加更多并行分支对整体延迟的影响微乎其微。

典型场景：研究型智能体

一个需要从多个来源搜集信息的研究智能体，若采用顺序方式：搜索新闻→摘要→搜索学术→摘要→搜索社交媒体→摘要→综合，每步平均 2 秒，共需 14 秒。并行化后仅需约 4 秒（搜索并行 + 摘要并行 + 综合），提速超过 3 倍。

解决方案：并行化模式

并行化模式的核心是识别独立任务并同时执行。点击各节点了解执行流程的每个阶段。

1
识别独立任务
任务分解阶段
分析工作流，找出哪些步骤的输出不依赖于其他步骤的结果。这些独立步骤是并行化的候选对象。例如：同时搜索多个数据源、对不同数据段应用不同分析方法、调用多个独立 API 等。关键原则：任务间无数据依赖，可以同时开始执行。
2
并发执行独立任务
并行执行阶段
使用框架提供的并行原语同时启动所有独立任务。在 LangChain 中使用 RunnableParallel，在 LangGraph 中通过图拓扑定义并行分支，在 Google ADK 中使用 ParallelAgent。所有任务同时运行，系统等待全部任务完成再进入下一阶段。异步编程（asyncio）是实现并发的常用技术手段。
3
汇聚并综合结果
结果整合阶段
当所有并行任务完成后，将各自的输出汇聚到一个综合步骤中。这个综合步骤通常是顺序的——它等待所有并行分支完成，然后将结果合并为统一的最终输出。例如：一个综合 LLM 接收所有研究摘要，生成最终报告。
4
输出最终结果
结果交付阶段
综合后的最终结果返回给用户或下游系统。整个并行工作流的总时间约等于最慢的单个并行任务耗时加上综合步骤耗时，而非所有任务时间之和，实现了显著的性能提升。

核心概念

掌握并行化模式所需的关键技术概念与术语。

异步执行（Async Execution）

Python 的 asyncio 提供并发而非真正的并行。它通过事件循环在单线程中智能地在任务间切换——当一个任务等待 I/O 时，切换到其他任务继续执行。这产生了多个任务同时推进的效果，非常适合 I/O 密集型的 LLM 调用。

async def run_parallel(): results = await asyncio.gather( task_a(), # 同时启动 task_b(), # 同时启动 task_c() # 同时启动 ) return results

RunnableParallel（LangChain）

LCEL 中的核心并行原语。将多个 Runnable 组件打包为字典或列表，当此集合作为输入传递给链中的下一个组件时，LCEL 运行时会并发执行所有包含的 Runnable。

map_chain = RunnableParallel( { "summary": summarize_chain, "questions": questions_chain, "key_terms": terms_chain, "topic": RunnablePassthrough() } )

ParallelAgent（Google ADK）

Google ADK 提供原生的 ParallelAgent 原语，用于协调多个子智能体的并发执行。各子智能体独立运行，将结果存储在会话状态中，由后续的综合智能体统一读取处理。

收敛点（Convergence Point）

并行分支汇聚的位置。在此点，系统等待所有并行任务完成，然后将结果传递给综合步骤。这通常是整个工作流中唯一的顺序依赖点，其后的步骤可以利用所有并行结果。

应用场景

并行化模式在多种实际场景中发挥关键作用。选择场景深入了解。

信息搜集与研究

从多个来源同时收集信息是并行化的经典用例。

1
场景：研究某家公司的智能体
2
并行任务：同时搜索新闻文章、拉取股票数据、检查社交媒体提及、查询公司数据库
3
收益：比顺序查找快数倍，获得全面的多维视角

数据处理与分析

对同一数据集并发应用不同分析技术或处理不同数据分片。

1
场景：分析客户反馈的智能体
2
并行任务：同时执行情感分析、关键词提取、反馈分类、紧急问题识别
3
收益：快速提供多维度分析报告，无需等待每项分析依次完成

多 API / 工具交互

同时调用多个独立的 API 或工具，收集不同类型的信息。

1
场景：旅行规划智能体
2
并行任务：同时查询航班价格、酒店可用性、当地活动、餐厅推荐
3
收益：更快呈现完整旅行方案，用户无需等待逐一查询

多组件内容生成

并行生成复杂内容的不同部分，最后汇聚合并。

1
场景：营销邮件创作智能体
2
并行任务：同时生成邮件主题行、正文草稿、配图建议、行动号召按钮文案
3
收益：显著加快完整邮件的组装速度，各部分独立生成无需等待

验证与核查

同时执行多项独立的检查或验证，快速获得全面的验证结果。

1
场景：用户输入验证智能体
2
并行任务：同时检查邮件格式、验证手机号、数据库核查地址、内容安全过滤
3
收益：更快速地提供输入有效性反馈，提升用户体验

多模态处理

对同一输入的不同模态（文本、图像、音频）同时进行处理分析。

1
场景：分析含图文的社交媒体帖子
2
并行任务：同时对文本进行情感与关键词分析，对图像进行物体检测与场景描述
3
收益：更快整合不同模态的洞察，获得全面的内容理解

A/B 测试与多选项生成

并行生成多个响应变体，快速比较选出最优方案。

1
场景：生成不同风格创意文案的智能体
2
并行任务：使用略微不同的提示词或模型，同时生成三个不同版本的文章标题
3
收益：快速完成比较与最优方案筛选，无需顺序等待每个变体生成

代码实战

以下展示两个框架的并行化实现：LangChain（使用 RunnableParallel）和 Google ADK（使用 ParallelAgent）。

LangChain — RunnableParallel 示例

三条独立链并行处理同一主题：摘要、问题生成、关键词提取，结果汇聚后由综合链整合。

import asyncio from langchain_openai import ChatOpenAI from langchain_core.prompts import ChatPromptTemplate from langchain_core.output_parsers import StrOutputParser from langchain_core.runnables import RunnableParallel, RunnablePassthrough llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.7) # 定义三条独立链 summarize_chain = ( ChatPromptTemplate.from_messages([ ("system", "请简洁地概括以下主题："), ("user", "{topic}") ]) | llm | StrOutputParser() ) questions_chain = ( ChatPromptTemplate.from_messages([ ("system", "围绕以下主题生成三个有趣的问题："), ("user", "{topic}") ]) | llm | StrOutputParser() ) terms_chain = ( ChatPromptTemplate.from_messages([ ("system", "从以下主题中识别5-10个关键术语，用逗号分隔："), ("user", "{topic}") ]) | llm | StrOutputParser() ) # 并行执行三条链 map_chain = RunnableParallel( { "summary": summarize_chain, "questions": questions_chain, "key_terms": terms_chain, "topic": RunnablePassthrough() } ) # 综合链整合并行结果 synthesis_prompt = ChatPromptTemplate.from_messages([ ("system", """基于以下信息： 摘要：{summary} 相关问题：{questions} 关键术语：{key_terms} 请综合生成全面的答案。"""), ("user", "原始主题：{topic}") ]) full_parallel_chain = map_chain | synthesis_prompt | llm | StrOutputParser() async def run_parallel_example(topic: str): response = await full_parallel_chain.ainvoke(topic) print(response) asyncio.run(run_parallel_example("太空探索的历史"))

Google ADK — ParallelAgent 示例

三个专业研究子智能体并行运行，结果存入会话状态，由综合智能体生成最终报告。

from google.adk.agents import LlmAgent, ParallelAgent, SequentialAgent GEMINI_MODEL = "gemini-2.0-flash" # 定义并行子智能体 researcher_1 = LlmAgent( name="可再生能源研究员", model=GEMINI_MODEL, instruction="研究最新可再生能源进展，用1-2句话总结发现。", output_key="renewable_result" ) researcher_2 = LlmAgent( name="电动汽车研究员", model=GEMINI_MODEL, instruction="研究电动汽车技术最新进展，用1-2句话总结发现。", output_key="ev_result" ) # ParallelAgent 并发运行所有子智能体 parallel_agent = ParallelAgent( name="并行研究智能体", sub_agents=[researcher_1, researcher_2] ) # 综合智能体读取并行结果生成报告 merger_agent = LlmAgent( name="综合智能体", model=GEMINI_MODEL, instruction="将研究发现整合为结构化报告。" ) # SequentialAgent 协调完整流程 pipeline = SequentialAgent( name="研究综合流水线", sub_agents=[parallel_agent, merger_agent] ) root_agent = pipeline

深入解析

并行化的实现涉及多个技术层面的考量。点击各层了解详细分析。

并发 vs. 真并行
asyncio 的工作原理
asyncio 提供的是并发而非真正的并行。它在单线程中通过事件循环在任务间智能切换——当一个任务等待网络响应时（如 LLM API 调用），切换到其他就绪任务继续执行。这受 Python GIL（全局解释器锁）的限制，但对 I/O 密集型的 LLM 工作流非常有效。若需 CPU 密集型真并行，可使用 concurrent.futures.ProcessPoolExecutor。
LCEL 中的并行机制
LangChain 实现原理
LangChain 表达式语言（LCEL）通过 RunnableParallel 实现并行。当字典或列表形式的多个 Runnable 作为链的输入时，LCEL 运行时自动并发执行所有包含的 Runnable。RunnablePassthrough 用于将原始输入透传到下游步骤，确保综合步骤能够访问原始数据。整个并行块的完成时间等于其中最慢的单个 Runnable 的执行时间。
LangGraph 中的并行分支
图拓扑实现并行
LangGraph 通过图结构的拓扑来实现并行。从单个公共节点触发的多个节点（没有直接顺序依赖关系）可以并行执行。这些并行路径独立运行，直到在图中的后续收敛点汇聚。这种方式允许以声明式的方式定义复杂的并行工作流，而无需手动管理 asyncio 协程。
Google ADK 的并行代理架构
原生多智能体并发
Google ADK 提供原生 ParallelAgent 原语，专为多智能体并发设计。子智能体通过 output_key 将结果存储在共享的会话状态中，后续的综合智能体读取这些状态。SequentialAgent 作为外层编排器，先等待 ParallelAgent 完成（所有子智能体执行完毕），再运行综合智能体。这种架构非常清晰地分离了并行研究阶段和顺序综合阶段。
成本与复杂性权衡
并行化的代价
并行化并非没有代价。引入并发或并行架构会显著增加系统复杂性，影响设计、调试和日志记录等关键开发阶段。多个并发 LLM 调用意味着更高的 API 成本（尽管总时间减少）。错误处理更复杂：一个并行分支失败如何影响其他分支？需要仔细设计容错机制。只有当性能收益明显超过这些代价时，才值得采用并行化。

一览概要

快速掌握并行化模式的核心要素，以及何时、为何、如何使用它。

问题（WHAT）

许多智能体工作流包含多个独立子任务，顺序执行时总时间是所有任务时间之和。当任务涉及外部 I/O（API 调用、数据库查询）时，这种延迟尤为显著，严重影响系统性能和响应性。

方案（WHY）

并行化模式通过识别工作流中无相互依赖的组件（如工具调用、LLM 请求），同时执行它们。LangChain 和 Google ADK 等框架提供内置构件管理并发操作，将总时间从各任务之和降至最慢任务的时间。

适用时机（WHEN）

当工作流包含多个可同时运行的独立操作时使用——如从多个 API 获取数据、处理不同数据分块、并行生成多个内容片段以供后续综合。任务间无数据依赖是关键判断标准。

经验法则（HOW）

分析任务依赖关系，识别可并行的独立步骤。使用框架的并行原语（RunnableParallel、ParallelAgent）并发执行，等待全部完成后进行综合。注意权衡：并行化会增加复杂性和调试难度，需确保性能收益值得这些代价。

知识测验

测试你对并行化模式的理解。

核心要点

本章关于并行化模式最重要的收获。

  • 并行化是一种通过并发执行独立任务来提升效率的智能体设计模式，将总时间从各任务之和降至最慢任务的时间。
  • 关键判断标准是任务独立性：只有当任务的输出不依赖于其他任务的结果时，才能安全地并行化。
  • 对于涉及外部 I/O（API 调用、数据库查询）的任务，并行化特别有效，因为这些任务大部分时间都在等待网络响应。
  • LangChain 的 RunnableParallel 和 Google ADK 的 ParallelAgent 是两种主流框架提供的并行化原语，各有其适用场景。
  • asyncio 在 Python 中提供的是并发（单线程事件循环中的任务切换）而非真正的多线程并行，但对 I/O 密集型 LLM 工作流同样有效。
  • 并行化会引入额外的系统复杂性，包括更复杂的错误处理、调试困难和日志管理，需要与性能收益仔细权衡。
  • 典型应用包括：信息搜集、数据分析、多 API 交互、内容并行生成、输入验证、多模态处理和 A/B 测试。
  • 结合顺序（链式）控制流和条件（路由）控制流，并行化使构建复杂、高性能的计算系统成为可能。
