What is MCP?

The Model Context Protocol (MCP) is an open standard designed to standardize how LLMs like Gemini, OpenAI's GPT models, Mixtral, and Claude communicate with external applications, data sources, and tools. Think of it as a universal adapter — any LLM can plug into any external system without a custom integration for each one. MCP provides a consistent and predictable interface for LLMs to obtain context, execute actions, and interact with various systems.

Client-Server Architecture

MCP operates on a client-server model. It defines how data (resources), interactive templates (prompts), and actionable functions (tools) are exposed by an MCP server. These are then consumed by an MCP client, which could be an LLM host application or an AI agent itself. This standardized approach dramatically reduces the complexity of integrating LLMs into diverse operational environments.

MCP vs. Direct Tool Calling

Tool function calling gives an AI a specific set of custom-built tools — like a particular wrench and screwdriver. MCP is like creating a universal, standardized power outlet system. It doesn't provide the tools itself, but it allows any compliant tool from any manufacturer to plug in and work, enabling a dynamic and ever-expanding workshop.

Design Considerations

MCP is a contract for an "agentic interface," and its effectiveness depends heavily on the design of the underlying APIs it exposes. Developers must consider not just the connection, but the nature of the data being exchanged. For example, returning files as PDFs is mostly useless if the consuming agent cannot parse PDF content. Agent-friendly data formats like Markdown are essential.

The Integration Problem
Without a standard like MCP, connecting LLMs to external systems creates numerous pain points. Click each card to explore the failure modes:

Custom Integration Explosion

Each LLM-to-tool connection requires bespoke code

Without a standard, every pair of LLM and external service requires a unique integration. With N models and M services, you need N×M integrations — an explosion of maintenance burden.

Vendor Lock-in

Tool integrations are tightly coupled to specific LLMs

When tool calling is proprietary (e.g., OpenAI-specific format), switching to a different LLM provider requires rewriting all tool integrations — a massive migration cost that discourages experimentation.

No Dynamic Discovery

Agents can't find new capabilities at runtime

In proprietary systems, the LLM must be explicitly told which tools are available in every conversation. There's no mechanism for an agent to dynamically discover new capabilities without code changes and redeployment.

Agent-Unfriendly Data Formats

APIs return data the agent can't process

Legacy APIs often return data in formats designed for humans or traditional software — PDFs, binary blobs, deeply nested XML. Agents need text-based, structured formats like Markdown or JSON to reason effectively over responses.

Weak Error Communication

Tool failures aren't communicated back to the LLM

Without a standard error protocol, when a tool call fails, the agent may receive an opaque error or nothing at all. It cannot understand the failure or try an alternative approach, leaving the task stuck.

MCP: A Standardized Interface
MCP solves the integration problem with a layered architecture. Click each component to explore:
1
LLM + MCP Client
The intelligent requester
The LLM is the core intelligence — it processes requests, formulates plans, and decides when to access external information. The MCP Client wraps the LLM, acting as the intermediary that translates the LLM's intent into a formal MCP request. It is responsible for discovering, connecting to, and communicating with MCP Servers.

Key responsibility: Dynamic capability discovery — querying servers to learn what tools and resources they offer.
2
MCP Server
The gateway to external capabilities
The MCP Server is the gateway to the external world. It exposes a set of tools, resources, and prompts to any authorized MCP Client. Each server is typically responsible for a specific domain — a company's internal database, an email service, or a public API.

Exposes three types: Resources (static data), Tools (executable functions), and Prompts (interaction templates).
3
Third-Party Service
The actual external system
The optional third-party service represents the actual external tool, application, or data source that the MCP Server manages and exposes. It is the ultimate endpoint that performs the requested action — querying a proprietary database, interacting with a SaaS platform, or calling a public weather API.

Examples: Google BigQuery, email SMTP server, weather API, CRM system.
4
Interaction Flow
Discovery → Request → Execute → Respond
1. Discovery: Client queries server for its manifest of available tools, resources, and prompts.
2. Request Formulation: LLM decides to use a tool (e.g., send_email) and formulates a request with parameters.
3. Client Communication: MCP Client sends the standardized call to the appropriate MCP Server.
4. Server Execution: Server authenticates, validates, and executes the action via the underlying service.
5. Response & Update: Server returns a standardized response; client passes result back to LLM.
Core Concepts
MCP defines three primitive types exposed by servers, plus key architectural considerations:

Resources, Tools, and Prompts

Resource: Static data — a PDF file, a database record, a configuration document. The agent reads it but does not execute it.

Tool: An executable function that performs an action — sending an email, querying an API, writing to a database. Tools have side effects.

Prompt: A template that guides the LLM in how to interact with a resource or tool, ensuring the interaction is structured and effective.

Transport Mechanisms

MCP defines the underlying transport layers for communication. For local interactions, it uses JSON-RPC over STDIO (standard input/output) for efficient inter-process communication. For remote connections, it leverages web-friendly protocols like Streamable HTTP and Server-Sent Events (SSE) to enable persistent and efficient client-server communication.

Example MCP tool manifest returned during discovery:

{ "tools": [ { "name": "send_email", "description": "Sends an email to a recipient.", "inputSchema": { "type": "object", "properties": { "recipient": { "type": "string" }, "subject": { "type": "string" }, "body": { "type": "string" } }, "required": ["recipient", "subject", "body"] } }, { "name": "query_customer_db", "description": "Query the customer database.", "inputSchema": { "type": "object", "properties": { "filter": { "type": "string" }, "limit": { "type": "integer", "default": 10 } } } } ] }

Security Considerations

Exposing tools and data via any protocol requires robust security measures. An MCP implementation must include authentication and authorization to control which clients can access which servers and what specific actions they are permitted to perform. Local servers offer speed and security for sensitive data, while remote servers allow shared, scalable access across an organization.

Database Integration

MCP allows LLMs and agents to seamlessly access and interact with structured data in databases. Using the MCP Toolbox for Databases, an agent can query Google BigQuery datasets to retrieve real-time information, generate reports, or update records, all driven by natural language commands.

5
Agent summarizes and presents the findings to the user

Hands-On Code
Two approaches: connecting an ADK agent to a local MCP filesystem server, and building a custom server with FastMCP.

Agent Setup with MCPToolset (ADK)

Connecting an ADK LlmAgent to the MCP filesystem server via StdioServerParameters:

import os from google.adk.agents import LlmAgent from google.adk.tools.mcp_tool.mcp_toolset import MCPToolset, StdioServerParameters # Build absolute path to the managed files directory TARGET_FOLDER_PATH = os.path.join( os.path.dirname(os.path.abspath(__file__)), "mcp_managed_files" ) os.makedirs(TARGET_FOLDER_PATH, exist_ok=True) root_agent = LlmAgent( model='gemini-2.0-flash', name='filesystem_assistant_agent', instruction=( 'Help the user manage their files. ' f'You are operating in: {TARGET_FOLDER_PATH}' ), tools=[ MCPToolset( connection_params=StdioServerParameters( command='npx', args=[ "-y", "@modelcontextprotocol/server-filesystem", TARGET_FOLDER_PATH, ], # tool_filter=['list_directory', 'read_file'] # optional ) ) ], )

Creating an MCP Server with FastMCP

FastMCP provides a high-level Python framework with automatic schema generation from type hints and docstrings:

# fastmcp_server.py — Simple MCP server exposing a greet tool from fastmcp import FastMCP mcp_server = FastMCP() # @mcp_server.tool registers this as an MCP-callable tool. # FastMCP reads the docstring + type hints to build the schema. @mcp_server.tool def greet(name: str) -> str: """Generates a personalized greeting. Args: name: The name of the person to greet. Returns: A greeting string. """ return f"Hello, {name}! Nice to meet you." if __name__ == "__main__": mcp_server.run( transport="http", host="127.0.0.1", port=8000 )

Consuming FastMCP Server with an ADK Agent

Configure an ADK agent as an MCP client connecting to the running FastMCP HTTP server:

from google.adk.agents import LlmAgent from google.adk.tools.mcp_tool.mcp_toolset import MCPToolset, HttpServerParameters FASTMCP_SERVER_URL = "http://localhost:8000" root_agent = LlmAgent( model='gemini-2.0-flash', name='fastmcp_greeter_agent', instruction='You are a friendly assistant that can greet people. Use the "greet" tool.', tools=[ MCPToolset( connection_params=HttpServerParameters( url=FASTMCP_SERVER_URL, ), tool_filter=['greet'] # restrict to specific tools ) ], )
Deep Dive: Key Considerations
Click each layer to explore important architectural and design considerations for MCP implementations:
Discoverability
Just-in-time capability detection
A key advantage of MCP is that an MCP client can dynamically query a server to learn what tools and resources it offers. This "just-in-time" discovery mechanism is powerful for agents that need to adapt to new capabilities without being redeployed. An agent running today can discover a new tool added to a server tomorrow — no code changes required.
Security & Authorization
Controlling access in a federated ecosystem
An MCP implementation must include authentication and authorization to control which clients can access which servers and what specific actions they are permitted to perform. Without this, exposing tools via MCP is as dangerous as leaving API keys in public repositories. OAuth 2.0 and API key schemes are common approaches.
Error Handling Strategy
Communicating failures back to the LLM
A comprehensive error-handling strategy is critical. The protocol must define how errors — tool execution failure, unavailable server, invalid request — are communicated back to the LLM so it can understand the failure and potentially try an alternative approach. Structured error responses with error codes and messages are essential.
Agent-Friendly API Design
The quality of the interface matters as much as the standard
MCP is only as good as the APIs it wraps. Simply wrapping legacy APIs without modification can be suboptimal. A ticketing API that only returns full ticket details one-by-one is inefficient for summarizing high-priority tickets. The underlying API should support filtering, sorting, and return agent-readable formats (JSON, Markdown) rather than binary blobs.
Local vs. Remote Deployment
Choosing the right transport for your use case
Local MCP servers (STDIO/JSON-RPC) are faster and more secure for sensitive data — only the local machine can access them. Remote MCP servers (HTTP/SSE) allow shared, scalable access across an organization or team. Real-time conversational agents often need local, low-latency servers, while batch processing pipelines may prefer remote servers for scalability.
At a Glance

WHAT

LLMs must move beyond text generation to interact with external systems. Without a standard, every LLM-to-tool integration is custom, complex, and non-reusable — making scalable agentic systems difficult to build.

WHY MCP

MCP provides a standardized client-server protocol where servers expose tools, resources, and prompts to any compliant client. This fosters an ecosystem of interoperable, reusable components and dramatically simplifies complex agentic workflows.

RULE OF THUMB

Use MCP for complex, scalable, enterprise-grade agentic systems interacting with diverse and evolving tools. For simple applications with a fixed, limited set of predefined functions, direct function calling may be sufficient.

KEY TOOLS

Google ADK's MCPToolset for consuming MCP servers. FastMCP for rapidly building Python-based MCP servers. MCP Toolbox for Databases. MCP Tools for Genmedia Services.

Key Takeaways
  • The Model Context Protocol (MCP) is an open standard facilitating standardized communication between LLMs and external applications, data sources, and tools — think of it as a universal power outlet for AI integrations.
  • MCP employs a client-server architecture where servers expose three primitive types: Resources (static data), Tools (executable functions), and Prompts (interaction templates).
  • A key advantage over proprietary tool calling is dynamic discoverability — MCP clients can query servers at runtime to learn what capabilities they offer, enabling agents to adapt without redeployment.
  • MCP is only as good as the APIs it wraps. Effective MCP servers expose agent-friendly, text-based data formats (JSON, Markdown) and provide efficient query capabilities, not just raw legacy API pass-throughs.
  • The Agent Development Kit (ADK) supports both consuming existing MCP servers via MCPToolset and building custom MCP servers. FastMCP simplifies server creation with automatic schema generation from Python type hints.
  • Security must be built into any MCP deployment — authentication, authorization, and careful scoping of tool access are critical to prevent unauthorized actions in a federated tool ecosystem.
  • Transport choices matter: STDIO/JSON-RPC for local, low-latency, secure interactions; HTTP/SSE for remote, scalable, shared-access deployments.
  • MCP enables LLMs to interact with databases, manage generative media workflows, control IoT devices, automate financial services, and orchestrate complex multi-step workflows — all through a single standardized interface.
